Hi:<br>

<br>

Playing with monodoc and mono-tools I have encountered two possible
problems with the build. First I have mono, gtk-sharp, monodoc and
mono-tools from svn. Gtk-sharp from svn installs in
$prefix/lib/pkgconfig all the foo-2.0.pc config files, so after I got
installed monodoc, when I try to autogen.sh in mono-tools I get an
error about not having installed gtk-sharp. A quick view to config.log
confirms that the configure script wants config files of type foo.pc
and not the ones that are installed like foo-2.0.pc. Another quick view
of <a href="http://configure.in/" target="_blank" onclick="return top.js.OpenExtLink(window,event,this)">configure.in</a> presents:<br>

<br>

#PKG_CHECK_MODULES(GTK_SHARP, glade-sharp-2.0 gnome-sharp-2.0
gconf-sharp-2.0 gtkhtml-sharp-2.0, enable_gtks=yes, enable_gtks=no)<br>

#if test &quot;x$enable_gtks&quot; = &quot;xno&quot; ; then<br>

&nbsp;&nbsp;&nbsp;&nbsp;&nbsp;&nbsp;&nbsp; PKG_CHECK_MODULES(GTK_SHARP,
glade-sharp gnome-sharp gconf-sharp gtkhtml-sharp, enable_gtks=yes,
enable_gtks=no)<br>

&nbsp;&nbsp;&nbsp;&nbsp;&nbsp;&nbsp;&nbsp; if test &quot;x$enable_gtks&quot; = &quot;xno&quot; ; then<br>

&nbsp;&nbsp;&nbsp;&nbsp;&nbsp;&nbsp;&nbsp;&nbsp;&nbsp;&nbsp;&nbsp;&nbsp;&nbsp;&nbsp;&nbsp; AC_MSG_ERROR([gtk-sharp not found])<br>

&nbsp;&nbsp;&nbsp;&nbsp;&nbsp;&nbsp;&nbsp; fi<br>

#fi<br>

<br>

So for some reason the line to get the right configs is commented and
one if block is commented too that will be of use to fallback if the
2.0.pc config are not found.<br>

To use gecko-sharp is the same thing, configure asks for gecko equal to
version 0.6 but I got the svn version (0.10) and changing the configure
to use gecko-sharp-2.0.pc I can run monodoc with no apparent breakage.<br>

So my question is this a conscious decision to use an older gtk-sharp and gecko-sharp to build the gui to monodoc?<br>

Another bug is that when a make distclean is done in mono-tools the
<a href="http://monodoc.in/" target="_blank" onclick="return top.js.OpenExtLink(window,event,this)">monodoc.in</a> file is deleted from the source and this is well very bad.<br>

That is all, I have attached a little patch that in my opinion fixes both errors.<br>
