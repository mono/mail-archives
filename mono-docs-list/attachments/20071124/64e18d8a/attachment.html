Hello everybody,<br><br>let&#39;s remove the dust from this list again ;-)<br><br>Today I would like to propose a new documentation subsystem to<br>completely replace the old one.<br><br>Because replacing seems like such a big step I would like to explain why
<br>I want to abolish it:<br> - Insufficient support for &quot;Multi-Version&quot; files (e.g. Class-Libs<br>&nbsp;&nbsp; from .NET 1.1 and 2.0) because the current &quot;Since Version X.Y&quot; does<br>&nbsp;&nbsp; not work if a member is only available in 
1.1 and not in 2.0.<br> - Generic Types and Members are not fully supported in the<br>&nbsp;&nbsp; documentation browser.<br> - Automated user-contributions do not work properly (I don&#39;t know if a<br>&nbsp;&nbsp; contribution has ever made it into the documentation).
<br> - Even if documentation has been updated the updates won&#39;t make it back<br>&nbsp;&nbsp; to the user before a new version of mono has been released. (With<br>&nbsp;&nbsp; other words, there is no way to update offline-documentation)<br>
 - Documentation work-cycle is pretty complicated. It is very unhandy to<br>&nbsp;&nbsp; run many tools to do one simple task. (monodoc, browser, mdassembler)<br> - No Revision control (this is not a big turn-off however it would be
<br>&nbsp;&nbsp; quite handy when user-contributions will finally work) and no user<br>&nbsp;&nbsp; comments (again not very important but nice to have)<br><br>The new documentation subsystem should be able to fulfill all<br>requirements above. This means the new documentation subsystem would
<br>consist of a user and server part.<br><br>The Server Part would mainly be used for data-storage and contribution<br>handling. A simple web-service which is backed by some db would be quite<br>sufficient. For Contribution-administration and adding of new assemblies
<br>to the documentation service a simple web-ui should suffice.<br><br>The Client part however will be more important. It will also have a<br>small SQLite-DB which contains all the data for offline availability.<br>This data will be updated with new documentation from the server
<br>regularly.<br>&nbsp;&nbsp;&nbsp;&nbsp;&nbsp;&nbsp;&nbsp;&nbsp;Apart form adding documentation to the class libraries this tool will<br>also support 3rd party assemblies. It should be possible to add<br>libraries or existing documentation to the client application which can
<br>than be easily documented. For distribution of the documentation I was<br>thinking of two possibilities:<br> 1. it should be able to export the documentation of the 3rd party<br>&nbsp;&nbsp;&nbsp;&nbsp;assembly to a single file (probably again a SQLite-DB). Users can
<br>&nbsp;&nbsp;&nbsp;&nbsp;than xcopy the exported file into some directory so that this db<br>&nbsp;&nbsp;&nbsp;&nbsp;will also be displayed in their browser.<br> 2. If 3rd party developers set up their own documentation server it<br>&nbsp;&nbsp;&nbsp;&nbsp;would be very neat if users get a URL to that server and then
<br>&nbsp;&nbsp;&nbsp;&nbsp;&quot;check-out&quot; a copy of the documentation for offline availability.<br><br>If this subsystem will be designed very nicely this will give 3rd party<br>developers also the ability to set-up a documentation browser as well so
<br>that they can also benefit from user-contributed documentation.<br><br>Hopefully I was able to communicate my ideas well enough and now I&#39;m<br>looking forward to some comments :-)<br><br>Kind Regards,<br>Valentin S.
