Hey,

        The card table is used keep track of stores of references to minor generation objects inside objects of the major generation (generally). As you can imagine, these stores happen mostly in JIT compiled code (which starts after the runtime invoke) and it doesn't make sense to transition to native code each time. This cardtable marking is called the write barrier and you can see how it is emitted in mini_emit_write_barrier. For the general case this generates the assembly associated with OP_CARD_TABLE_WBARRIER.


         Especially at startup we might have some objects that are allocated directly in the major heap, so there can be cardtable markings without yet having any collections and normal object promotion.
