Hello Atsushi,
>  Can you please don't randomly make cosmetic code changes that does 
> not only prevent active hacking but also removes variable/member 
> references that we can find by IDEs? There are very often reason we 
> have such extra code.
I am not doing this for a fun but it helps me greatly if I can build 
mono class libraries with as little warnings as possible. It's much 
easier for me to look for warnings regressions or testing new warnings 
checks when the warnings count in our base class library is manageable. 
Unfortunately, not many people care about the warnings (I know there are 
some exceptions like Zoltan and others which I am very grateful for) but 
it not possible for me to track every maintainer of every change in a 
file where I comment out or fix a warning.

I always try to fix cosmetic warnings first and keep the hard one for 
later when it's clear what the code should be doing. It'd be very 
helpful if you tell me what of my change prevents you from active 
hacking. I'd like to fix it immediately and avoid same conflict in the 
future.

Looking at your latest code changes I cannot see anything what ordinary 
git merge cannot merge in your latest change. It seems to me it's more 
matter of setting up your git to merge changes correctly.
