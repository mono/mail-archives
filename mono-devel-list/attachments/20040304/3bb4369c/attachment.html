<DIV>Another general thing that I came up with is localization:<BR>Currently we 
use Locale.GetText (string theText). This delivers (or should deliver) the 
localized version of the theText string.<BR>Localized will probably mean: In the 
locale of the current user, right?</DIV>
<DIV>&nbsp;</DIV>
<DIV>However what happenes in the following case:<BR>Think about a multilanguage 
application. Every user gets (e.g. UI) - Resources in her language. However if 
an error occurs<BR>the application will catch that and output it to a file, so 
that an administrator or developer can look at it later.<BR>Effect would now be 
that this file contains localized!! messages in whatever languages. The poor 
admin might not even<BR>be able to read all those languages.<BR>So I would 
suggest to add an overload like Locale.GetText (string theText, bool 
systemLocale)<BR>to add the option to suggest to mono that this message should 
be displayed in system locale and not in user locale. You could<BR>even make 
this modifyable at runtime to determine if this should also display user locale 
(in a completely standalone scenario.)</DIV>
<DIV>&nbsp;</DIV>
<DIV>Especially in the context of web - development this seems to make a lot of 
sense.<BR>Candidates for systemLocale strings are especially the Exception error 
messages<BR></FONT></DIV></FONT></DIV></BODY></HTML>
