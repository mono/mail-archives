<p class=MsoNormal><span lang=EN-US>The attached patch files are to fix a bug
when hosting an MVC 1.0 website on Mono 2.4.2.3 (and XSP 2.4.2) in combination
with a FastCGI setup on the web server.<o:p></o:p></span></p>

<p class=MsoNormal><span lang=EN-US><o:p>&nbsp;</o:p></span></p>

<p class=MsoNormal><span lang=EN-US>The problem was initially reported on 5 August
2009 by Sergii Volchkov (see <a
href="http://www.mail-archive.com/mono-list@lists.ximian.com/msg30411.html">http://www.mail-archive.com/mono-list@lists.ximian.com/msg30411.html</a>
titled &#8220;MVC site on fastcgi + nginx produces NotImplementedException in
System.Web.Routing.Route.GetRouteData unless PATH_INFO is specified&#8221;).&nbsp;
However, a similar problem is also encountered with Apache when using a FastCGI
setup (for example with mod_fastcgi).<o:p></o:p></span></p>

<p class=MsoNormal><span lang=EN-US><o:p>&nbsp;</o:p></span></p>

<p class=MsoNormal><span lang=EN-US>The issue seems to arise from the override
method implementation in Mono.WebServer.FastCgi.WorkerRequest.GetPathInfo,
which can sometimes return null (although this should perhaps not be a valid return
value). &nbsp;Looking at the base implementation in System.Web.HttpWorkerRequest.GetPathInfo
(and the MSDN documentation for the System.Web.HttpRequest.PathInfo property),
this virtual method should maybe rather return String.Empty for the undefined
case.&nbsp; Also, it may be better if System.Web.Routing.Route.GetRouteData is
more forgiving about acceptable values and should perhaps also allow null
strings to pass through (since it doesn&#8217;t actually use the pathInfo
variable anyway).&nbsp; The System.Web.HttpRequest.PathInfo property&#8217;s implementation
should perhaps also never allow null to be returned (this one may be the most
essential change).&nbsp; The three above-mentioned changes are illustrated in
the three patch files.<o:p></o:p></span></p>

<p class=MsoNormal><span lang=EN-US><o:p>&nbsp;</o:p></span></p>

<p class=MsoNormal><span lang=EN-US>Perhaps one should also review the rest of
the FastCGI code for similar scenarios when parameters are not defined, if they
should instead also be returning empty strings.<o:p></o:p></span></p>

<p class=MsoNormal><span lang=EN-US><o:p>&nbsp;</o:p></span></p>

<p class=MsoNormal><span lang=EN-US>Could someone please review/commit the
patches to both version 2.6 and the trunk?<o:p></o:p></span></p>
