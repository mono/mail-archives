<div>&nbsp;</div>
<div>I'd like to see the following changes in the next gendarme release (well, I guess not all will make it ;-)&nbsp;- some are cosmetic, some make rule writing more easier imho, ...)</div>
<div>&nbsp;</div>
<div>1) new ctor overloads for Location</div>
<div>&nbsp;</div>
<div>- Location (AssemblyDefinition)</div>
<div>
<div>- Location (ModuleDefinition)</div>
<div>
<div>- Location (TypeReference)</div>
<div>
<div>- Location (MethodDefinition)</div>
<div>
<div>- Location (MethodDefinition, Instruction ins)</div>
<div>&nbsp;</div>
<div>Currently nearly every rule builds the location strings by themselves (new Location(method.DeclaringType.FullName, method.Name, insn.Offset)). Imho those new overloads make the code more readable by writing code like&nbsp;new Location (method, ins) or new Location (type).
</div>
<div>&nbsp;</div>
<div>2) message texts should be stored in &lt;assembly&gt;.xml</div>
<div>&nbsp;</div>
<div>An example: UseStringEmptyRule.cs: Message msg = new Message (&quot;instance of an empty string has been found.&quot;, loc, MessageType.Warning). But on the other side rule problem and solution are stored in the rule information xml file. This is just a &quot;to be consistent&quot; change, but perhaps will help you later to bring i8n to gendarme. A proposal:
</div>
<div>&nbsp;</div>
<div>&lt;rule&gt;</div>
<div>&nbsp; &lt;problem&gt;...&lt;/problem&gt;</div>
<div>&nbsp; &lt;solution&gt;...&lt;/solution&gt;</div>
<div>&nbsp; &lt;messages&gt;</div>
<div>&nbsp;&nbsp;&nbsp;&nbsp; &lt;message Id=&quot;instance-found&quot;&gt;instance of an empty string has been found.&lt;/message&gt;</div>
<div>&nbsp; &lt;/messages&gt;</div>
<div>&lt;/rule&gt;</div>
<div>&nbsp;</div>
<div>Code: new Message (&quot;instance-found&quot;, new Location (method, ins));</div>
<div>&nbsp;</div>
<div>3) Remove the full qualified assembly name inside rule information file.</div>
<div>&nbsp;</div>
<div>Currently each rule stores its assembly name via a Makefile preconditioned attribute (<font size="2">Type=&quot;Gendarme.Rules.Performance.EmptyDestructorRule, Gendarme.Rules.Performance, Version=@VERSION@, Culture=neutral, PublicKeyToken=null&quot;). As proposed before this may not be necessary if you don't want to load your rules from the GAC (which imho will never happen). By just specifying Type and/or AssemblyName you could drop the Makefile dependency and let other developers easier work with these files. It's not very &quot;sexy&quot; to modify those files manually if you are using a IDE (like Visual Studio or #Develop).
</font></div>
<div><font size="2"></font>&nbsp;</div>
<div><font size="2">4) Redefine &quot;Rule Failure&quot; and &quot;Rule Success&quot;</font></div>
<div><font size="2"></font>&nbsp;</div>
<div><font size="2">With the current framework it is possible to create a violation by just returning an empty collection. IMHO each violation should have a location to make tracking down the source easier. My proposal is that rule &quot;fails&quot; if one or more messages are returned by the CheckXXX methods. On a side node, I don't like the &quot;static RuleFailure&quot; and &quot;RuleSuccess&quot; properties which imho can be dropped if the above change may be accepted.
</font></div>
