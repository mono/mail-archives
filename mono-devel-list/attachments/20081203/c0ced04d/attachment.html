After some discussions with Mark we decided to do a few changes to Mono.Simd and<br>it would be really nice to have some feedback from everyone on them.<br><br>One of the motivation is that the current C# of Mono.Simd is not endianess safe and doesn&#39;t<br>
work as expected on platforms such as ppc. The other is to advance the quality of the API.<br><br>The first change is to remove byte oriented methods from all vector types that don&#39;t work on byte quantities.<br><br>Right now the only method is ExtractByteMask which is mostly used together with compare ops and will be replaced<br>
&nbsp;with proper operator overloads.<br><br>The second change is actually a clarification, Mono.Simd have some arch dependent methods such as casting between <br>vector types that are endian unsafe. The alternative would be to define those methods to have little-endian semantics<br>
and, latter on, when we add support to a big-endian arch revert this decision.<br><br>The last one is a call if extension methods should be used or not. Extensions methods will make coding with vector<br>types quite a bit simpler as a lot less typing will be needed. For example, &quot;Vector8us.AddWithSaturation (a, b)&quot; would<br>
become &quot;a.AddWithSaturation (b)&quot;. Right now we already expose some extension methods for interoperability with arrays.<br><br>There are two issues with extension methods, first that it requires C# 3.0 and .NET 3.5 if you want your code to compile for it.<br>
AFAIK this is no big deal as those methods would still be available under C# 2.0. The second issue is a limitation of the C# language,<br>which require extension methods to be defined of sealed classes. This would mean that code written against mono 2.2 version<br>
of Mono.Simd might not compiler with newer versions.<br><br>Keeping both versions of the methods is an option as well, but I&#39;m not very keen on the extra bloat imposed. Either way, I would<br>love to rear feedback from people evaluating or using Mono.Simd.<br>
