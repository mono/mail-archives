&gt; Glad to hear codecs are not a focus for Mono.Media (at present). Somehow<br>
&gt; most discussion in this area seems to be focused on re-implementing<br>
&gt; gstreamer or creating bindings for it (a/v format coding/decoding in<br>
&gt; general). Last weekend I submitted an application for Mono.Media GSoC. My<br>
&gt; proposal seeks to provide high performance, low-level audio/MIDI features<br>
&gt; for Mono, rather than a generic coder/decoder framework. One thing I have<br>
&gt; focused on is fairly complete MIDI support.<br>
&gt;<br>
&gt; I didn&#39;t base my proposal on the Java media framework (I don&#39;t have much,<br>
&gt; if<br>
&gt; any, experience with Java, plus I saw discussion on how JMF&#39;s heavily<br>
&gt; asynchronous design is undesirable), so I might be missing some obvious<br>
&gt; features. I&#39;ll appreciate if the Mono dev community could comment on my<br>
&gt; design.<br>
&gt;<br>
&gt; Following is an excerpt from the proposal:<br>
&gt;<br>
&gt; The architecture of the Mono.Media.Audio section would be as follows:<br>
&gt;&gt;<br>
&gt;&gt; - Classes representing audio devices will be implemented. These will<br>
&gt;&gt; abstract out platform-specific code (ie. the same AudioDevice would use<br>
&gt;&gt; WDM/KS on Windows and Jack/Alsa on Linux)<br>
&gt;&gt; - Classes representing audio sources will be implemented. These will have<br>
&gt;&gt; methods to *provide* audio.<br>
&gt;&gt; - Classes representing audio consumers will be implemented. These act<br>
&gt;&gt; like<br>
&gt;&gt; sinks. An AudioSourcePlayer for example will continuously stream audio<br>
&gt;&gt; from<br>
&gt;&gt; an AudioSource to an AudioDevice. The AudioDevice will have a callback<br>
&gt;&gt; method on its own high priority thread to get audio buffers.<br>
&gt;&gt; - Further classes like Mixer, CrossFader, Equalizer etc. might be<br>
&gt;&gt; implemented as a combination of sinks and sources.<br>
&gt;&gt; - Classes for reading/writing simple audio files (.wav, .aiff etc.) will<br>
&gt;&gt; also be implemented.<br>
&gt;&gt;<br>
&gt;<br>
&gt;<br>
&gt; The MIDI architecture will consist of :<br>
&gt;<br>
&gt; - Classes for MIDI input and output devices. These will represent actual<br>
&gt;&gt; devices on Windows (which does not support the concept of virtual<br>
&gt;&gt; devices),<br>
&gt;&gt; and software ports on Linux and Mac. The winmm API will be used on<br>
&gt;&gt; Windows,<br>
&gt;&gt; CoreMIDI on Mac, and ALSA sequencer API on Linux.<br>
&gt;&gt; - Classes for MIDI events (a generic event, NoteOn, NoteOff events,<br>
&gt;&gt; SysEx,<br>
&gt;&gt; MetaEvents, Controller etc.). These will allow easy creation/manipulation<br>
&gt;&gt; of<br>
&gt;&gt; MIDI data, shielding the programmer from the actual byte-stream.<br>
&gt;&gt; - Classes for storing/manipulating these (e.g. MidiTrack, MidiSequence,<br>
&gt;&gt; MidiRecorder etc.)<br>
&gt;&gt; - Classes for handling MIDI file input/output<br>
&gt;&gt; - Classes for handling MIDI time code and transport control messages<br>
&gt;&gt;<br>
&gt;&gt; Basically the MIDI framework will be a fairly complete implementation of<br>
&gt;&gt; everything that MIDI has to offer (except perhaps highly specialized<br>
&gt;&gt; standards like Sample Dump Standard). If we have time, an OSC interface<br>
&gt;&gt; might also be implemented.<br>
&gt;&gt;<br>
&gt;<br>
&gt;<br>
&gt; Of course this design is open to discussion. Thanks for your time!<br>
