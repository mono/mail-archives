<div class="">In general, you shouldn't need to care about the .NET Framework profile your app is compiled against when running it on (newer) Mono.</div>
<div class=""><br class="">
</div>
<div class="">Today*, we only ship a single set of assemblies which implement the BCL (mscorlib, System.*) so your app always runs on</div>
<div class="">those no matter what profile it was compiled against. .NET is really good at keeping backwards compatibility in the API</div>
<div class="">so this works out fine. You get the same experience e.g. on Windows 10 which doesn't ship with .NET 2.0 by default, yet</div>
<div class="">an old app still works fine because it just runs on the newer framework.</div>
<div class=""><br class="">
</div>
<div class="">I'd just keep the config file you posted which will make the warning go away on newer Mono's (it won't change anything about the</div>
<div class="">actual execution since there's only a single set of BCL assemblies there).</div>
<div class=""><br class="">
</div>
<div class="">By the way, I'd strongly encourage you to not deal with Mono 3.x anymore and upgrade to the latest if possible, it'll save you a lot of trouble.</div>
<div class=""><br class="">
</div>
<div class="">Hope this helps,</div>
<div class="">Alex</div>
<div class=""><br class="">
</div>
<div class="">* a long time ago like in Mono 3.2.8 we actually compiled the BCL for each .NET Framework profile (2.0, 3.5, 4.0) and simply used ifdefs</div>
<div class="">to exclude APIs which were available in newer profiles, but that got unwieldily and made no real sense anymore since the underlying</div>
<div class="">implementation is the same anyway so we stopped doing that: <a href="http://www.mono-project.com/docs/about-mono/releases/4.0.0/#dropped-support-for-old-frameworks" class="">http://www.mono-project.com/docs/about-mono/releases/4.0.0/#dropped-support-for-old-frameworks</a></div>
