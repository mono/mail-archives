Dear Mr. de Icaza and the Mono Developer Community,<br><br>First, I would like to salute you for producing an excellent, open platform for software development. Second, I would like to apologize for bothering anyone who is not interested in a student&#39;s proposal to improve it. I appreciate your time and any feedback you might give.<br>

<br>I am proposing to expand Mono&#39;s C++ interop support to enable the creation of managed wrappers directly around native C++ objects. This would make C++ libraries callable directly from managed code without the need for wrapping them in a flat C API, COM interface, or requiring the use of mixed binaries (C++/CLI). In fact, this could also provide a way to implement the C++/CLI language, including IJW, in a cross-platform manner.<br>
<br>The first place I read about calling C++ functions directly from managed code was on Mono&#39;s &quot;<a href="http://www.mono-project.com/Interop_with_Native_Libraries#Invoking_Unmanaged_Code">Interop with Native Libraries</a>&quot; page. It suggested setting the EntryPoint of the DllImport attribute to the C++ mangled function name to call that function directly through P/Invoke. However, it wasn&#39;t until I read <a href="http://blogs.msdn.com/vcblog/archive/2008/12/08/inheriting-from-a-native-c-class-in-c.aspx">this blog post</a> by Jim Springfield that I realized that, not only could this be a viable technique, but that by messing with virtual tables, native C++ classes could effectively be subclassed by managed code. This technique could allow for seamless managed wrappers around native C++ classes.<br>
<br>Jim Springfield&#39;s example is tied completely to Microsoft&#39;s Visual C++ compiler, and this illustrates the largest problem with this approach: that C++ ABIs are different among different compilers and even between different versions of the same compiler. To help ameliorate this issue, I have taken the basic principles in Springfield&#39;s design and abstracted out any ABI-specific components into an abstract class. A different subclass of this CppAbi class can then be implemented to support each compiler&#39;s different name mangling schemes and other idiosyncrasies. At runtime, the correct CppAbi instance can be chosen when loading the C++ library depending on platform or other conditions. Reflection.Emit is then used to generate the P/Invoke code and implement trampolines to facilitate virtual method calls. Eventually I hope to support seamless exception handling and other features supported by C++/CLI on Windows.<br>
<br>I realize this sounds very ambitious, but I&#39;ve already implemented a proof-of-concept based on a simple C++ class, similar to the one Jim Springfield uses in his example. It is hosted on Google code at <a href="http://code.google.com/p/cppinterop/">http://code.google.com/p/cppinterop/</a>. Please note that this really is just a proof-of-concept-- I&#39;ve only implemented the<a href="http://www.codesourcery.com/public/cxx-abi/abi.html"> Itanium C++ ABI</a>, and only in part. If you are using a recent version of GCC to compile C++, you should be able to compile the example and call it directly from managed code. I&#39;ve only tested this on an Intel Mac running OS X 10.4.11.<br>
