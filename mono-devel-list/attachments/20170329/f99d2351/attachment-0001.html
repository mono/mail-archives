<div>Integrating the CoreRT threadpool into Mono will be more of an experiment than a project that we envision merging by the end of GSoC into Mono. If that's find by you, then it's perfectly fine by us. Another project that would have more impact
 and that we are looking forward to merge would be importing System.Threading.<wbr>Mutex, System.Threading.<wbr>Semaphore and System.Threading.<wbr>EventWaitHandle from CoreRT into Mono. Whichever project you choose is definitely up to you, and whichever you choose, it will
 be of great use for us.</div>
<div><br>
</div>
<div>To discuss each project in more details, importing System.Threading.ThreadPool would consist in getting <a href="https://github.com/dotnet/corert/blob/master/src/System.Private.CoreLib/src/System/Threading/ThreadPool.cs" target="_blank">https://github.com/<wbr>dotnet/corert/blob/master/src/<wbr>System.Private.CoreLib/src/<wbr>System/Threading/ThreadPool.cs</a><wbr> and
 its dependencies from CoreRT, and adapting it to the use of Mono. Some features are not supported by the CoreRT ThreadPool such as SetMinThreads, SetMaxThreads, GetMinThreads, GetMaxThreads, or other APIs, and because we cannot just drop the support for these,
 we need to make sure they are implemented and supported. Right now, our ThreadPool implementation is coming from ReferenceSource for the managed part and we reimplemented the native part into Mono from CoreCLR (<a href="https://github.com/mono/mono/blob/master/mono/metadata/threadpool.c" target="_blank">https://github.com/mono/mono/<wbr>blob/master/mono/metadata/<wbr>threadpool.c</a> and <a href="https://github.com/mono/mono/blob/master/mono/metadata/threadpool-worker-default.c" target="_blank">https://<wbr>github.com/mono/mono/blob/<wbr>master/mono/metadata/<wbr>threadpool-worker-default.c</a>).</div>
<div><br>
</div>
<div>For System.Threading.Mutex, <wbr>System.Threading.Semaphore and System.Threading.<wbr>EventWaitHandle, the project would consist in replacing our implementation of these classes with the one from CoreRT. Our implementation resides in <a href="https://github.com/mono/mono/blob/master/mcs/class/corlib/System.Threading/Mutex.cs" target="_blank">https://github.com/mono/<wbr>mono/blob/master/mcs/class/<wbr>corlib/System.Threading/Mutex.<wbr>cs</a>, <a href="https://github.com/mono/mono/blob/master/mcs/class/referencesource/System/sys/system/threading/semaphore.cs" target="_blank">https://github.com/mono/<wbr>mono/blob/master/mcs/class/<wbr>referencesource/System/sys/<wbr>system/threading/semaphore.cs</a> <wbr>and <a href="https://github.com/mono/mono/blob/master/mcs/class/referencesource/mscorlib/system/threading/eventwaithandle.cs" target="_blank">https://github.com/mono/<wbr>mono/blob/master/mcs/class/<wbr>referencesource/mscorlib/<wbr>system/threading/<wbr>eventwaithandle.cs</a> <wbr>respectively.</div>
<div><br>
</div>
<div>For both these projects, as well as the other integration of .NET sources into Mono, we need to ensure that we keep supporting the same platforms as before, thus fixing or extending the CoreRT implementations.</div>
