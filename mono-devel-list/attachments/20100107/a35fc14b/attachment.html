The bottom line of this message is that I don&#39;t see how can one write a<br>
high-performance socket server in Mono...<br>
Here is the story:<br>
I am writing a proxy server using .NET Socket API. This proxy does almost<br>
entirely I/O work - copying buffers from one socket to another. Now, Mono<br>
doesn&#39;t implement the newer<br>
<a href="http://msdn.microsoft.com/en-us/library/system.net.sockets.socketasynceventargs.aspx" target="_blank">http://msdn.microsoft.com/en-us/library/system.net.sockets.socketasynceventargs.aspx</a><br>
XXXAsync Socket API  that was introduced in .NET 2.0 SP1 (or more<br>
accurately, implements it<br>
<a href="http://www.mail-archive.com/mono-list@lists.ximian.com/msg28621.html" target="_blank">http://www.mail-archive.com/mono-list@lists.ximian.com/msg28621.html</a><br>
perfunctorily ). So I was left to use the APM Socket API which produces tons<br>
of garbage objects under heavy load.<br>
When testing the server on Linux under load we saw very frequent CPU bursts<br>
that crippled the throughput of the server. After profiling with the mono<br>
built-in profiler I confirmed that the reason for the high CPU usage was the<br>
GC collections that got more and more frequent and took more and more time.<br>
I then read a little bit and realized that the Mono GC is non-generational<br>
which might explain the long GC cycles (if it was generational it could have<br>
collected the garbage objects that were created during async socket<br>
operations in generation 0 and probably stop there, but it has to traverse<br>
the entire managed heap).<br>
So the combination of the non-generational GC and the unimplemented XXXAsync<br>
Socket API result in very poor performance of the Mono server. The maximum<br>
throughput of the server with Mono on Linux is about half of that on Windows<br>
using .NET.<br>
