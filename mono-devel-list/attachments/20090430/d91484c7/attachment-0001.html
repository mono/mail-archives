<font size="2">Just trying to make it a bit more clear (Michael and Miguel, from the outside seems like you both weren&#39;t discussing exactly the same subject):<br><br>Michael, you can make your reflection-based plugin that accepts any library, GPL or not, as long as you publish the needed interfaces as non-GPL, or reuse some standard one (<a href="http://ADO.NET">ADO.NET</a>). <br>
You can&#39;t just wrap the GPL library to conform to your particular interface,
the library should natively implement your non-GPL interface or the standard one, or if you do really build a
wrapper it has to be made separately available as a GPL open-source project, even if it is reflection-based, as it IS LINKING to the GPLed API or it wouldn&#39;t be needed.<br>What you can&#39;t do is distribute the GPL library with your non-GPL app. <br>
You should instruct your end-user how to download and install and configure the GPL library (and your GPL wrapper if you needed to develop one), if he would like to use it with your app. <br>The end-user is king in such matters.<br>
<br>Personally, I find MySQL not worth all that effort for turn-key (all-in-a-box) solutions,  for those situations I use PostgresSQL or SQLite according to the need to scale.<br></font><br>Regards,<br><br><div class="gmail_quote">

<div class="im">Miguel de Icaza &lt;<a href="mailto:miguel@novell.com">miguel@novell.com</a>&gt; wrote:<br>
<br>
</div><div class="im">&gt; Loading GPL code into a proprietary application is not allowed by the<br>
&gt; GPL.    You can do that in the privacy of your home, but you can not<br>

<br>
</div>I am not sure what you mean by that.  Linking GPL code into a<br>
proprietary application is certainly not permitted.  If I author a<br>
proprietary application and I link against a library that is GPL, then<br>
I must either retain the application in-house and never distribute it,<br>
or I must distribute it under the GPL when I _do_ distribute it.  Since<br>
the GPL&#39;s effects only kick in upon distribution, and not usage, this<br>
makes sense.<br>
<br>
Here&#39;s a couple of examples to illustrate.<br>
<br>
Example 1:<br>
  Let&#39;s say that I have a proprietary application, and we&#39;ll call it<br>
  TheApplication.  (No, I am not creative; oh, well.)  TheApplication<br>
  depends on MySQL, but the driver libraries for MySQL are GPL&#39;d.<br>
  Having written the application, I don&#39;t feel like keeping it in<br>
  development for several more days/weeks/months/whatever just to write<br>
  an adapter to the database to replace the GPL&#39;d library.  So,<br>
  instead, I write a &quot;shim&quot; and GPL this shim.  The shim exposes the<br>
  MySQL client library&#39;s API via a named pipe and I just connect to the<br>
  named pipe and call the client library&#39;s functions by asking the shim<br>
  to do so for me.<br>
<br>
Example 2:<br>
  Let&#39;s say that I have the same proprietary application,<br>
  TheApplication.  However, I say, &quot;I will not write the database<br>
  functionality.  I will leave this up to you to provide.  Here is the<br>
  interface that I expect to be able to use, you give me the file, I<br>
  will load it, and I will use its interfaces.&quot;  So, John Doe decides<br>
  that he needs to use this application with MySQL, and creates a GPL&#39;d<br>
  module for this application program which enables the proprietary<br>
  software to work with MySQL.  He then distributes his work under the<br>
  GPL, because he likes freedom.  He doesn&#39;t have the time to write<br>
  a clone of TheApplication, so this is the best he can do.<br>
<br>
In example 1, I as the application developer have intentionally and<br>
deviously attempted to get out of my responsibility to release my code<br>
under the GPL by creating a wrapper, to separate the MySQL client<br>
library from my program by intentionally placing the library in a<br>
different address space.  This is wrong.  It goes exactly against what<br>
the GPL tries to do with regards to liberating software.  Basically, I<br>
have said, &quot;Nah-nah-nah-boo-boo, stick your head in doo-doo,&quot; and taken<br>
free software and used it improperly.  This isn&#39;t cool.<br>
<br>
In example 2, however, I decide instead that it&#39;s not a good idea for<br>
me to try to support eleventy-one databases directly; I cannot<br>
anticipate my end-user&#39;s needs.  So instead, I say, &quot;Here is the<br>
application software.  If your needs are simple, you can use SQLite,<br>
which I have written a module for, and is in the public domain.  If you<br>
need to support another database in your environment, that&#39;s up to<br>
you.  Here are the interfaces that I will use in a module.&quot;  John Doe<br>
does just that.  It doesn&#39;t _matter_ what license someone writes a<br>
module for, that module isn&#39;t a derived work, and it&#39;s not linked into<br>
my program.  It&#39;s distributed by someone else as a completely separate<br>
work.  John Doe did just that, and he should be commended for choosing<br>
the GPL for the module that he wrote.  This is just fine.  It&#39;s less<br>
than ideal (at least in my opinion, as I believe that all software<br>
should be free software in the manner in which the FSF has defined it),<br>
but TheApplication doesn&#39;t try to do anything nasty or uncool, and John<br>
Doe has provided a very useful and free software module that people are<br>
free to improve upon and redistribute (separately, of course, from the<br>
application, for obvious reasons).<br>
<br>
The line between the two examples is indeed a fine one.  However,<br>
interface usage does not constitute linking.  Dependencies do.<br>
<br>
As has already been mentioned elsewhere in this thread, if mechanisms<br>
like LoadLibrary() or dlopen() or Java or CLR Reflection were<br>
considered linking, the ramifications would be very troublesome and<br>
dangerous for those who did write proprietary software.  So much so<br>
that they would choose either to not write it at all, or make it free<br>
software in the first place.<br>
<br>
Let&#39;s not forget that the GPL does not apply to usage.  It _only_<br>
applies to distribution.  If I write a free software module for a<br>
proprietary program and distribute it, I have in no way violated GPL.<br>
What the GPL covers is not determined by whether or not code lives in a<br>
single address space.  It is determined at build time for the program<br>
in question.  If you statically link against a GPL&#39;d component, then<br>
you must also be GPL by definition.  If you dynamically link against a<br>
GPL&#39;d component, there is no difference from static linking other than<br>
a technical one.  Dynamic linking was designed to reduce memory and<br>
storage requirements since many programs use the same libraries.<br>
However, dlopen() and LoadLibrary()&#39;s usage isn&#39;t linking.  It&#39;s<br>
declaring a dependency on an interface, not a library.  Reflection is<br>
functionally equivalent to using dlopen() and/or LoadLibrary(), but<br>
significantly more powerful due to the technical nature of the CLR.<br>
And even so, if there were an agreed-upon convention that made it<br>
possible to preserve metadata in native executables, then we&#39;d have<br>
reflection at a native level, which still wouldn&#39;t be linking.<br>
