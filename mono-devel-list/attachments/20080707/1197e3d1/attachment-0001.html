<p><font face="Courier New" size="2"><span style="font-size: 10pt;">Everything works fine – only the resize of the terminal does not!</span></font></p></div></div></blockquote><div>Which terminal are you referring to?&nbsp; They come in all shapes and sizes on Unix*, and, as far as I know, there isn&#39;t any single API in which they have all have standardized on to control things like the width and height of the window and buffer.&nbsp; So while this is only a guess, my assumption would be that in the Windows world where -- up until a couple of years ago when Monad, now PowerShell, came into being -- there was really only one terminal application in cmd.exe, it was easy to map the .NET System.Console API to a specific application, whereas in the Unix* there is no real way to know ahead of time which terminal/shell will exist.&nbsp; In this regard (and assuming that I&#39;m not way off base as to what seems to be the most logical reason) I really don&#39;t think it&#39;s possible to expect any the GUI portions of System.Console to work in the same way they work on Windows.<br>
<br>Of course you could always hack on the Mono source directly, binding calls to the System.Console GUI API to the related commands of a given terminal/shell application via p/invoke (or via writing a C-based bindings wrapper), but that would require you distribute not only your application, but the modified System.dll file as well.&nbsp; You could instead write your own System.Console-like API, but that wouldn&#39;t be very portable unless you also wrote bindings for cmd.exe.&nbsp; In this regard, it might be easier to simply create your own Windows.Forms-based terminal application of which you would then have full control over the entire look, feel, and functionality on both Windows and Unix variants.<br>
