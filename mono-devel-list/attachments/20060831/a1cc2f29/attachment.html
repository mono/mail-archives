Hey guys,<br><br>I was about to send this suggestion to Maoni (of the .NET GC - <a href="http://blogs.msdn.com/maoni/">http://blogs.msdn.com/maoni/</a>) and wanted to send it here first, if nothing else, to make the idea less patentable by MS.
<br><br>I would like for the whole memory manager system to be componentized and replaceable. This would allow the user to choose what is the right approach for their application. Then, if done well, third-parties could develop alternate memory managers that were geared for specific purposes.
<br><br>Here's my rationale:<br>1) In some scenarios, I want a memory manager with explicit finalization. That is a priority in scenario A. I will take the performance hit to have full reference counting and immediate finalization. The benefit to me is better control of memory and RAM utilization. 
<br><br>2) I may want a Workstation version that is memory footprint priority. Running on a Citrix server with 4GB RAM but needing to limit the app to 60MB per client (by customer request) is nearly impossible. Either the customer's request cannot be met or .NET Framework / Mono cannot be the platform.
<br><br>3) I don't assume that MS will implement the best memory manager for me. Make allowances for others to improve on it.<br><br><br>Potential issues against it:<br>1) If a memory manager could be replaced, I could re-implement one (or modify an OpenSource one) to circumvent DRM. I can see that being why MS won't do it.
<br><br>2) Complexity of the integration.<br><br>3) Not a large enough perceived benefit/need.<br><br>What are your thoughts?<br><br>-Mark E.<br>
