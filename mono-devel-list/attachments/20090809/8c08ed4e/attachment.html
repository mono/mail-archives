Hi,<br><br>Andreas is right and I completely forgot about this when we discussed :(. There are several parts of Windows Forms that use reflection to instantiate types part of which are in System.Design. As he says some of those types are the TypeEditors and TypeConverters. And if we don&#39;t ship WinForms together with System.Design things will break. The same is also probably valid for <a href="http://ASP.NET">ASP.NET</a>. And you won&#39;t be able to find a hard dependency between the WinForms/<a href="http://ASP.NET">ASP.NET</a> and System.Design because the way it works is via an attribute on a class which is initialized with a Type name (string) in order to be loose coupled. Now the problem is that if System.Design is missing those type won&#39;t be resolved from code that uses them will break.<br>

<br>An example:<br><br>namespace System.Windows.Forms {<br>    [Editor(&quot;System.Windows.Forms.Design.DockEditor, &quot; + Consts.AssemblySystem_Design, typeof(System.Drawing.Design.UITypeEditor))]<br>    public enum DockStyle {<br>

        None    = 0,<br>        Top    = 1,<br>        Bottom    = 2,<br>        Left    = 3,<br>        Right    = 4,<br>        Fill    = 5<br>    }<br>}<br clear="all"><br>If System.Design is missing the WinForms property grid will no long work for editing the Dock property of a Control (TypeEditor) and also won&#39;t be able to convert the type to a human readable string (TypeConverter).<br>

<br>Now, I am not familiar with <a href="http://ASP.NET">ASP.NET</a> but a fgrep showed that there are quite a lot of those attributes with references to System.Design and I guess say if I am developing an app and I have a form where a user can select a Color and I am using the System.Design Color editor and System.Design is missing my app will break.<br>

<br>Unfortunately that leaves you with no option of avoiding aspnet pulling winforms through System.Design. :(<br>Kind Regards,<br><br>Ivan Zlatev<br><a href="http://ivanzlatev.com">http://ivanzlatev.com</a><br>
