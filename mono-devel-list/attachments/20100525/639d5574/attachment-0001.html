<div><div>When doing major collections, sgen must know where a given object lives (it&#39;s space), if it&#39;s on the nursery,</div><div>a major block, a large object and so on.</div><div><br></div><div>Right now, to identify a large object we calculate the object size and check against a constant.</div>
<div>This is not very cache friendly and requires 5 memory loads for a regular object.</div><div><br></div><div>The idea then is to have a faster way to identify this using an address-&gt;space dictionary and this is</div>
<div>were the block map comes in.</div><div><br></div><div>The current design requires only 3 loads to identify the space of an object of which 2 will probably</div><div>be very cache hot.</div><div><br></div><div>Since object spaces are very large memory regions we can get away with a simple 2 level sparse</div>
<div>array with a granularity of 4096 bytes or more.</div><div><br></div><div>The current design is heavily inspired on boehm&#39;s implementation. It uses a 2 level sparse map</div><div>indexed on the pointer value, which is a fancy way of saying an array of arrays.</div>
<div><br></div><div>pointer bits</div><div>31....22      21...12           11..0</div><div>first level    second level    ignored</div><div><br></div><div>To avoid a null check when resolving the second level, we use a single entry filled with null.</div>

&gt; I benchmarked a modified binary-trees without valuetypes. Block maps<br>
