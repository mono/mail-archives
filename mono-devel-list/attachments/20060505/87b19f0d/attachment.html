I've been using Mono.GetOptions for a compiler project of mine and it is a great library for option parsing.&nbsp; However, there is an issue that I've run into that is slightly troublesome.&nbsp; I wanted to run this issue by the community in case I missed a workaround or a solution--documentation is a bit sparse and the source doesn't seem to indicate a way to work around the described issue.
<br><br><span style="font-weight: bold;"></span>Currently the option bundle is defined as a sub-class of the <span style="font-family: courier new,monospace;">Options</span>
 class.&nbsp; For bundles that don't want the standard options, you can attribute an overridden method that is attributed as an option with the  <span style="font-family: courier new,monospace;">KillOptionAttribute</span> attribute.&nbsp; This works well for virtual methods that are decorated as part of the option bundle, but for non-virtual members (
i.e. fields and properties) this poses a problem since you cannot override such members.
<br><br>This affects the <span style="font-family: courier new,monospace;">VerboseParsingOfOptions</span> and <span style="font-family: courier new,monospace;">DebuggingOfOptions</span> members in particular since these are non-virtual properties and you will always have these options regardless of what you may desire.
<br><br>If there is no known work around for this, I would suggest (and may even develop the fix for if I have time) that an <span style="font-family: courier new,monospace;">OptionBundle</span> interface be defined.&nbsp; This interface would be member-less and the existing 
<span style="font-family: courier new,monospace;">Options</span> class could implement it and store it as a property (defaulting to a self reference).&nbsp; An implementation could then explicitly define its own <span style="font-family: courier new,monospace;">
OptionBundle</span> irrespective of the standard <span style="font-family: courier new,monospace;">Options</span> class which may 
invoke the standard options defined.&nbsp; The point would be that the application could have better control over what options are in the bundle.&nbsp; The meta-data approach is a nice one, but the inheritance hierarchy and non-virtual members could get in the way; this would solve that.
<br><br>The <span style="font-family: courier new,monospace;">OptionList</span> class (and possibly the <span style="font-family: courier new,monospace;">OptionDetails</span> class, but I am pretty sure it is not dependant on introspecting the option bundle) would need to be changed to work with 
<span style="font-family: courier new,monospace;">OptionBundle</span>.&nbsp; Furthermore, backwards compatibility would be maintained with how Mono.GetOptions is used today since the default value of the property in the <span style="font-family: courier new,monospace;">
Options</span> class would be itself.&nbsp; To use this new feature you would probably override the <span style="font-family: courier new,monospace;">InitializeOtherDefaults</span> method to set the property to the desired bundle implementation.
