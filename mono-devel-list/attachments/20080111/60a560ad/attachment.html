It only seems fair.&nbsp; I would do the localization in the OptionException
class itself.&nbsp; I see other mono Exception types calling
Locale.GetText() to translate their text (e.g. FileNotFoundException),
but I don't see where its declared.&nbsp; (I see one in
mcs/class/Managed.Windows.Forms/Assembly/Locale.cs but I don't think
that's it).<br>
<br>
A couple of other thoughts: <br>
<br>
-- I think it would be convenient to the user if the error message
included the option value whenever ConvertFromString() failed.&nbsp; That
way if there are many instances of the same option the user can easily
figure out which one is causing the problem.&nbsp; An alternative is to
include the index of the offending argument as a property of the
OptionException, leaving it to the caller's exception handler to point
out the value.<br>
<br>
-- I was looking at your FooConverter example and it occurred to me
that the code might be more concise if Options.Add&lt;T&gt;() had an
overload that took a TypeConverter.&nbsp; That way I could simply say:<br>
<br>
<tt>&nbsp;&nbsp;&nbsp;&nbsp;&nbsp;&nbsp;&nbsp;&nbsp; Options p = new Options () {<br>
&nbsp;&nbsp;&nbsp;&nbsp;&nbsp;&nbsp;&nbsp;&nbsp;&nbsp;&nbsp;&nbsp; { "v", v =&gt; ++verbose },<br>
&nbsp;&nbsp;&nbsp;&nbsp;&nbsp;&nbsp;&nbsp;&nbsp;&nbsp;&nbsp;&nbsp; { "foo", <b>new FooConverter()</b>, (int v) =&gt; foo = v
},<br>
&nbsp;&nbsp;&nbsp; &nbsp;&nbsp;&nbsp; &nbsp;&nbsp;&nbsp; { "h|?|help", v =&gt; ShowHelp () }<br>
&nbsp;&nbsp;&nbsp;&nbsp;&nbsp;&nbsp;&nbsp;&nbsp; };<br>
</tt><br>
... rather than having to use a wrapper type (Foo) as a stand-in for
the real type.<br>
