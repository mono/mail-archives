The behaviour is as follows:<br>
1) the runtime will choose whether a method is optimized or not<br>
depending on the optimization flags (-O=simd, on by default) and on<br>
the features of the current processor.<br>
2) the attributes on the methods are never inspected by the runtime:<br>
they are there to guide the programmers using Mono.Simd in determining<br>
what kind of optimizations are usually available or currently enabled.<br>
<br>
The reasoning is this: using unoptimized Mono.Simd is currently<br>
significantly slower than he equivalent scalar code. This has mostly to<br>
do with the additional copies that happen because of the operator<br>
overloading. This overhead is expected to decrease as we add more jit<br>
optimizations. So you have two cases:<br>
<br>
1) the slowdown is not significant to you (you must test! Run your<br>
program with mono -O=simd and with mono -O=-simd): in this case<br>
you should ignore completely the acceleration attributes and just enjoy<br>
the speedup that the jit will give you when it can optimize the methods.<br>
<br>
2) if the slowdown is significant you might want to have two codepaths,<br>
mostly in the same way in C/C++ you have a C implementation and a simd<br>
implementation of the critical functions. Now the question becomes:<br>
how do you choose at runtime if you want to use Mono.Simd or the scalar<br>
codepath? We offset two patters:<br>
<br>
a) do a coarse decision: you take a look at the methods you use in your<br>
algorithms and see that they are optimized when SSE2 is enabled, so you<br>
just do:<br>
 &nbsp; &nbsp; &nbsp; &nbsp;static readobly bool use_mono_simd = (SimdRuntime.AccelMode &amp; AccelMode.SSE2) != 0;<br>
 &nbsp; &nbsp; &nbsp; &nbsp;...<br>
 &nbsp; &nbsp; &nbsp; &nbsp;if (use_mono_simd)<br>
 &nbsp; &nbsp; &nbsp; &nbsp; &nbsp; &nbsp; &nbsp; &nbsp;// simd codepath<br>
 &nbsp; &nbsp; &nbsp; &nbsp;else<br>
 &nbsp; &nbsp; &nbsp; &nbsp; &nbsp; &nbsp; &nbsp; &nbsp;//scalar codepath<br>
<br>
b) a fine-grained decision based on all or some of the methods you use:<br>
for each method you check<br>
 &nbsp; &nbsp; &nbsp; &nbsp;(SimdRuntime.MethodAccelerationMode (typeof(...), &quot;...&quot;) &amp; SimdRuntime.AccelMode) != 0<br>
until you determine that enough of your methods are accelerated to make<br>
it worth using the Mono.Simd codepath.<br>
<br>
Note that we may eventually either return the attribute not based on the<br>
metadata in the assembly, but based on the runtime understanding: this<br>
will avoid the need to have an updated Mono.Simd assembly when new<br>
optimizations are added. Just use the b pattern if you want to avoid<br>
that issue and remember that you don&#39;t usually need to check all the<br>
methods, but just the ones you actually need to be optimized.<br>
