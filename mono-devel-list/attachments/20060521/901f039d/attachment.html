exceptions, events, delegates, ...</font></div></div></div></blockquote><div><br>Just a note, from the JVM point of view, all exceptions are unchecked.&nbsp; The JVM &quot;has&quot; checked exceptions in the same way that it &quot;has&quot; generics; they are represented as meta-data in the class file but actually are not used by the JVM itself.&nbsp; It is only the compiler that deals with this.&nbsp; And a lot of exceptions in the Java are unchecked, the whole &quot;Error&quot; and &quot;RuntimeException&quot; hierarchy for instance.

</div></blockquote></div><br>The simple answer is that I'm not--at least not right now.&nbsp; I am trying to keep the scope maintainable and as such have explicitly laid out what minimal set of CLI features that I am supporting in my project proposal.&nbsp; Now, despite bowing out of implementing such features right now, I have given such things considerable thought and I have no doubt that most of them can be implemented--it is the cost of implementation that is probably the big question.
<br><br>Let me address you question a little more specifically, it is a good one.<br><ul><li>Generics will be tough, I hate to cop out and type erase like Java does, but that could be one implementation option.</li><li>Regarding un-managed pointers, there are ways you can emulate this, but because of the JVM programming model, it will come at a cost.
</li><ul><li>Others have dealt with this by doing things like paging with arrays--NestedVM (IVME '04), does this for the memory model of a MIPS R2000 ISA which it emulates.</li><li>This is not in my radar yet, because I think it would be a huge win just to support the verifiable subset of CIL.&nbsp; Un-managed pointers are not in this subset.
</li></ul><li>Regarding managed pointers, I will likely employ a boxing technique to do this.&nbsp; Managed method pointers could be handled by using reflection.</li><ul><li>These will of course take a performance hit (especially if reflection is involved), but at least we can have it.
<br></li></ul><li>Delegates are classes with methods   from the programming model point of view&nbsp; (there are CLI rules with regard to what can be in a delegate's definition, however).</li><ul><li>With regard to the run-time provided method implementations in a delegate, one approach could be to actually emit the implementation of the delegate methods.
<br></li></ul><li>Events and properties are really just meta-data &quot;sugar&quot;, there are no CIL instructions associated with them specifically and they look like methods in the programming model.<br></li><li>Although you didn't list these, here are a couple of other items.
</li><ul><li>I am actually very concerned with &quot;newslot&quot; methods, non-virtual instance method calls, and explicit interface method definition for near term future work.&nbsp; This will undoubtedly be a pain to implement; there is a lot of opportunity to make really inefficient implementations.
</li></ul><ul><li>Tail calls will be another really tough thing to implement since the JVM programming model does not have native support for tail recursive calls.&nbsp; This is probably not a highly used CLI feature anyhow (Unless you are a Scheme compiler), so it is not really on my radar at the moment because there are much larger fish to fry before I wrestle with that.&nbsp; A lot of literature on the subject--some kind of trampolining implementation may be a way to support this.
