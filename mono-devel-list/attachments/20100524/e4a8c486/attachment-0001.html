Hey folks,<div><br></div><div>The attached patch set implements block map support for sgen. It uses a schema similar to boehm&#39;s, which is a 2 level sparse map.</div><div>Under 64 bits it uses hashing.</div><div><br></div>
<div>I benchmarked a modified binary-trees without valuetypes. Block maps gives a very modest speedup under major-copying (about 2%) and</div><div>nothing under major-marksweep. I&#39;ve only used the block map for major_copy_or_mark_object thou. There are probably other places it</div>
<div>oould be used too.</div><div><br></div><div>The design is basically the same as boehm&#39;s except for a few things:</div><div><br></div><div>-It doesn&#39;t store list heads or address on each segment. This allows segment&#39;s size to be a power of 2; and</div>
<div>-LOS is handled by filling all covering slots with its block instead of using forwarding</div><div><br></div><div>Few notes:</div><div><br></div><div>Segments are not deallocated since this requires either scanning whole segments on each deallocation or keeping block counts.</div>
<div>And it&#39;s probably not needed since Boehm doesn&#39;t do it. It&#39;s doable as long as the block map is only read during GC and mutated</div><div>with the gc lock held.</div><div><br></div><div>64bits support has not been committed since it is a minor change to the code in sgen-gc.c and I want to have the current change set</div>
<div>validated first.</div><div><br></div><div>A small config option that uses either a 3 level map or just hashing under 32bits can be done with ease.</div><div><br></div><div>The embedding of Block in MSBlockInfo wastes a word of memory. This could be worked out by either factoring Block::role into a separate</div>
<div>struct or by using Block::next in place of MSBlockInfo::next.</div><div><br></div><div>Cheers,</div><div>Rodrigo</div>
