<p>Hello team,</p>
<p><br>
</p>
<p>For the last couple of nights, this has been bothering me at night time, so I figured I would take this out of my head.</p>
<p><br>
</p>
<p>A couple of years ago Paolo put together a design for inline assembly from C#.   </p>
<p><br>
</p>
<p>At the time, the design had the following constrains:</p>
<p><br>
</p>
<p></p>
<ul style="margin-bottom: 0px; margin-top: 0px;">
<li><span style="font-size: 12pt;"></span>It had to work with an out-of-the-box C# compiler, no changes to the language were needed.<br>
<br>
</li><li>We wanted a single source file to allow multiple architectures in one place<br>
<br>
</li><li>It was intended to be similar to the way inline assembly works in GNU C: you can drop some assembly language code that can be inlined in the body of a method.</li></ul>
<div><br>
</div>
<div>We ended up with something like this that you could insert in your C# code: asm.x64 ("mov $1, rax"), this was a method call in the class "asm" to the method x64 with a string.   Then a post-processor would have to extract the string, use a system assembler
 and then inject the resulting bytes into the body of the code.   The idea being that this call would be ignored in other platforms, so you could write code like:</div>
<div><br>
</div>
<p></p>
<blockquote style="margin: 0 0 0 40px; border: none; padding: 0px;">
<p></p>
<div><span style="font-family: Consolas, monospace;">// this would be a constant at runtime, so there is no actual probing/branching:</span></div>
<p></p>
<p></p>
<div><span style="font-family: Consolas, monospace;">if (asm.x64supported) {</span></div>
<p></p>
<p></p>
<div><span style="font-family: Consolas, monospace;">   asm.x64 ("mov $1, rax");</span></div>
<p></p>
<p></p>
<div><span style="font-family: Consolas, monospace;">} else if (asm.x86supported) {</span></div>
<p></p>
<p></p>
<div><span style="font-family: Consolas, monospace;">   asm.x86 ("mov $1,eax");</span></div>
<p></p>
<p></p>
<div><span style="font-family: Consolas, monospace;">} else if (asm.ppcsupported){</span></div>
<p></p>
<p></p>
<div><span style="font-family: Consolas, monospace;">   asm.ppc ("li r3,1");</span></div>
<p></p>
<p></p>
<div><span style="font-family: Consolas, monospace;">}</span></div>
<div><span style="font-family: Consolas, monospace;"><br>
</span></div>
<p></p>
</blockquote>
<p></p>
<div></div>
<div>The post-processor had to rewrite calls to asm.x64 (string) with asm.x64 (token) and inject the blob with the assembly code with the token inside.     </div>
<div><br>
</div>
<div>Now that Mono is switching to platform-specific assemblies for the core assemblies as opposed to cross-platform assemblies that work across Windows, Mac and Linux (a price we pay to converge with CoreFX) it made me wonder if we could not reuse instead
 the existing support for mixed-mode assemblies in the CLR.</div>
<div><br>
</div>
<div>Essentially, instead of having inline assembly in the body of a method, we could define entire method as inline assembly, and set the method implementation flags in the ECMA metadata to be native and point to the code.   It is not as flexible, but would
 eliminate the need for a post-processor, and would merely require the C# compiler to perform the assembly code generation and injection.</div>
<div><br>
</div>
<div>This would require some changes to the language, but it would look something like this:</div>
<div><br>
</div>
<p></p>
<blockquote style="margin: 0 0 0 40px; border: none; padding: 0px;">
<p></p>
<div><span style="font-family: Consolas, monospace;">public </span><span style="font-family: Consolas, monospace;">int</span><span style="font-family: Consolas, monospace;"> ReturnOne ()
</span><span style="font-family: Consolas, monospace;"> {</span></div>
<p></p>
<p></p>
<div><span style="font-family: Consolas, monospace;">   x86 {</span></div>
<p></p>
<p></p>
<div><span style="font-family: Consolas, monospace;">      </span><span style="font-family: Consolas, monospace;">mov $1, rax</span></div>
<p></p>
<p></p>
<div><span style="font-family: Consolas, monospace;">   }</span></div>
<p></p>
<p></p>
<div><span style="font-family: Consolas, monospace;">   ppc {</span></div>
<p></p>
<p></p>
<div><span style="font-family: Consolas, monospace;">      li r3, 1</span></div>
<p></p>
<p></p>
<div><span style="font-family: Consolas, monospace;">   }</span></div>
<p></p>
<p></p>
<div><span style="font-family: Consolas, monospace;">}</span></div>
<p></p>
</blockquote>
<p></p>
<div></div>
<div></div>
<div><br>
</div>
<div>The difference being that we would be doing assembly-language blocks at the method level.</div>
<div><br>
</div>
<div>This does not solve the issue of which assembler to use, nor which style of assembly to use.   </div>
<div><br>
</div>
<div>What we had considered in the past is to take a dependency on the LLVM assembler, another option was to write our own assembler in managed code to reduce the dependencies.   </div>
<div><br>
</div>
<div>The assembly style comes down to whether we want raw assembly language, or we want a Go-like assembly language, where they introduced sort of an abstraction layer to the underlying assembly language exposing things like "MOV" as an operation, regardless
 of what the actual underlying platform likes to call their stuff.   </div>
<div><br>
</div>
<div></div>
<div>Miguel.</div>
