<div>Based on code inspection it looks to me like Mono is partially vulnerable.</div><div> </div><div>There are/were two basic problems with the MS implementation:</div><div> </div><div>1) when an incorrect string is passed in the &#39;d&#39; paramater to WebResource.axd, which is processed by System.Web.Handlers.WebResourceHandler.ProcessRequest, the exception that is thrown depends on whether the padding in the decrypted string is correct or not.  This is the padding oracle that lets an attacker encrypt (or decrypt) arbitrary strings with the key used by the oracle.</div>
<div><br></div><div>Mono&#39;s implementation appears to also have this problem in both WebResourceHandler and ScriptResourceHandler.</div><div><br></div><div>The fix for this is to include a MAC as part of the request string.</div>
<div><br></div><div><br></div><div>2) given an encrypted string representing a file ScriptResource.axd, which is processed by System.Web.Handlers.ScriptResourceHandler.ProcessRequest, allows for the download of arbitrary files from within the applications Virtual Path.</div>
<div><br></div><div>Mono&#39;s implementation doesn&#39;t appear to allow for anything other than embedded resources to be downloaded through this path and so isn&#39;t vulnerable.</div><div><br></div><div>Dan Witt</div><div>
