<div>currently there are 4 interfaces for rules: IAssemblyRule, IModuleRule, ITypeRule and IMethodRule. Each one declares a method which looks like this:</div>
<div>&nbsp;</div>
<div>IList Check&lt;Type&gt; (IAssemblyDefinition ad, ...);<br>&nbsp;</div>
<div>The return value IList contains a list of objects. If this return value is null it means that there's no violation. If it is non null, then &lt;Type&gt; has violated this rule and the list may contain a list of objects with detailed information. Currently there are two types of information, a simple string (Example: &quot;Found string: \r\n&quot;) or a complex object of type &quot;Message&quot; - but the interface allows a rule to return everything it wishes to return. This makes it really, really difficult for runners to determine what to &quot;print&quot; to the user since it doesn't know at designtime what a rule might return (imho missing type safety). Therefore I'd like to&nbsp;propose&nbsp;a change into a return value of a typed collection based on the Message object. 
</div>
<div>&nbsp;</div>
<div>An example might by:</div>
<div>&nbsp;</div>
<div>IMessageCollection Check&lt;Type&gt; (IAssemblyDefinition ad, ...)</div>
<div>&nbsp;</div>
<div>This would also eleminate the ability of using a string, which IMHO is good thing since now every rule needs to provide a text and location (like type and offset).</div>
<div>&nbsp;</div>
<div>I'd like to here what other think of this change.</div>
