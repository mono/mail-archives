We now have a `make doc-update` target in the makefiles, which will<br>

Now, mdoc-update (which generates/updates monodoc documentation) has XML<br>
documentation import support via the -import argument.<br>
<br>
And gmcs has the -doc flag to generate XML documentation.<br>
<br>
Putting these together, and it&#39;s quite trivial to always generate an XML<br>
documentation file and have `make doc-update` import this file (~3 LOC<br>
to the current build system, iirc).<br>
<br>
There is just one &quot;minor&quot; problem with it: importing is currently an<br>
overwrite operation, not an appending operation. &nbsp;Consequently, if you<br>
have existing documentation, then import documentation for the *same*<br>
member, you lost the original documentation.<br>
<br>
(Normally this is the right thing to do, as if you always appended you&#39;d<br>
always get duplicated documentation each time you imported...)<br>
<br>
The problem in this case is mscorlib (and possibly other assemblies; I<br>
only looked at mscorlib): there is XML documentation on some members<br>
within the source files that would replace the existing documentation<br>
(which was imported from the ECMA standard), and the imported version is<br>
*far* worse than the current documentation.<br>
<br>
Case in point: mcs/class/corlib/System.Reflection/CallingConventions.cs,<br>
which contains &lt;summary/&gt; elements (i.e. an empty element), thus<br>
replacing *actual* documentation with *nothing*. &nbsp;Oops.<br>
<br>
So much as I would *love* to enable XML documentation importing by<br>
default (thus allowing people to write documentation *either* within<br>
monodoc format OR with XML documentation comments), this isn&#39;t currently<br>
possible because the net result would be detrimental.<br>
