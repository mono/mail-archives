<div>&nbsp;</div>
<div>Consider the following code:</div>
<div>using (Stream s = FileStream.Create()) {</div>
<div>...</div>
<div>}</div>
<div>&nbsp;</div>
<div>If you abort the thread just after the file handle is opened, but before that function returned, the stream will remain opened. In some cases even Finalizer won&#39;t close it. Thus you must restrict anything that may need explicit disposing. (In general same is true even with lock statement. As far I know MS CLR explicitly frees all the locks held by the aborted thread, but I don&#39;t know how mono CLR behaves in this case).</div>

<div>&nbsp;</div>
<div>Now consider this:</div>
<div>&nbsp;</div>
<div>Object x = null;</div>
<div>try {</div>
<div>&nbsp; x = new Something();</div>
<div>}</div>
<div>finally {</div>
<div>&nbsp; x.Whatever();</div>
<div>}</div>
<div>&nbsp;</div>
<div>If Something ctor never throws, normally, the user code will always run without exceptions. Now you come and abort it inside Something ctor -&gt; NullRef Exception is thrown!</div>
<div>&nbsp;</div>
<div>So the generalization is that you must restrict usage of try/catch/finally blocks. This will imply restriction on objects that need explicit disposing, since without try/catch/finally blocks&nbsp;they cannot be correctly implemented.</div>
