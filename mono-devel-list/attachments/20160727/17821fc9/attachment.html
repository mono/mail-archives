<div dir="ltr">I personally think it is a terrible idea to make Mono completely unable to run code that compiles and runs just fine on Microsoft's .NET framework. Could <span style="color:rgb(189,38,13);font-family:Consolas,"Liberation Mono",Courier,monospace;font-size:15.2px;line-height:22.8px;background-color:rgb(238,238,238)">get_OffsetToStringData</span> be made to convert the ASCII representation back to UCS-2 on-the-fly for that edge case where the code actually uses the <span style="color:rgb(189,38,13);font-family:Consolas,"Liberation Mono",Courier,monospace;font-size:15.2px;line-height:22.8px;background-color:rgb(238,238,238)">fixed (char *ptr = str)</span> pattern? It's not a very common pattern, so the overhead of the conversion, while defeating the purpose of using that pattern in the first place, would affect only the tiniest minority of code.<div><br></div><div>If this were the direction taken, it might be nice also to provide a way to force an ASCII-capable string to be UCS-2 anyway, in case there are people who want the <span style="color:rgb(189,38,13);font-family:Consolas,"Liberation Mono",Courier,monospace;font-size:15.2px;line-height:22.8px;background-color:rgb(238,238,238)">fixed (char *ptr = str)</span> pattern to remain performant -- perhaps an environment variable?? Obviously we wouldn't want the Mono runtime to scan the environment block every time it allocates a string, so perhaps it could do the check & cache the result once on startup, and then allow some innocuous method that's already doing a lot of work, such as <span style="color:rgb(189,38,13);font-family:Consolas,"Liberation Mono",Courier,monospace;font-size:15.2px;line-height:22.8px;background-color:rgb(238,238,238)">string.IsInterned</span>, to re-check it. This avoids adding Mono-specific API, so that code written to be aware of Mono's peculiarity still runs just fine on other frameworks.</div><div><div><br></div><div>Jonathan Gilbert</div></div></div><div class="gmail_extra"><br <p style="background:white"><span><span style="font-family:Calibri;color:black">I have written a description of my prototype implementation of adaptive ASCII/UTF-16 strings in Mono:</span></span><span style="font-family:Calibri;color:black"><u></u><u></u></span></p>

<p style="background:white"><span style="font-family:Calibri;color:black"><u></u> <u></u></span></p>
<p style="background:white"><span style="font-family:Calibri;color:black">Introduction:<u></u><u></u></span></p>
<p style="background:white"><span style="font-family:Calibri;color:black"><u></u> <u></u></span></p>
<p style="background:white"><span><span style="font-family:Calibri;color:black">> For historical reasons, System.String uses the UCS-2 character encoding, that is, UTF-16 without surrogate pairs.</span></span><span style="font-family:Calibri;color:black"><u></u><u></u></span></p>
<p style="background:white"><span style="font-family:Calibri;color:black"><u></u> <u></u></span></p>
<p style="background:white"><span><span style="font-family:Calibri;color:black">> However, most strings in typical .NET applications consist solely of ASCII characters, leading to wasted space: half of the bytes in a string are likely
 to be null bytes!</span></span><span style="font-family:Calibri;color:black"><u></u><u></u></span></p>
<p style="background:white"><span style="font-family:Calibri;color:black"><u></u> <u></u></span></p>
<p style="background:white"><span><span style="font-family:Calibri;color:black">> Since strings are immutable, we can scan the character data when the string is constructed, then dynamically select an encoding, thereby saving 50% of string
 memory in most cases.</span></span><span style="font-family:Calibri;color:black"><u></u><u></u></span></p>
