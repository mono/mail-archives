<DIV><FONT face=Arial size=2>Hi,</FONT></DIV>
<DIV><FONT face=Arial size=2></FONT>&nbsp;</DIV>
<DIV><FONT face=Arial size=2>right now there is nearly no localization support 
in the Mono class libraries and all strings (mainly for errors) are hardcoded 
into the source-files.</FONT></DIV>
<DIV><FONT face=Arial size=2>If someone wants to localize these strings, it is 
nearly impossible if we do not add the required support now. Moreover coding all 
the strings into the source - files and compiling them adds to the size of the 
libraries, wich not only (probably) degrades general performance, but also will 
make it hard to make extra-small versions of Mono (e.g. for mobile 
devices).</FONT></DIV>
<DIV><FONT face=Arial size=2></FONT>&nbsp;</DIV>
<DIV><FONT face=Arial size=2>As rationale just a simple example:</FONT></DIV>
<DIV><FONT face=Arial size=2>Say you want to localize the whole libraries and 
use a technique like currently used in the System Assembly. Something 
like:</FONT></DIV>
<DIV><FONT face=Arial size=2>&nbsp;&nbsp;public static string GetText (string 
msg)<BR>&nbsp;&nbsp;{</FONT></DIV>
<DIV><FONT face=Arial size=2>// Do localization<BR>&nbsp;&nbsp;&nbsp;return 
LocalizedString;<BR>&nbsp;&nbsp;}</FONT></DIV>
<DIV><FONT face=Arial size=2>with a call like:</FONT></DIV>
<DIV><FONT face=Arial size=2>Print (FromWherever.GetText ("This text will be 
printed somewhere"));</FONT></DIV>
<DIV><FONT face=Arial size=2>and assuming that all strings in the class 
libraries are about 500KB (and I personally&nbsp;really like verbose error 
messages, so there might even be more - 1 MB?)</FONT></DIV>
<DIV><FONT face=Arial size=2></FONT>&nbsp;</DIV>
<DIV><FONT face=Arial size=2>Using the scheme above you will need: the strings 
hardcoded in the sources, a hashtable to lookup with the original values and the 
replace values, per supported fallback level (e.g. en-US -&gt; en) at 
least&nbsp;one additional set which makes 4 Times all the strings -&gt; 2 MB - 4 
MB RAM just wasted the first time you try to access a single string of a few 
bytes size.</FONT></DIV>
<DIV><FONT face=Arial size=2>If you write a small tool it is guaranteed to use 
megs of RAM</FONT></DIV>
<DIV><FONT face=Arial size=2>Moreover we are not just talking about memory, but 
the required lookups will need an possibly considerable amount of 
time.</FONT></DIV>
<DIV><FONT face=Arial size=2></FONT>&nbsp;</DIV>
<DIV><FONT face=Arial size=2>I would therefore suggest the 
following:</FONT></DIV>
<DIV><FONT face=Arial size=2></FONT>&nbsp;</DIV>
<DIV><FONT face=Arial size=2>Replace all hard-coded strings with an enumeration 
value. This gets compiled into an int in IL and allows much more efficient 
lookups ( O(1) instead of X * O (ln n) for the previous solution.</FONT></DIV>
<DIV><FONT face=Arial size=2>The strings are stored in an external XML file and 
are compiled through a simple program into a binary data format (using an 
integer&nbsp;lookup index)</FONT></DIV>
<DIV><FONT face=Arial size=2>This reduces (compiled) code size and RAM 
usage&nbsp;and improves&nbsp;speed cosiderably, also is much more extensible and 
easily translatable.</FONT></DIV>
<DIV><FONT face=Arial size=2></FONT>&nbsp;</DIV>
<DIV><FONT face=Arial size=2>&nbsp;public static string GetString (MonoString 
stringResource)<BR>&nbsp;{<BR>&nbsp;&nbsp;return StringForMonoStringEnumValue 
(MonoString);<BR>&nbsp;}</FONT></DIV>
<DIV>&nbsp;</DIV>
<DIV><FONT face=Arial size=2>&nbsp;public enum 
MonoString<BR>&nbsp;{<BR>&nbsp;&nbsp;GenericENullNotAllowed,<BR>&nbsp;&nbsp;GenericEValueMustBeLarger0,<BR>&nbsp;&nbsp;GenericEValueMustBeLargerOrEqual0<BR>&nbsp;}<BR></FONT></DIV>
<DIV><FONT face=Arial size=2>with a call like</FONT></DIV>
<DIV><FONT face=Arial size=2>Print 
(MS.GetString&nbsp;(MonoString.GenericENullNotAllowed));</DIV></FONT>
<DIV><FONT face=Arial size=2></FONT>&nbsp;</DIV>
<DIV><FONT face=Arial size=2>All strings</FONT></DIV>
<DIV><FONT face=Arial size=2></FONT>&nbsp;</DIV>
<DIV><FONT face=Arial size=2>The Advantages are:</FONT></DIV>
<DIV><FONT face=Arial size=2>* Smaller Assemblies (probably leads to faster 
runtime performance in Jit also because Jiting a constant&nbsp;int should be 
faster than Jiting a constant&nbsp;string)</FONT></DIV>
<DIV><FONT face=Arial size=2>* Far faster String lookups</FONT></DIV>
<DIV><FONT face=Arial size=2>* Far less memory usage (ideally we practicly 
wouldn't need ANY memory overhead)</FONT></DIV>
<DIV><FONT face=Arial size=2>* More flexible solution (allows simple changes 
from entirely removing all strings for extra-small versions, to fully cached 
versions)</FONT></DIV>
<DIV><FONT face=Arial size=2>* Even in worst case with&nbsp;full 
caching&nbsp;of&nbsp;all strings&nbsp;memory usage is is only a fraction of what 
it would be now</FONT></DIV>
<DIV><FONT face=Arial size=2>* Easily extensible</FONT></DIV>
<DIV><FONT face=Arial size=2>* Data would be created outside&nbsp;from an XML 
file, which e.g. is easily translatable</FONT></DIV>
<DIV><FONT face=Arial size=2>* Comparable speeds when using localization or 
not</FONT></DIV>
<DIV><FONT face=Arial size=2></FONT>&nbsp;</DIV>
<DIV><FONT face=Arial size=2>The Disadvantages:</FONT></DIV>
<DIV><FONT face=Arial size=2>* Need to change existing code (for most strings 
this is true anyway if localization is needed)</FONT></DIV>
<DIV><FONT face=Arial size=2>* More programing work needed:</FONT></DIV>
<DIV><FONT face=Arial size=2>&nbsp;&nbsp;&nbsp; To add a new string you would 
have to:</FONT></DIV>
<DIV><FONT face=Arial size=2>&nbsp;&nbsp;&nbsp; &nbsp;&nbsp;&nbsp; Add a code 
line (that is the code line you would also have to write normally</FONT></DIV>
<DIV><FONT face=Arial size=2>&nbsp;&nbsp;&nbsp; &nbsp;&nbsp;&nbsp; Add an 
enumeration member to MS.Strings (additional)</FONT></DIV>
<DIV><FONT face=Arial size=2>&nbsp;&nbsp;&nbsp; &nbsp;&nbsp;&nbsp; Add the 
key/string value to an XML file (additional)</FONT></DIV>
<DIV><FONT face=Arial size=2>* Compiling gets a (little) more complicated 
because the string resources file has to be (auto)generated in addition with 
each compile</FONT></DIV>

<DIV><FONT face=Arial size=2>Open&nbsp;questions are:</FONT></DIV>
<DIV><FONT face=Arial size=2>* Should every class get its own internal version 
of the localization support (would help to not change the public signature of 
the assembly, but possibly values between assemblies would have to be stored 
multiple times (things like GenericErrorNullNotAllowed will probably 
be&nbsp;needed in every assembly)</FONT></DIV>
<DIV><FONT face=Arial size=2>* Should data files be added to the assemblies as 
resources or stay independently (independently would allow easy switch of 
language by simply overwriting the file, but also adds the risk of invalid/ 
tampered&nbsp;files)</FONT></DIV></BODY></HTML>
