Stack overflow due to 18368 stack frames caused by the recurvise 
quicksort implementation in sgen-qsort.c. The application is creating a 
high number of short lived objects, and the memory is badly fragmented 
(229138 entries in freelist...). Stack size has already been increased 
to 16M, and GC nursery size is set to 2G to cope with the high number of 
temporary objects, which keeps the number of mmap'ed fragments lower (~ 
60.000 instead of ~120.000).

Does mono honor the system stack size limit (and thus allows larger 
stacks for larger values of ulimit -s)?
