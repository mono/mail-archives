Hey,

On 2/20/09, Sebastien Pouliot <sebastien.pouliot at gmail.com> wrote:
> After some talk with TuningMaster JB Evain we came to the conclusion
>  that it would be simpler to move the tuning process into moon itself.

I finished the move last friday. The tuning process now resides in

moon/class/tuning

And is called prior to build System.Windows and
System.Windows.Browser, but only if the net_2_1_raw assemblies in mcs
have changed since the last tuning.

To actually tune the net_2_1_raw assemblies, the tuning process needs
to get some assemblies from the mcs module, at this time they are:

monolinker.exe
Mono.Tuner.dll
Mono.Cecil.dll
Mono.Cecil.Mdb.dll

They are copied to moon/class/lib/tuner.

When make enters into moon/class/tuning, it will simply trigger the
same tuning process it was doing in mcs, but it will output all
Moonlight assemblies in moon/class/lib/2.1. We may have still have
some issues in the Makefile where we used to set MONO_PATH to
moon/class/lib/2.1, only for Mono to find System.Windows.dll and
System.Windows.Browser.dll. Now if we do that, mono will run using the
mscorlib and System assemblies in moon/class/lib/2.1.

>  In technical terms it means that:
>
>  * /moon will need to know a bit more about /mcs, i.e. its location to
>  get the 2.1 assemblies (defaulting to ../mcs from moon)

This is now configurable using the --with-mcspath switch when
configuring moon. It indeeds default to ../mcs from moon, but can be
specified otherwise (for the buildbots for instance).
