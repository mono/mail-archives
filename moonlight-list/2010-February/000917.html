> We'd like our profile to be a
> subset of Silverlight 2 (minus the GUI stuff as we're running server
> side). 

A subset will definitively make your job a _lot_ easier (since you'll be
inheriting the security of SL class library design).

The tools will mostly (98%*) adjust the security attributes to match
your own profile. So you will need to review (at least**) any extra
[SecuritySafeCritical] that your changes produce (since it's a subset it
should be minimal or even an empty set).

        * use of reflection is not covered by the tools and needs some
        manual tweaking
        
        ** more if you don't completely trust our own review ;-)

> We'd also like to use the Mono tools to build the assemblies
> from the Mono class libraries in the same way that Moonlight builds
> its libraries, so pointers to those tools and process would be great.

It's fairly simple and mostly contained in the following Makefile:
http://anonsvn.mono-project.com/viewvc/trunk/moon/class/tuning/SecurityAttributes/Makefile.am

> > The current state (in mono 2.6 branch) is very stable and unlikely to
> > change much API-wise. I suggest you start with this then build your own
> > class libraries (profile) with the coreclr requirements - that later
> > part (and its audit) will likely take some time (and Moonlight 2.0 will
> > likely have shipped by then).
> 
> We're currently planning to embed Mono 2.6.1 and track Mono 2.6 for bug fixes.

You'll also want to track the verifier/JIT bugs (on mono's bugzilla)
since they can be far more critical in a server environment than on a
client/plugin environment.

E.g. in our case a crash (leading to a DoS) only affects the Firefox
instance of a single user (sad but minor in comparison to running
invalid, and untrusted, code) while you know better than me what the
same crash on a server could produce.

Have fun!
