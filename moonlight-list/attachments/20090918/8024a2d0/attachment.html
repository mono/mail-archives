Hi,<br>
<br>
Unfortunately I&#39;ve found a major issue with out shutdown design, which<br>
loosely goes like this<br>
<br>
1) firefox notifies us that the plugin has been removed from the page<br>
2) we do managed cleanup: (Deployment::Dispose () is called) which calls<br>
mono_domain_finalize (which will finalize every single object in that<br>
appdomain)<br>
3) we do native cleanup: we wait until the managed objects have freed their<br>
native refs, which will cause a cascading effect of native objects unreffing<br>
other native objects.<br>
4) the last object alive is the Deployment instance, and in<br>
Deployment::~Deployment  we call mono_domain_unload (which will abort all<br>
managed threads)<br>
<br>
The problem is in 2): mono_domain_finalize assumes all threads running in<br>
the domain have already been aborted. Since that&#39;s not done until 4), a lot<br>
of weird things happen in those threads (mostly crashes of course, but also<br>
weirds NREs).<br>
<br>
The reason we can&#39;t call mono_domain_unload in 2) (and skip 4) completely),<br>
is that we might end up invoking managed delegates in 3), which would crash<br>
too.<br>
<br>
I&#39;m able to reproduce these issues with <a href="http://hdbeta.ctv.ca/" target="_blank">http://hdbeta.ctv.ca/</a> 99% of the<br>
time.<br>
<br>
I can think of 2 solutions:<br>
<br>
a) Somehow guarantee that we won&#39;t call into managed code in 3), and call<br>
mono_domain_unload in 2) instead of mono_domain_finalize. Possibly by having<br>
a &#39;shutting_down&#39; flag on the Deployment instance that can be checked before<br>
calling function pointers (event emitting being the biggest I think, but<br>
there are quite a few others too). Or maybe add managed finalizers to clear<br>
out the native function pointer (this may have threading issues though since<br>
finalizers aren&#39;t run on the main thread).<br>
<br>
b) Expose a method in the mono runtime that aborts all the managed threads,<br>
and do that just before calling mono_domain_finalize. However I have a<br>
feeling that the runtime folks won&#39;t think this is a good idea ;-) To me it<br>
feels like a dirty hack, especially since we&#39;d basically duplicate<br>
mono_domain_unload.<br>
