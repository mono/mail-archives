<div dir="ltr">I replaced the --with-mono configure arg with --with-managed.&nbsp; The possible values (--with-managed=$value) are:<br><br><ul><li>desktop: attempts to configure in support for just desktop assemblies (the &quot;3.0&quot; profile.&nbsp; GtkSilver is here)</li>
<li>browser: attempts to configure in support for just browser assemblies (the &quot;2.1&quot; profile.&nbsp; This enables silverlight2 browser plugin support)</li><li>yes (the default when you use &quot;--with-managed&quot;):&nbsp; probes for dependencies for both.</li>
<li>no (specified either as &quot;--with-managed=no&quot; or &quot;--without-managed&quot;): disables managed code entirely.<br></li></ul>If you don&#39;t have the right dependencies installed for the 2 options (gtk-sharp-2.0 and rsvg-sharp-2.0 for desktop, mono &gt;= 1.9 with smcs for browser), the support still won&#39;t be enabled, and if neither desktop nor browser dependencies are available, configure will still disable managed code entirely.&nbsp; It&#39;ll tell you what&#39;s going on in the final configuration output, though.<br>
<br>for hackers:<br><br>This also requires differentiating between Silverlight2 support in the src/ and plugin/ directories.&nbsp; In the case where moonlight is configured --with-managed=desktop, the plugin will be built only to support 1.0 (since we won&#39;t be compiling the 2.1 assemblies).&nbsp; So, any code that needs to be ifdefed in the plugin for 2.0-ness should use the new SL_PLUGIN_2_0 define.&nbsp; This will be defined iff the 2.1 assemblies are being compiled.&nbsp; The old SL_2_0 ifdef should still be used in src/, and will be defined if either 2.1 or 3.0 assemblies are being built.<br>
