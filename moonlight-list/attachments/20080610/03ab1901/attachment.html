I made a few changes, mostly to isolate the Silverlight 2.0 stuff that&#39;s in the source tree now from the 1.0 stuff.&nbsp; As it stands at the moment, there should be zero 2.0 specific code/types/anything included in the 1.0 build.&nbsp; This means both managed and unmanaged.&nbsp; Before we built all the 2.0 stuff and just left out the hooks to run 2.0 content in the 1.0 build.&nbsp; But there&#39;s no point in including additional code that&#39;s never going to run in a released 1.0 (nor is there a reason to waste time compiling the files at all in a 1.0 environment.)<br>
<br>Keeping the managed side segregated is easy: We don&#39;t build any of it if 2.0 isn&#39;t enabled.&nbsp; The unmanaged side isn&#39;t as easy though, so here&#39;s some conventions I&#39;ve implemented:<br><br><ol><li>All 2.0 specific classes/methods/files/blocks of code/etc are wrapped in #if INCLUDE_MONO_RUNTIME.&nbsp; This was mostly partially done before.</li>
<li>All 2.0 specific dependencyobject descendants need a special typegen pragma of the form:&nbsp; /* @SilverlightVersion=&quot;2&quot; */ on the line above the class declaration in their header files.&nbsp; If you grep for this string you&#39;ll find a number of examples.&nbsp; This enables the typegen machinery to wrap the generated code (and enum values) for these types in the proper #if INCLUDE_MONO_RUNTIME.</li>
<li>Files that are 2.0 specific are listed in a special block in src/Makefile.am, so they aren&#39;t compiled at all.&nbsp; Examples include grid.cpp, xap.cpp, deployment.cpp.<br></li></ol>The reason 2 is still needed even with 3 above is that typegen isn&#39;t smart enough (and it would be a pain to make it smart enough) to deal with 1.0 vs 2.0 files, and some files (control.h) have both 1.0 and 2.0 classes in the same file.<br>
<br>I&#39;m planning to go through and replace INCLUDE_MONO_RUNTIME with a better ifdef.&nbsp; There are a couple of possibilities:<br><ol><li>SILVERLIGHT_VERSION, which we can use in comparisons, like &quot;#if SILVERLIGHT_VERSION &gt;= 2&quot;</li>
<li>a scheme more like mcs/ uses, where we&#39;ll define SILVERLIGHT_2 in 2.0 builds, SILVERLIGHT_2 and SILVERLIGHT_3 in 3.0 builds, etc.</li></ol>Anyone have an opinion on which one to use?<br><br>chris<br>
