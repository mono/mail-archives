&gt; Hello,<br>
&gt;<br>
&gt; I see no activity in the mailing list and github; Silverlight 5 is<br>
&gt; already released, Moonlight is two versions behind; does it all mean<br>
&gt; that Moonlight is dead? Will there be any successor for moonlight or<br>
&gt; we may consider silverlight as dead platform for cross-platform<br>
&gt; webapplications?<br>
&gt;<br>
&gt;<br>
&gt; None of the developers that used to work full time on Moonlight are working<br>
full time on Moonlight anymore.<br>
&gt;<br>
&gt; Some work for Xamarin now, some have gone on different paths.<br>
&gt;<br>
&gt; Some of the work that we did for Moonlight improved Mono in many significant<br>
ways (a stable IL verifier, a working sandbox, coreclr security, error recovery<br>
and much more).   Some of the other work I believe is a gold mine of technology,<br>
in particular our media pipeline, the rendering/layout code and the animation<br>
framework.   Perhaps someone would find a use for that code in the future, we<br>
currently do not have plans to continue developing Moonlight.<br>
&gt;<br>
&gt; There are several reasons.   From a community perspective, we never managed to<br>
get much traction to get contributions to improve the stack and share the<br>
development effort with.  This had the very negative side effect that without a<br>
sponsored team, Moonlight development would quickly stop.<br>
&gt;<br>
&gt; Although Microsoft released some key bits of the style and controls from<br>
Silverlight 2 to use in Moonlight 2, the more advanced and complex components<br>
for supporting 3 and 4 were not yet open sourced, so those areas in Moonlight<br>
suffer as an effect.<br>
&gt;<br>
&gt; The Silverlight vision was on a great path (despite some bloat making it into<br>
its design, or some very complicated and over-engineered bits of technology<br>
making it into the stack).   But key components that would have allowed<br>
Silverlight to become a real cross-platform framework were cut from the<br>
releases, like full File System Access which just doomed Silverlight to be a<br>
web-only plugin.   Silverlight&#39;s fate was sealed when they started supporting<br>
Windows-only features in the plugin.<br>
&gt;<br>
&gt; I felt that Silverlight had a bright future, and that it could turn to fill an<br>
important void, not only for web development, but for desktop development in<br>
general.  And this was largely one of my motivators.   I am very sad that<br>
Microsoft strategy cut the air supply to Silvelright.<br>
&gt;<br>
&gt; Finally, Microsoft seems to be going back to a Windows-only vision of the<br>
world.   They have a third reimplementation of the XAML/UI stack, this time on<br>
top of WinRT.  <br>
&gt;<br>
&gt; I do not think that a full open source WinRT stack makes sense at this point<br>
for two reasons:  (a) the world has changed and developers now believe in<br>
writing the business logic once, and multiple UIs (for example, a desktop UI and<br>
a web UI, or a desktop UI and a mobile UI) and (b) because judging by previous<br>
experience, we would hardly get contributors, we would be 12-24 months late to<br>
get some apps on Mac/Linux.<br>
&gt;<br>
&gt; At this point, I just advise developers to write their apps by splitting their<br>
UI from their engine code.   This ensures that you will have a great UI<br>
experience on every platform (Mac, Linux, Android, iOS, Windows and Web) and<br>
that the only thing preventing you from shipping your code is not a third party<br>
catching up to the latest Microsoft APIs, but yourself.<br>
&gt;<br>
