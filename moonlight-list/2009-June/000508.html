Well to be clear to everyone: reflection itself works but Invoke-ing (or
[Get|Set]ting values) the result does not always wor. You can see some
of moonlight's tests in:

        /moon/tests/2.0/moon-unit/security/ReflectionTest.cs

But you're right about private members - they are *really* private.
Otherwise reflecting/invoking on internal methods is ok.

> I can see why private members in platform code aren't allowed to be 
> touched, as one could interfere
> with the management of SafeCritical code.   

No it would not. Reflection is aware of the CoreCLR security attributes.

> However for non platform 
> code, is there a good reason
> why the silverlight security model does not allow reflection on private 
> members?

Well you're not asking the right team ;-) so take my own guesses with a
grain of salt.

The reflection/invoke is limited to what normal (non-reflecting) code
could do (visibility-wise), because:

        * it simplifies code auditing (IMHO the major goal behind the
        transparency model on which CoreCLR expands/customize a bit)
        since you don't have to worry about external exposure on private
        methods;
        
        * it's easy to fix, without breaking binary compatibility
        (promoting privates to internals if required). So the previous
        gain is not done by causing (much) new pains;
        
        and
        
        * it's the simplest way to explain it (i.e. it removes the
        different rule that the normal FX has wrt reflection);
