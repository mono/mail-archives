<p class=MsoNormal><span lang=EN-GB>Hello,<o:p></o:p></span></p>

<p class=MsoNormal><span lang=EN-GB>I want to develop a control that allows the
user to switch language for the website, while its default is still detected
using browser&#8217;s headers. I want it to scan the current assembly to find
if there have been deployed satellite assemblies in order to display the user
only the list of languages actually available, without recompiling the code
every time a new language is added. I do localization using local resources and
satellite assemblies, the simplest way.<o:p></o:p></span></p>

<p class=MsoNormal><span lang=EN-GB><o:p>&nbsp;</o:p></span></p>

<p class=MsoNormal><span lang=EN-GB>What I&#8217;m stuck in is the method that
does the scan: its behaviour differs between Visual Studio development server
and Mono (I&#8217;ll consider opening a bug ticket if you tell me that my code is
correct).<o:p></o:p></span></p>

<p class=MsoNormal><span lang=EN-GB><o:p>&nbsp;</o:p></span></p>

<p class=MsoNormal><span lang=EN-GB>Basically, when you compile your Console/WinForms
application either under Windows or with xbuild, satellite assemblies are
deployed as Appname.resources.dll inside a subdirectory of bin directory named
as language code (ie. bin/fr/Appname.resources.dll for French).<o:p></o:p></span></p>

<p class=MsoNormal><span lang=EN-GB>When you compile an ASP.NET application
with Visual Studio the same happens.<o:p></o:p></span></p>

<p class=MsoNormal><span lang=EN-GB>However, I found that xbuild does not create
satellite assemblies, however localization still works on my app.<o:p></o:p></span></p>

<p class=MsoNormal><span lang=EN-GB><o:p>&nbsp;</o:p></span></p>

<p class=MsoNormal><span lang=EN-GB>So, let&#8217;s come to the code and its differences

<p class=MsoNormal><span lang=EN-GB>First of all, I add the current assembly&#8217;s
language, usually English (</span><span lang=EN-GB style='font-family:Wingdings'>J</span><span
lang=EN-GB>), to the list, then start scanning the CodeBase directory. I use
substring(6) because CodeBase is expressed as URI (file://...) and it won&#8217;t
be accepted by Path class. If there exists Appname.resources.dll I add the
directory&#8217;s name to the list.<o:p></o:p></span></p>

<p class=MsoNormal><span lang=EN-GB><o:p>&nbsp;</o:p></span></p>

<p class=MsoNormal><span lang=EN-GB>Now, I develop under Windows using a local
directory such as E:\MyApp. This code WORKS FINE, as the codebase coincides
with the bin/ directory of the website, and the code is also reusable for Winforms
applications.<o:p></o:p></span></p>

<p class=MsoNormal><span lang=EN-GB>When running the same code on Mono, using
an independent instance of mod-mono-server2 serving Apache via TCP socket, I
found that the codebase is moved to /tmp/djechelon-temp-aspnet-0 and its
subdirectories rather than the application&#8217;s path as seen by mod-mono.
This alters the codebase, and I also found, by exploring it, that assemblies
are deployed to leaf directories.<o:p></o:p></span></p>

page with a few translations, then trying to list all the available languages
on-screen using this method.<o:p></o:p></span></p>
