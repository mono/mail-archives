<B><FONT SIZE="6">Mono 1.1.9 Release</FONT></B><BR>
We consider Mono 1.1.9 stable enough to recommend it for all users. Those upgrading from the 1.0.x series should note that these notes only contain the differences between 1.1.8 and 1.1.9. All of the changes since 1.0 are documented in the following release notes: <A HREF="http://www.go-mono.com/archive/1.1.1">1.1.1</A>, <A HREF="http://www.go-mono.com/archive/1.1.2">1.1.2</A>, <A HREF="http://www.go-mono.com/archive/1.1.3">1.1.3</A>, <A HREF="http://www.go-mono.com/archive/1.1.4">1.1.4</A>, <A HREF="http://www.go-mono.com/archive/1.1.5">1.1.5</A>, <A HREF="http://www.go-mono.com/archive/1.1.6">1.1.6</A>, <A HREF="http://www.go-mono.com/archive/1.1.7">1.1.7</A>. <A HREF="http://www.go-mono.com/archive/1.1.7">1.1.8</A>. <BR>

<H3>
<B><FONT SIZE="4">Globalization/Internationalization: String Collation.</FONT></B>
</H3>
We have a completely new reimplementation of the CompareInfo infrastructure in this release of Mono, a managed implementation of string collation that is compatible with Windows collation. <BR>
<BR>
Atsushi Enomoto worked on this project for the past four months before we merged it on this release. Currently the code has to be turned on by setting the MONO_USE_MANAGED_COLLATION environment variable to &quot;yes&quot; <BR>
<BR>
In the past we had used ICU but this approach had two problems: the code lived in the C world and the cost of transitioning from managed to unmanaged code for string collation was fairly high. <BR>
<BR>
ICU also implemented different semantics than those exposed by .NET and a mapping of one system into the other was not really possible. <BR>
<BR>

<B><FONT SIZE="4">ASP.NET</FONT></B>
</H3>
A major rewrite to ASP.NET is now available as part of this release, the highlights of the new code include: <BR>
<UL>
    <BR>
    <LI>Tests: 67,700 lines of new tests: 
    <UL>
        <LI>NUnit test suite for about 50% of the controls. 
        <LI>Extensive standalone tests. 
        <LI>JSUnit (see section later). 
    </UL>
    <LI>Unmanaged I/O: the new implementation uses unmanaged buffers for uploads (HTTP POST for example) and content generation as opposed to the managed buffers that we have today, which greatly reduces the pressure on Mono's GC and also avoids redundant copies of data by sharing buffers as much as possible improving performance. 
    <LI>Use of TCP Cork on Linux to avoid TCP glitches and delays, this reduces the latency to get a full page. 
    <LI>Support for Linux sendfile to transfer static pages (support for more platforms will come later). 
    <LI>XSP now transfers Socket ownership to the AppDomain to avoid round trips and expensive AppDomain boundary crossing increasing performance. 
    <LI>New controls: about 40% of the existing controls were rewritten from scratch with test suites to validate their output. 
    <LI>New application pipeline: a new iterators-based design reduces the complexity and increases the maintainability of the old version. 
    <LI>Support for HttpClientCertificate on XSP, soon to come to Apache. 
    <LI>Improved tracing support. 
    <LI>Latency has been reduced in various key places and the new unmanaged buffers accelerate the processing of medium and large sized pages (small pages remain about the same speed) and large uploads wont disrupt your Mono process. 
</UL>
<BR>
The new ASP.NET stack is brought to you by Eyal Alaluf, Peter Bartok, Jackson Harper, Miguel de Icaza, Ben Maurer, Jordi Mas, Gonzalo Paniagua, Dick Porter, Sebastien Pouliot and Chris Toshok. <BR>

<H3>
<B><FONT SIZE="4">XSP Web Server</FONT></B>
</H3>
XSP has been split up in two: Mono.WebServer.dll and xsp.exe. XSP only handles command line parsing and Mono.WebServer.dll is the assembly that implements the functionality. <BR>
<BR>
Mono.WebServer.dll is an embeddable library that can be used to host ASP.NET in your own applications. In the past people resorted to making a replica of XSP in their applications if they wanted to host ASP.NET. This was contributed by Brian Ritchie a few months back, and its finally on the main trunk. <BR>
<BR>
The Mono.WebServer.dll deployment model follows the new <A HREF="http://www.mono-project.com/Guidelines:Application_Deployment">Guidelines for Library Deployment</A> and there are versions available for running on the 1.x and 2.x profiles. <BR>
<BR>
XSP now also takes advantage of certain Linux features like sendfile and TCP Cork to improve performance. <BR>
<BR>
XSP also contains support for HTTPS connections using the --https flag by Brian Ritchie. <BR>

<H2>
<B><FONT SIZE="5">Installing Mono 1.1.9</FONT></B>
</H2>
<B>Important:</B> Mono 1.1.9 can not be installed in parallel with Mono 1.0.x series on the same prefix. To work around this issue, you must use a different prefix at configure time, for example: <BR>
<BR>
<PRE>
        
        $ ./configure --prefix=/devel
        
</PRE>
<BR>
You can then setup your PATH to include /devel/bin to access the Mono 1.1. Alternatively you can replace your Mono installation with 1.1.9 <BR>
<BR>
<B>Binary Packages:</B> <BR>
<BLOCKQUOTE>
    <BR>
    Pre-compiled packages for SUSE, SLES, Fedora Core, RHEL, MacOS and Windows in a variety of platforms available from our web site from the <A HREF="http://www.mono-project.com/Downloads">download</A> section. <BR>
    <BR>
</BLOCKQUOTE>
<BR>
<B>Source code:</B> <BR>
<BLOCKQUOTE>
    <BR>
    <UL>
        <LI>Mono Source Code: <A HREF="http://go-mono.com/sources/mono-1.1/mono-1.1.9.tar.gz">mono-1.1.9.tar.gz</A> 
    </UL>
</BLOCKQUOTE>
<BR>
<B>Quick source code installation:</B> <BR>
<BLOCKQUOTE>
    <BR>
    If we have no packages for your platform, installing from source code is very simple. <BR>
    <BR>
    mono:<BR>
    <BR>
<PRE>
        
    $ tar xzf mono-1.1.9.tar.gz
    $ cd mono-1.1.9
    $ ./configure
    $ make
    $ make install

</PRE>
    <BR>
    Then compile libgdiplus: <BR>
    <BR>
<PRE>
        
    $ tar xzf libgdiplus-1.1.9.tar.gz
    $ cd libgdiplus-1.1.9
    $ ./configure
    $ make
    $ make install

</PRE>
