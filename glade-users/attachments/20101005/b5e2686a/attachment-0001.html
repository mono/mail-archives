</div></div>I&#39;m a little confused about what exactly is not working for you.<br>
<br>
If you have a custom widget that is a GtkContainer, so long as<br>
it works like an ordinary container you should not need to implement<br>
anything custom along the lines of get-children or any of that.<br>
(take the GtkBox implementation for reference in this case; it<br>
uses a virtual &#39;size&#39; property to decide how many placeholders<br>
to expose in the Glade workspace).<br>
<br>
Note that get-internal-child is for indirect children of composite<br>
widgets. Thats what lets a GtkDialog configure properties and<br>
add children to its internal &#39;vbox&#39; or &#39;action-area&#39; (I think thats<br>
out of context for you, I think you just have a normal container).<br>
<br>
What you need to do when implementing a normal container<br>
is usually add placeholders to it if and when appropriate; then<br>
you need to implement add-child-function/remove-child-function<br>
and replace-child-function (replace-child will be called if your<br>
plugin uses GladePlaceholder ... the Glade core will replace<br>
the placeholder with a widget from the palette; giving it the<br>
initial packing properties of the placeholder).<br>
<br>
So, generally you should just start with add-child and remove-child<br>
functions... you should not need to call glade_widget_adaptor_create_internal()<br>
unless you are exposing an internal child of a complex composite<br>
<br>
You dont need to implement get-children-function either unless you need<br>
to add something to the generic child list already returned by your<br>
container&#39;s implementation of GtkContainerClass-&gt;forall().<br>
<br>
