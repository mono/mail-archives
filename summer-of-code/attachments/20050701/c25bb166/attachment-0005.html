The first objective to meet is to use Mozilla to render the contents of
Monodoc (instead of GtkHtml). As a first step we will support two of
them isolated by an Interface. I've been looking to make that Interface and
I end up with 3 problems to solve:<br>
<br>
1) Some of the code that we push to the browser has special addresses
that, for example, point to images inside compressed files. With
GtkHtml you can catch those loads and provide the image. With
GtkMozEmbed (Mozilla) you are only able to send html code. The solution
is to process the html source and insert those images as Base64 strings
like the mail programs do. I have a little program already working.<br>
<br>
2) Things like copy, paste and select are not directly supported by
GtkMozEmbed. The thing is that you can select some text and copy it
with Ctrl+C, but there is no functions to support that and, thus, you
are not able to make a MenuItem that allows copying or selecting (which
Monodoc already has). I think my little knowledge of Gtk/Glib et al.
doesn't help with that (Is it possible to send some kind of Ctrl+C
event??). <br>
I didn't found a solution so I look at Galeon source code and talked
with the developers and they give me a partial solution. Basically,
from the GtkMozEmbed C API you can obtain a XPCOM object to ask for all
those things; but, that function is not available in the Mono wrapper
and I don't liked the idea of P/Invoking. So this problem is pending
yet (and any help will be really appreciated).<br>
<br>
3) Monodoc implements a history mechanism because GtkHtml lacks it but
GtkMozEmbed implements one. I think that can be resolved without much
difficulty, but to be sure I have to test it.<br>
<br>
I think that while I find a solution for problem 2, I will code
everything else and leave the copy/paste/select menu feature for the
end. When I have that, I will start to change some of the html code
generated from the docs to use CSS (but maintaining the actual code in
some way because GtkHtml does not support CSS).<br>
