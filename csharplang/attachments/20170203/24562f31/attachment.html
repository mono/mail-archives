<p class="MsoNormal">My concern with that is that the async-state-machine really should be an implementation detail.  There is no reason for that state machine to be specified (and, indeed, I’d like a future version of a runtime that could just provide this
 for us, without us having to spit out the state machine ourselves).  <o:p></o:p></p>
<p class="MsoNormal"><o:p> </o:p></p>
<p class="MsoNormal">On the other hand, the shorthand property syntax is specifically defined to have a backing, unutterable, field.  i.e.:<o:p></o:p></p>
<p class="MsoNormal"><o:p> </o:p></p>
<h3 style="margin-left:1.0in;mso-list:l0 level3 lfo1"><a name="_Ref173841257"></a><a name="_Toc251613319"><span style="mso-bookmark:_Ref173841257"><![if !supportLists]><b><span style="mso-list:Ignore">1.1.1
</span></b><![endif]><b>Automatically implemented properties</b></span></a><b><o:p></o:p></b></h3>
<p class="MsoNormal" style="margin-left:.5in">When a property is specified as an automatically implemented property, a hidden backing field is automatically available for the property, and the accessors are implemented to read from and write to that backing
 field.<o:p></o:p></p>
