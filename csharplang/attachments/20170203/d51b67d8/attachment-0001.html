<p class="MsoNormal">As a historical perspective, I can say that this choice was made extremely deliberately and after MUCH debate while creating the initial version of C#.</p></div></div></blockquote><div><br></div><div>Thanks for sharing the historical perspective.   </div><div><br></div><blockquote class="gmail_quote" style="margin:0px 0px 0px 0.8ex;border-left:1px solid rgb(204,204,204);padding-left:1ex"><div lang="EN-US"><div class="gmail-m_9220462751057201873WordSection1"><p class="MsoNormal">One of the design principles that we felt strongly about was the core data types (short, int, long, etc.) should be of standardized size and operation across all platforms. This is one of the things that we felt that Java had right, and
 C had wrong. We felt that it was doing programmers a grave disservice to allow them to write code where integers had different size limits or overflow behavior on different platforms, and this just led to code that was very hard to port between platforms with
 virtually no benefit. If you need to store integers larger than 2G, use a long, otherwise use an int. This is an algorithmic decision and the behavior shouldn’t change just because you are running on a 32 or 64 bit platform.</p>
<p class="MsoNormal"><u></u> <u></u></p>
<p class="MsoNormal">IntPtr and UIntPtr were introduced in the base class libraries for various reasons, and C# had to decide how to handle them. We realized that they did have a usefulness in P/Invoke and native interop scenarios, so we added the ability to
 explicitly cast from the regular integers to them and back. Further than that we were unwilling to go, very deliberately.</p>
<p class="MsoNormal"><u></u> <u></u></p>
<p class="MsoNormal">We really didn’t want programmers to write libraries that used size-specific integers in their public interface. By NOT putting operators on them, and by making casts to other integers have to be explicit, we made them painful enough that
 people did not use them as general “platform-specific integers”, like “int” is in C.</p></div></div></blockquote><div><br></div><div>While this might have been an acceptable design decision 15 years ago for a v1 of the language on the dominant platform, C# now lives in diverse ecosystem, one where it needs to interoperate well with other runtimes and languages.   This is one of those things that just needs to be revisited.</div><div><br></div><div>Apple's APIs are both extensive and exhibit this issue, but they are by no means the only ones.   </div><div><br></div><div>Miguel</div></div></div></div></div></div>
