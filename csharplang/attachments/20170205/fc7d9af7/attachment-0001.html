<div dir="ltr">Hello,<div><br></div><div>This is a good observation.</div><div><br></div><div>While the general observation is correct, today we already have operators that can alter the contents of the operators without an explicit annotation, this happens for operators declared in classes, where "operator +" can do whatever it pleases with the internals of the parameters.</div><div><br></div><div>Ref means more "I might alter the incoming parameter and return a new value there".</div><div><br></div><div>Operators just do not work very well in practice with structures and code that wants to have a modicum of performance.   The scenarios that I have seen repeatedly include structures for vectors (assorted versions), matrices and other mathematical objects that are very heavy weight.</div><div><br></div><div>Given that we already have a cognitive split on when an operator would alter the content (class vs struct), I would like to suggest that this feature does belong in the language and that we need to address the fundamental problem of side effects by a two-prong system, guidance/guidelines, and perhaps analyzers.</div><div><br></div><div>Miguel.</div></div><div class="gmail_extra"><br><div class="gmail_quote">On Sat, Feb 4, 2017 at 6:42 PM, Jared Parsons via csharplang <span dir="ltr"><<a href="mailto:csharplang@lists.dot.net" target="_blank">csharplang@lists.dot.net</a>></span> wrote:<br><blockquote class="gmail_quote" style="margin:0 0 0 .8ex;border-left:1px #ccc solid;padding-left:1ex">



<div style="word-wrap:break-word;color:rgb(0,0,0);font-size:14px;font-family:Calibri,sans-serif">
<div>
<div>
<div>In general C# forces passing by ref to be an explicit by forcing a call site annotation.  This makes it obvious to the caller that the callee is capable of mutating the values here.  Allowing operators to silently accept ref arguments means the developer
 loses that declarative feedback.  I think that would be a bit of a step backwards here.  Forcing ref at operator call sites seems a bit awkward as well. </div>
<div><br>
</div>
<div>There is a feature under consideration [1] that allows for ‘readonly ref’ parameters.  It’s likely these parameters can be passed without call site annotation.  There is no possibility of mutation by caller hence the annotation doesn’t add a lot of value.
  </div>
<div><br>
</div>
<div>I think forcing ‘readonly ref’ here would be more appropriate. </div>

<p class="MsoNormal"><span lang="EN-US">Hello everyone.</span></p>
<p class="MsoNormal"><u></u> <u></u></p>
<p class="MsoNormal"><span lang="EN-US">I propose that C# is changed to allow ref-parameters in operators (original discussion in
<a href="https://github.com/dotnet/roslyn/issues/12364" target="_blank">https://github.com/dotnet/<wbr>roslyn/issues/12364</a>). This is useful for expressing operations on large structs in a performant and natural way.<u></u><u></u></span></p>
<p class="MsoNormal"><span lang="EN-US"><u></u> <u></u></span></p>
<p class="MsoNormal"><span lang="EN-US">I tested this with an immutable struct of 3 doubles (attached).<u></u><u></u></span></p>
<p class="MsoNormal"><span lang="EN-US"><u></u> <u></u></span></p>
<ol style="margin-top:0cm" start="1" type="1">
<li class="m_-275194688371142427MsoListParagraph" style="margin-left:0cm"><span lang="EN-US">By value static methods (current operators)<br>
</span><span lang="EN-US">470ms</span><u></u><u></u></li><li class="m_-275194688371142427MsoListParagraph" style="margin-left:0cm"><span lang="EN-US">By value/reference static methods (to allow chaining them without temps)<br>
360ms</span><u></u><u></u></li><li class="m_-275194688371142427MsoListParagraph" style="margin-left:0cm"><span lang="EN-US">Pure by reference static methods<br>
380ms</span><u></u><u></u></li><li class="m_-275194688371142427MsoListParagraph" style="margin-left:0cm"><span lang="EN-US">By reference static methods with an out-parameter<br>
270ms</span><u></u><u></u></li></ol>
<p class="MsoNormal"><u></u> <u></u></p>
<p class="MsoNormal"><span lang="EN-US">The ref/out approach is the clear winner.<u></u><u></u></span></p>
<p class="MsoNormal"><span lang="EN-US"><u></u> <u></u></span></p>
<p class="MsoNormal"><span lang="EN-US">For these operators both the parameters and the return type must be prefixed with `ref` for the compiler to rewrite them into a ref/out method:<u></u><u></u></span></p>
<p class="MsoNormal"><span lang="EN-US">           Unary +, -, !, ~<u></u><u></u></span></p>
<p class="MsoNormal"><span lang="EN-US">           Binary +, -, *, /, %, &, |, ^, <<, >><u></u><u></u></span></p>
<p class="MsoNormal"><span lang="EN-US"><u></u> <u></u></span></p>
<p class="MsoNormal"><span lang="EN-US">For these operators only the parameters must be prefixed with ref, and the method is rewritten into a ref method:<u></u><u></u></span></p>
<p class="MsoNormal"><span lang="EN-US">           Unary: true, false<u></u><u></u></span></p>
<p class="MsoNormal"><span lang="EN-US">           Binary: ==, !=, <, >, <=, >=<u></u><u></u></span></p>
<p class="MsoNormal"><span lang="EN-US"><u></u> <u></u></span></p>
<p class="MsoNormal"><span lang="EN-US">I am not sure how to approach ++ and – operators and user-defined conversions. The former don’t look like they will benefit from this in any way, while the latter might be quite useful, but the conversions spec is hard
 to follow already.<u></u><u></u></span></p>
<p class="MsoNormal"><span lang="EN-US"><u></u> <u></u></span></p>
<p class="MsoNormal"><span lang="EN-US">Mixed-refness is an open issue as well. It’s useful when one of the parameters is a lightweight type, but it introduces additional complexity.<u></u><u></u></span></p>
