Filing a github issue is the right way to report a spec defect. If investigation reveals that the report is incorrect, that can be noted when the issue is closed. Using the mailing list would be counterproductive because few members of the C# design community should need to be involved in addressing defects in the specification documents.

I've noticed that the process for identifying, discussing and correcting spec defects is undocumented in the early days of csharplang.

Therefore, I'd like to open a discussion to address the following questions:
- Currently, spec defects are identified and tracked by GitHub issues - is this the correct way to do this?
- What role do these mailing lists play into identifying spec defects? It seems that the spirit of the mailing lists recommends some kind of discussion before any kind of action.
- Is there a formal proposal process/template for spec defects? Or are GitHub issues (the exact problem we're trying to avoid, right?) good enough.
