> Thanks for the clarification on the stance here. I think the general
> stance on a proposal can be confusing when it is something that would
> (likely) be accepted to the language if implemented but is not necessarily
> ever going to be implemented by the compiler team itself (due to
> priorities).
>
>
>
> I would like to clarify on your understanding:
>
> Ø  The operators for IntPtr ship as part of the library
>
> The runtime treats IntPtr and UIntPtr as primitive types (`native int` and
> `unsigned native int`, respectively). As such, there are no implementations
> for these operators in the library (in the same way that Int32 and Double
> don’t actually define any operators). I would guess this is because it was
> expected that the operators on these primitive types would be supported by
> the various languages (as is common practice).
>
>
>
> Ø  They are implemented directly in IL with a one-instruction body
>
> They can be implemented in IL with a single instruction, the same as all
> other primitive operators. Adding two IntPtrs involves calling `ldarg` with
> each of their values and then calling `add`. There is currently no way to
> get the code implemented in the single instruction outside of creating a
> custom library written in pure IL to expose the functionality.
>
>
>
> ·         Per today’s functionality, the compiler translates application
> of the operator to a static method call
>
> Would a library be implemented (or would these be exposed in CoreFX) as
> described above (written in pure IL), then the compiler would emit the call
> the same as any other method call (presumably)
>
>
>
> ·         Per today’s functionality (presumably), the JIT inlines the
> operator
>
> Would the library be implemented, the JIT would presumably inline the
> operator (since it is small enough). The JIT could even be given special
> handling so-as to handle the calls as an instrinsic, so that the
> inlining/optimization is even better.
>
>
>
>
>
> I actually have a very similar proposal against CoreFX (
> https://github.com/dotnet/corefx/issues/10457) that proposes friendly
> operators be exposed (these are static methods with the corresponding name,
> rather than actual operator implementations) so that languages which do not
> have ingrained support for System.IntPtr and System.UIntPtr as primitive
> types (C#)  can still use the functionality. Such a thing has the drawbacks
> the have already been listed.
>
>
>
> As for
>
> Ø  I’d be fine with this as an *up-for-grabs* feature that could be
> implemented by the community.
>
>
>
> Is there any further work required on the proposal before it can be placed
> in the ‘inactive’ bin (which is where I presume such things would go)?  For
> example: I imagine you may want a full explicit list of the operators
> System.IntPtr and System.UIntPtr support, but I could also see just
> referencing the ECMA-335 spec (as I did) would be reasonable.

>
> It’s not that the runtime has a special opcode, it is that the runtime
> declares a set of primitive types, which C# follows for everything except
> `System.IntPtr` and `System.UIntPtr`.
>
>
>
> Each of the primitive types defines a set of operators that are applicable
> to it. These all have intrinsic support in the compiler to directly emit
> the appropriate IL instructions, except for `System.IntPtr` and
> `System.UIntPtr`.
>
>
>
> These types, regardless of how often they are mentioned in the C# spec,
> are used quite frequently: in Roslyn, in CoreFX, in CoreCLR, in 3rd party
> libraries, in Interop Code, etc…
>
>
>
> Anytime the user needs to perform one of the primitive operations around
> one of these types (operations that are defined by the runtime) they
> currently need to either:
>
>    - Write unsafe code (generally involves calling IntPtr.ToPointer(),
>    performing straight pointer arithmetic and then casting back) -or-
>    - To avoid writing unsafe code, they need to cast to an integer the
>    same size as IntPtr (so Int64 on x64 and Int32 on x86), perform the
>    arithmetic and then cast back
>
>
>
> This is very suboptimal and involves emitting a method call, which may or
> may not be optimized.
>
>
>
> However, if the compiler has built in support for the operators then it
> gets emitted as a single IL instruction (rather than a call) for each
> operation (same as with every other primitive type) and it requires no
> downstream work by the runtime or framework for the benefit.
>
>
>
> -Tanner

>
> *Issue:*
> The CLR supports a set of operators for the System.IntPtr and System.
> UIntPtr types (native int). These operators can be seen in III.1.5 of the
> Command Language Infrastructure specification. However, these operators are
> not supported by the C#.
>
> *Workaround:*
> Either directly emit IL or write non-optimal code that does not directly
> translate to the appropriate IL instructions.
>
> *Request:*
> Language support should be provided for the full set of operators
> supported by System.IntPtr and System.UIntPtr. These operators are: Add,
> Divide, Multiply, Remainder, Subtract, Negate, Equals, Compare, And, Not,
> Or, XOr, ShiftLeft, ShiftRight.
>
> *Other Thoughts:*
> C# seems to treat both IntPtr and UIntPtr as a type only used for interop
> and for representing pointers in a 'safe' manner, when it is really just:
>
> an integer whose size is platform-specific.
>
> It should be possible for a user to treat the IntPtr and UIntPtr types as
> an equivalent to the native size_t type. However, in order to do that,
> one must may have to deal with one or more of the following:
>
>    - Perform multiple levels of casting
>    - Use integers whose size is larger than their platform's native size
>    - Have branching to use integers that are platform-specific
>    - Work with unsafe code and do pointer arithmetic
>
>
