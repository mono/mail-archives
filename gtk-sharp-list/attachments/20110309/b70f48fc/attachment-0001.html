Subclasses may need to free their own unmanaged resources, which may not<br>
be related to gtk at all. For example, I could implement a widget<br>
subclass which pinvokes a C api to get some data, which has to be<br>
explicitly freed. I would free that data in the finalizer, but since the<br>
finalizer will not be called if the object is explicitly disposed, I<br>
also have to free that data in the Dispose method. So, I will end having<br>
a DeleteMyData method which is called from the finalizer and from<br>
Dispose.<br>
<br>
The Dispose pattern provides an standard solution to the above scenario.<br>
It states the need for a Dispose(bool disposing) method, to be called<br>
from Dispose and from the finalizer. The &#39;disposing&#39; parameter is &#39;true&#39;<br>
when calling from Dispose, meaning that it is safe to access other<br>
reference objects, and it is &#39;false&#39; when called from the finalizer,<br>
meaning that only unmanaged data should be accessed.<br>
<br>
There is some documentation about this pattern here:<br>
<br>
<a href="http://msdn.microsoft.com/en-us/library/b1yfkh5e.aspx" target="_blank">http://msdn.microsoft.com/en-us/library/b1yfkh5e.aspx</a><br>
<div><div></div><br></div></blockquote><div><br>I disagree strongly with the implicit destruction of managed resources through the use of that pattern and the finalizer. Having worked extensively on embedded systems where tight control over memory is always required, in my experience, it is always better to provide tools for programmers to fix bad usage than it is to brush it under the rug and do their job for them. The finalizer should be there only to warn that memory is leaking and if you must, free that memory, but the warning is key so that the code can be fixed. Finalizers add a cost that can be very hard to measure.<br>
All objects that have managed resources to discard should implement Dispose and call base. Destroy should probably call Dispose (and not the other way around). Either way, documentation has to state clearly which is which and how to properly subclass widgets/other things. <br>
