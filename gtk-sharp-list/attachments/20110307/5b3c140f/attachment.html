If we want to bring up the debate of how gtk resources are destroyed, I would strongly like to suggest _NOT_ piggy backing on the finalizers to garbage collect GTK objects.<br>This will play very poorly with the GC heuristics.<br>

GTK native resources should use the same practices that the rest of the C#/.net community uses: explicit usage of the dispose pattern.</blockquote><div> </div><div>Another voice in the choir.  For the record, my reluctance on this is an equally strong practice in the Gtk+ community: the use of an explicit Destroy pattern.  I think if we are going to make Dispose mean Destroy in 3.0, we&#39;ll probably need to completely remove the Destroy methods to avoid confusion, and that will probably be a big API break for existing codebases.  Everyone seems to be arguing for it, though.</div>
<div><br></div><blockquote class="gmail_quote" style="margin:0 0 0 .8ex;border-left:1px #ccc solid;padding-left:1ex;">To highlight why I do not like the current behavior:<br>1- Create a bunch of widgets<br>
2- Manually remove their references<br>3- GC kicks in and marks those objects as &quot;dead&quot; and queues them up for the execution of their finalizers<br>4- The gtk object finalizers executes, which adds again a reference to that object so that the GTK main thread can release it<br>

5- The gtk main thread picks up the queued objects for destruction and destroys them<br>6- GC runs again and marks those objects as &quot;dead&quot; and queues them up for the execution of their finalizers<br>7- The gtk objects finalizers execute and since the is no more work to be done, the object is finally destroyed and its memory reclaimed<br>

<br>The &quot;clever&quot; use of the finalizer in step 4, in a multi generational GC heap, will move that object to a longer lived heap and it will take potentially MUCH longer for step 6 to run on that longer lived heap. You are effectively promoting soon to be dead objects to a long life object heap which means their memory will linger in &quot;use&quot; much longer than it would need to.<br>
</blockquote><div><br></div><div>Your analysis doesn&#39;t really track.  Dispose invokes GC.SuppressFinalize, so an explicitly Disposed, or Destroyed as is proper in 2.12, Object will never run the finalizer. </div><div>
<br></div><blockquote class="gmail_quote" style="margin:0 0 0 .8ex;border-left:1px #ccc solid;padding-left:1ex;">It would make much more sense to have the finalizer simply check if the object has been disposed of manually and emit a warning that the application is leaking GTK memory instead of circumventing the proper flow of things. Perhaps that check could only be in debug as well, reducing the overhead in release/final builds.<br>
</blockquote><div><br></div><div>This thread has kind of gone a little far afield from the user&#39;s original post, but I guess I can reply to this the same way as the original.  If you have a sample that profiles as leaking Destroyed objects, I&#39;d be happy to take a look.  Otherwise, this is a theoretical discussion about what I consider working code until someone proves otherwise to me.</div>
