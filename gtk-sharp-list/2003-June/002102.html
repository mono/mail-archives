Of course: GtkSharpRuntimeEnvironment.Init is a hack. Of course: It 
would be much more elegant, if Gtk.Application.Init would call the code 
found in GtkSharpRuntimeEnvironment.Init() behind the scene. But 0.10 
has been out already, when that I had the idea for this hack. And well: 
I'd say using that hack for 0.10 is better than distributing a patched 
0.10 or even waiting for 0.11. ;-) Btw: The word "elegant" refered to 
the need of manually tweaking the PATH variable.

>>Ximian will do official windows packages, based on the 0.9 nsis script.

> I won't be doing the Windows packages for a few more hours, if you could

No problem. Would be nice if the code would be merged into the offical 
package. Attached you'll find the package. If Cygwin, NSIS, the Gtk+ 
Runtime Environment and the Gtk+ Development Environment are installed, 
all you need to do is to run "build.sh" from within Cygwin. I'd suggest 
to use a fresh installation of the Gtk+ Development Environment, since 
build.sh patches the pkg-config files shipped with the Gtk+ Development 
Environment.

The bash script and the NSIS script are quite verbose and documented.
The file GtkSharpRuntimeEnvironment.cs contains my runtime 
initialization hack.

To strongly sign the Gtk# assemblies build.sh first calls Micrsoft's 
"sn" tool to create "gtksharp.snk" and then links all the assemblies 
with StrongName.netmodule (generated from StrongName.cs).

Build.sh also manually recompiles gtksharpglue.dll since the package's 
makefile links against the Cygwin runtime.
