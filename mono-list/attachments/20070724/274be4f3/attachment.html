a problem when we try to use automation tools on an app running on Mono on

each control to determine what kind of control it is (BUTTON, STATIC) etc.&nbsp; 
On .Net, new Windows classes are registered for each control type, but they are 
still unique enough that you can map them to your own SilkTest classes.&nbsp; 
While we can use the .Net extension of SilkTest to automatically give us all the 
class details (methods, etc of each control) we obviously cannot use this on 
Mono.&nbsp; because of this we also don't use it when we run our app on .Net, 
since we want to create an environment where we can seamlessly test our app on 
both .Net and Mono.</SPAN></FONT></DIV>

own SilkTest classes.&nbsp; Unfortunately SilkTest is not flexible in how it
