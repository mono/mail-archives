<br>It is safe to use a socket in most unices in two threads if one does only reading and one does only writing.<br><br>Is this true of Mono/C#?<br><br>The only documentation I can find is from Socket:<br><br>&quot;All public static members of this type are safe for multithreaded
operations. No instance members are guaranteed to be thread safe.&quot;<br><br>Which I believe is boilerplate, but would imply that it might not be safe to read in one thread and write in another.<br><br>My experiments suggest that reading in one thread and writing in another works in mono (tested with UDP), but I am wondering if there is an &quot;official&quot; position.&nbsp; Is this unclear in the .Net class library?&nbsp; Might it stop working in the future and not be considered a bug?
<br><br>The case I am working with is the following: two hosts have a socket between them.&nbsp; At any time one or the other may send data.&nbsp; What is the lowest latency way to make sure that data is sent quickly and received quickly.&nbsp; Being strict about the documentation would imply it is not safe to do a blocking read in one thread while sending in another thread.&nbsp; Thus, it seems that polling (or select) is required, both of which introduce latency.
<br><br>If anyone else has some good pointers on how to use one socket to send and receive without polling or using the asynchronous methods I would be happy to learn of them (polling introduces latency and the asynchronous methods often lead to deadlocks I guess due to their internal use of threadpools).
