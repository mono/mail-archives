<div dir="ltr">I would not suggest MEF even if it works with mono because it will force your plugin developers to have a hard reference to MEF itself, which might not be optimal if you'd like to have the ability to have plugins as PCL's (portable class libraries).<div><br></div><div>Take a look at what we're doing with Media Browser:</div><div><br></div><div><a href="https://github.com/MediaBrowser/MediaBrowser">https://github.com/MediaBrowser/MediaBrowser</a><br></div><div><br></div><div>We have a plugins directory where we find and load assemblies. Then we have a library of interfaces that plugins can implement. For example, here's the interface for a video channel, called IChannel:</div><div><br></div><div><a href="https://github.com/MediaBrowser/MediaBrowser/blob/master/MediaBrowser.Controller/Channels/IChannel.cs">https://github.com/MediaBrowser/MediaBrowser/blob/master/MediaBrowser.Controller/Channels/IChannel.cs</a><br></div><div><br></div><div>And here is an implementation from the Vimeo plugin, contained in another repository:</div><div><br></div><div><a href="https://github.com/MediaBrowser/MediaBrowser.Channels/blob/master/MediaBrowser.Plugins.Vimeo/VimeoChannel.cs">https://github.com/MediaBrowser/MediaBrowser.Channels/blob/master/MediaBrowser.Plugins.Vimeo/VimeoChannel.cs</a><br></div><div><br></div><div>The only thing a plugin has to do is implement the interfaces it intends to support. No manual wireup. No dependencies on MEF or any other similar third party library. No custom attributes required. Then once we have the assemblies loaded into memory, we use SimpleInjector to find and instantiate the instances:</div><div><br></div><div><a href="https://github.com/MediaBrowser/MediaBrowser/blob/master/MediaBrowser.ServerApplication/ApplicationHost.cs#L737">https://github.com/MediaBrowser/MediaBrowser/blob/master/MediaBrowser.ServerApplication/ApplicationHost.cs#L737</a><br></div><div><br></div><div>This also gives us dependency injection via constructor p
