Anyway, I was quite pleased to see the Mono.Simd experimental addition. It&#39;s<br>
about time someone in the CIL world put efforts in exploiting the power of<br>

processing power increase in these CPU comes from SSE and al.<br>
<br>
I have a couple of comments on Mono.Simd:<br>
<br>
1) Why calling it something that ties it to the underlying technology used?<br>
Why not calling it Mono.Math.Advanced, or Mono.Math.Vectors, or<br>
Mono.Math.Matrix? It doesn&#39;t have a big impact and it&#39;s just a question of<br>
taste I guess, but I find that Mono.Simd relates too much to the<br>
implementation technology.</blockquote><div><br>The idea is to expose the Single Instruction Multiple Data extensions of modern cpus.<br>It&#39;s not specific to vector or matrix math as it can be used with cryptography and image<br>
processing with the same degree of success.<br>&nbsp;<br></div><blockquote class="gmail_quote" style="border-left: 1px solid rgb(204, 204, 204); margin: 0pt 0pt 0pt 0.8ex; padding-left: 1ex;">
2) Would it be possible to use generic instead of &quot;hardcoding&quot; the types and<br>
sizes of vectors in each class? I don&#39;t know if it&#39;s possible, but there are<br>
probably ways to pre-instantiate templates like you do in C++, so you have<br>
each supported template specification ready, and/or design the native<br>
underlying module that actually does the transformation to SIMD instructions<br>
do it in hardware only for the supported types. This would make this library<br>
quite flexible I think.</blockquote><div><br>Generics would pose some of very complicated problems to solve, such as<br>the default implementation (can&#39;t do math over generic arguments), and confusion<br>with the users as to what would get accelerated or not. The second issue is due to the<br>
fact that not all operations are available for all element types and the vector unit is 128 bits<br>long - so a Vector16&lt;double&gt; would not flight.<br> <br><br></div><blockquote class="gmail_quote" style="border-left: 1px solid rgb(204, 204, 204); margin: 0pt 0pt 0pt 0.8ex; padding-left: 1ex;">

<br>
3) I would add more high level constructs like Matrix that make use of the<br>
SIMD hardware.<br>
</blockquote><div>&nbsp;<br>We first need to expose a rich and usable set of intrinsics to what SSE currently has.<br>Once we do finish with that, we might look at exposing more high level functionality.<br><br>On the other hand, such experimentation should happens outside of Mono.Simd to be<br>
