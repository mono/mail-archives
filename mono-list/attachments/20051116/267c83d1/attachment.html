&gt; plan to make a provider for mysql since Microsoft does not have that?<br>
<br>
Well, there are some basic stubs that I did (a long time ago). But<br>
AFAIK, nobody is working on implementing one of the providers.<br>
<br>
IMHO, SqlMembershipProvider (if it is implemented) should be written so<br>
that it queries a mssql server with the schema that msft's framework <br>
uses -- there would be some value to being able to share an<br>
authentication database with apps that are written on windows. Also, if<br>
somebody wants to quickly migrate an application from windows to linux,<br>
it may be easiest to first do the web part, then the db part. (in fact <br>
one app that I am working on for school would benefit greatly from being<br>
able to attach to the MSFT schema).<br>
<br>
If we want to have a provider that uses a free database (which is<br>
needed), it should probably be a mono specific assembly (that way people <br>
on Windows can use it too, if they don't want to pay &gt; $1000 for<br>
sqlserver (and for some reason can't use don't want sqlexpress). We<br>
could still make this class the default in Mono's machine.config, so<br>
that things worked out of the box (however, for this to really work, <br>
we'd need something like sqlexpress's database attaching).<br>
<br>
In terms of mysql, pgsql might be a better choice as the mysql provider<br>
is now GPL and we are no longer updating it from upstream.<br>
