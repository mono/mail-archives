For security reasons we don't allow Web servers to connect to databases.&nbsp; Instead, we split the application into layers.&nbsp; The presentation logic layer runs on the Web server.<BR>
<BR>
We use forms authentication.&nbsp; Users log in and their credentials (as entered) are sent to a remote object, on the business logic layer server, that checks with AD and returns true/false.<BR>
<BR>
When logged in (we use attributes to protect objects that can only be accessed by logged in users, and we also check the users' roles the same way), users can perform functions that access the db.&nbsp; The code behind pages on the Web server pass the data storage and retrieval requests to another remote object on the business layer server.&nbsp; This remote object validates any data being passed in and connects to the database, which may or may not be remote, returning whatever is required to the code behind page.<BR>
<BR>
The Web server is not on the domain, but the business logic server and the database server both are.&nbsp; This means that we can use trusted connections between the business logic layer and the database layer.&nbsp; The business logic server is configured to accept requests only from the Web server.&nbsp; The database server is configured to only accept requests from the business logic layer server.<BR>
<BR>
The upshot of all this is the Web server can be any type we like running on any OS we like.<BR>
<BR>
That's how we do it, anyway.<BR>
