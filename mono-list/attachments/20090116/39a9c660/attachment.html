I think mono would benefit, more than any other stack, from an investment in<br>
SWIG or some similar technology. &nbsp;<a href="http://www.swig.org" target="_blank">http://www.swig.org</a><br>
<br>
For better or worse Linux is pretty much a C++ domain. &nbsp;There are some<br>
aspiring stacks out there, but thus far none have gained much traction. The<br>
Mono stack is gaining adoption simply because despite all the FUD Mono<br>
simply works, period.<br>
<br>
Hence the need for C++ interoperability. For those not familiar with SWIG,<br>
here&#39;s a brief introduction:<br>
<br>
<br>
SWIG &nbsp;is a partial C/C++ compiler which takes C/C++ header files and<br>
generates two outputs:<br>
1. A C++ wrapper source file to include in your library (mylib_wrap.cc), as<br>
well as C# files which will invoke the C++ library. &nbsp;Its simple and for the<br>
most works incredibly well.<br>
<br>
There are however some limitations. It fully supports C, and somewhat<br>
supports C++. In the domain of C++, templates are supported but its a<br>
struggle.<br>
<br>
Now, the thing to keep in mind is that SWIG is designed to generate bindings<br>
for multiple languages. Java, python, csharp, perl, and many more. As such<br>
its architecture is somewhat encumbered by the fact that not all target<br>
languages support the same constructs.<br>
<br>
It seems to me the mapping of C++ templates to c# generics holds some<br>
exciting possibilities. &nbsp;I would think that the SWIG %template directive, or<br>
some derivative of it could be used to push template instantiation further<br>
into the runtime domain (away from compile time).<br>
<br>
Unfortunately, making this happen is well beyond my abilities. So at this<br>
point I&#39;m simply throwing the idea out for discussion.<br>
