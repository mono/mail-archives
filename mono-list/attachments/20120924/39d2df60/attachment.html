It looks like <a href="http://ASP.NET">ASP.NET</a> doesn't like EXE references, as it seems the same issue happens on Windows:<br><a href="http://stackoverflow.com/questions/3132150/how-to-add-exe-as-reference-in-my-web-application">http://stackoverflow.com/questions/3132150/how-to-add-exe-as-reference-in-my-web-application</a> <div>
<a href="http://www.pcreview.co.uk/forums/re-cannot-reference-exe-t4006582.html">http://www.pcreview.co.uk/forums/re-cannot-reference-exe-t4006582.html</a> </div><div><br></div><div>I'm guessing the <a href="http://ASP.NET">ASP.NET</a> compiler doesn't handle EXE references the same way as DLL references. All view files are compiled at runtime using the <a href="http://ASP.NET">ASP.NET</a> compiler, which may compile the files with different build parameers compared to the rest of the app. The shared code should probably be in a .DLL assembly - I can't think of any reason why a website should reference an EXE file and not a DLL file.</div>
<div><br></div><div>You could try precompiling your views and seeing if that changes anything (set MvcBuildViews to true in your .csproj file). This compiles the views when you compile the rest of your application and it might fix this issue. This is an option in <a href="http://ASP.NET">ASP.NET</a> MVC 3 and 4 but I'm not sure if it's available in <a href="http://ASP.NET">ASP.NET</a> MVC 2. I'd suggest just moving the shared classes into a class library (DLL) file though.</div>
