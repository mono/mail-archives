In mojoPortal we create parallel MonoDevelop projects and solution<br>
file (though MonoDevelop could just use the same projects and solution<br>
as VS so you may not need to do this) because we sometimes need to do<br>
things differently on the Mono side. It works ok for us but there are<br>
some things to consider.<br>
1. I don&#39;t think its yet possible to debug <a href="http://ASP.NET" target="_blank">ASP.NET</a> or step through the<br>
code using MonoDevelop. A work around is logging with log4net but its<br>
a big disadvantage.<br>
2. If you&#39;re planning to use the latest greatest .NET 3.5 stuff or the<br>
beta MVC stuff you&#39;re going to run into not yet implemented or<br>
supported things.<br>
3. If your project is going to use <a href="http://ASP.NET" target="_blank">ASP.NET</a> webparts its not supported.<br>
4. You will occasionally run into bugs where things work on the<br>
windows side but not on Mono. Reporting these will help Mono improve.<br>
<br>
If you&#39;re an experienced developer in other languages and experienced<br>
with Linux you may be able to do it, but if its all new to you, you<br>
are making the learning curve much steeper for yourself, and it may be<br>
more productive to just use VS. I will say, myself, I am able to work<br>
on mojoPortal from Linux with MonoDevelop, but I use VS day to day for<br>
most things because its more productive for me.<br>
