<div>
<p class="MsoNormal">I’m not sure if this is an option for you, but if you control the host for your containers, then supplying mono to your container via data volume works for me to keep the images small:<o:p></o:p></p>
</div>
<div>
<p class="MsoNormal"><o:p> </o:p></p>
</div>
<div>
<p class="MsoNormal"><a href="https://docs.docker.com/v1.8/userguide/dockervolumes/#adding-a-data-volume">https://docs.docker.com/v1.8/userguide/dockervolumes/#adding-a-data-volume</a><o:p></o:p></p>
</div>
<div>
<p class="MsoNormal"><o:p> </o:p></p>
</div>
<div>
<p class="MsoNormal">To describe my setup a little more, I run Mesos, and each slave has a mono distribution in a directory that mounts to the location of the shared mono distribution.  I also have a directory that contains nuget packages on each slave which
 I also mount as a docker volume.  <o:p></o:p></p>
</div>
<div>
<p class="MsoNormal"><o:p> </o:p></p>
</div>
<div>
<p class="MsoNormal">With this configuration the Docker image really bootstraps itself by<o:p></o:p></p>
</div>
<div>
<p class="MsoNormal">1. Adding the docker volume containing mono to the PATH<o:p></o:p></p>
</div>
<div>
<p class="MsoNormal">2. Set LD_LIBRARY_PATH<o:p></o:p></p>
</div>
<div>
<p class="MsoNormal">3. Add nuget source<o:p></o:p></p>
</div>
<div>
<p class="MsoNormal">4. Install packages that contain .dll’s and .exe’s<o:p></o:p></p>
</div>
<div>
<p class="MsoNormal">5. Start the application itself.<o:p></o:p></p>
</div>
<div>
<p class="MsoNormal"><o:p> </o:p></p>
</div>
<div>
<p class="MsoNormal">The images are relatively small, at least they don’t contain copies of mono or my fairly large application, but they are entirely ephermal, and I can update packages without a need to rebuild and maintain many the images.<o:p></o:p></p>
</div>
<div>
<p class="MsoNormal"><o:p> </o:p></p>
</div>
<div>
<p class="MsoNormal">Some or all of course, is optional - you could put all your code on the image just as well.<o:p></o:p></p>
</div>
