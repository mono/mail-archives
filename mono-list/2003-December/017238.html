Hi!

Yesterday (or today for the first hours) I was in the #mono channel and
everybody (more Miguel than others) about the MonkeyGuide, to write more
docs and in a 'better and easy' format, so my ideas are:

* Create a system, that will have a client, a website (for moz&ie users)
and a WebService. The concept idea:

* A Client where the 'documentation' guys can Add/Delete/Modify
sections, subsectiosn, content in a easy way, for example if I like to
create documentation I just run the client, connect to the site and
communicdate via SOAP or XMLRPC and check the sections that exists or
create new sections (for example a gtkhtml-sharp section) and Add
content and th econtent will be send to the Webservice (that could be in
phpxmlrpc, or pure ASP.NET). 

* But Why just a client and a webservice? I think that the client could
have a 'parser' that will convert the typical html, text, docbook style
to ECMA. So, when the documentation is send to the Webservice, the
content will be 100% compatible with the ECMA Standard Documentation.

* So a mortal user can download the monkeyguide module for monodoc and
ALL the documentation of monodoc will be 100% ECMA Style.

* And also, the website of MonkeyGuide (like gotmono.com) could give so
many options for the 'mortal user', for example: using templates, many
languages, select their sections and download them as pdf, html, docbook
or ecma style!.

Maybe sound very stupid my idea, but this may help for new users to
download the source code and learn to use XmlRpc (or Soap), WebServices,
Gtk#.. etc..etc..

And however, thanks!
