Some further comments:

The big benefit that *guaranteeing* a StackOverflowException is thrown
instead of segmentation fault occurring is that it makes "plugins" safe.
For example, a database can allow stored procedures to be written in any
CLI-based language with the certainty that no stored procedure can ever
crash the database. This is why Oracle databases allow java-based code
to run in-process (with major performance benefits).

Of course the plugin code can still leak all available memory, or hog a
significant percentage of the CPU, or use up all available file handles,
etc. Still, preventing crashes is a good start. And if you could provide
a mechanism to limit other types of resources allocated by a specific
thread or assembly, then that would be great!

I'm not into C# in a big way yet, but this would definitely be a feature
that I would look for (code cannot crash the application) before moving
from java to C#.

Cheers,

Simon
