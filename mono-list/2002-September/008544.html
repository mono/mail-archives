Firstly, I think you'll find that most of the time you never need to do
this because list boxes usually contain runtime data.  And when you have
runtime data, the MVC shines because you never need to store and
synchronize data in and between two locations.

Anyway, of course you visually design MVC based controls.

An example using VS.NET would be like this:

+Drag-Drop the ListBox onto the form.
+Drag-Drop the ArrayListModel onto the form.
+Set the "Model" property of the ListBox to the ArrayListModel.
+Configure the ArrayListModel with the list of data you want.

This is actually more flexible.  If you ever want to replace the listbox
with some other list control (like a combo box), you won't have to
re-enter the data.  This can be quite a common occurrence and a source
of frustration when using drag and drop IDEs.  Using MVC, the problem is
solved.
 
You could also support other drag'n'drop listmodel components like
DataSourceListModel, XmlListModel, ResourceListModel etc...
