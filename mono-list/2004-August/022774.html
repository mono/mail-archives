Hello.  I just started learning mono 2 days ago as
well.  I found the following link helpful to get
started with a 
"Hello World" program:

http://www.ondotnet.com/pub/a/dotnet/2004/08/09/gtk_mono.htm

I had many runtime errors until I created another
directory outside my c:\mono101 main mono directory. 
The directory tree I created was
c:\mono\Mono-1.0\lib\.  I then copied the pango
directory tree to this directory and everything ran
fine.

Also, note that my mono installation is not in
c:\Program Files, which is the default installation
director in Win.  This gives my editor, UltraEdit,
fits  when I try to invoke mcs.exe and mono.exe from
inside UltraEdit.  This is not a mono issue, but Perl,
Python, Zope, and other *nix originated software don't
install in the c:\Program Files directory (that silly
space in the directory name!).

System: WinXP Pro and Suse Linux dual boot, 2.4 GHz,
1GB ram Dell Inspiron 8500.

Cheers!

&gt; For the most part developing applications and
&gt; solutions for Mono in 
&gt; Win32 is not that different if any than
developing
&gt; applications using 
&gt; the Microsoft .NET Framework SDK.  In fact, a lot
of
&gt; the samples in the 
&gt; MS SDK should compile and build without much
&gt; problems.
&gt; 
&gt; There are some differences between both SDKs. 
There
&gt; are tools available 
&gt; in Mono that are none existent in the Microsoft
SDK,
&gt; for example we use 
&gt; mcs.exe as the C# compiler, we have monop which
&gt; creates stdout output of 
&gt; the description of a given assembly.  Mono also
has
&gt; its own Global 
&gt; Assembly Cache and this is not shared with the
&gt; Microsoft GAC.
&gt; 
&gt; There are also technologies that are very closely
&gt; associated with Mono 
&gt; and are just now becoming more common place in
the
&gt; Windows world like is 
&gt; the case for Gtk# a GUI toolkit based on GTK+
&gt; technology.
&gt; 
&gt; The Microsoft Framework Class Library (FCL) does
not
&gt; include some of the 
&gt; great offerings that the Mono FCL has such as
&gt; Mono.GetOptions or the 
&gt; Novell.Directory.Ldap assembly.
&gt; 
&gt; Thanks to contributors like Daniel Morgan, there
has
&gt; been a Mono 
&gt; presence in the Win32 world even in earliest
&gt; releases before going 1.0.  
&gt; However, Mono has seen a lot of use and
development
&gt; in OS platforms like 
&gt; Linux were developers are very comfortable with
&gt; command line compilation 
&gt; and source code editing with text editors like
vim
&gt; and EMACS were the 
&gt; majority of Windows business application
development
&gt; has generally been 
&gt; done in richer integrated development
environments
&gt; geared towards rapid 
&gt; application development.  In non Windows OS
&gt; platforms there has also 
&gt; been a proliferation of IDE development such as
&gt; MonoDevelop (an awesome 
&gt; community led IDE) and Eclipse enrich with C#
&gt; plug-ins.
&gt; 
&gt; There is a recent push to mature or rather
enhance
&gt; the Mono development 
&gt; experience for those programmers using Microsoft
&gt; Windows as their OS.  
&gt; There are now projects to generate Makefiles from
&gt; Visual Studio C# 
&gt; solutions and projects or create mirror solutions
&gt; and/or projects for 
&gt; MonoDevelop, Microsoft Help 2.0 versions of Gtk#
&gt; library documentation, 
&gt; and preliminary work Visual Studio .NET 2003
Add-ins
&gt; aimed specifically 
&gt; at testing and simplifying the deployment of
&gt; projects build for/with MS 
&gt; .NET Framework tools.  The future may also hold
&gt; plug-ins and add-ins for 
&gt; GPL licensed Win32 IDEs like #Develop.
&gt; 
&gt; So before I give you a list of links to help you
&gt; make sense of it all, 
&gt; keep in mind that for the most part, when you
&gt; develop for MS .NET you 
&gt; are developing for Mono :)
