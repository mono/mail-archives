Hello,

> Some background about the issues of mapping:
> 1)	objective-c allows instance and class methods with the same name: - 
> (id) class; + (id) class;
> 2)	objective-c allows C# keywords as methods names: (private, public, 
> class, struct, ...)
> 3)	objective-c does not have properties, therefore we need to detect 
> them with some pattern
> 4)	objective-c uses 'method: arg with: arg2' pattern, which results in 
> 'method:with:' selector

I personally need some more background to understand, maybe a couple of
sample test cases would be useful.

> Proposal for rules:
> 1)	always start with lower case on instance methods/properties, and 
> upper case on class methods/properties --> object class { get; }, 
> object Class { get; }

Do you mean:

class Demo {
	static object ClassMethod () 
	{
	}

	object instanceMethod () 
	{
	}
}

If so, this is not a good idea for two reasons:

	* It is not CLS compliant (which means, no VB.NET support, Lisp,
	  or Fortran).

	* It does not follow the naming conventions of .NET

If there is a name clash, I would rather deal with those clashes on a
case-by-case basis.   Can we make a generalization about class methods
vs instance methods?  Are they typically factory methods?  Are they
typically helper routines that are just namespaced?

For more details:

	http://dotnet.di.unipi.it/EcmaSpec/PartitionI/cont10.html
