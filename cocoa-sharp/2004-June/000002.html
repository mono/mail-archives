Ok, I'll jump in...

(1) Binary Installation for our Mono runtime, published so that all Cocoa#
dev's can use the same foundation.

(2) Binary Installation for our Mono.Framework.

(3) Binary Installer for the Xcode plugin's to make it recognize C# code for
integration into Cocoa applications.

(4) Polish and Prep for public consumption the Cocoa calling to Mono effort
and build a binary installation of the tools.  This means Xcode templates
and project files.

(5) Package XSP Installation (Mod_mono pending Apache 1.3 integration)

Here is where there is a fork in the road, and I think we need a strategic
division of labor.

On one path we need the Cocoa# Bindings and glue code effort, while on the
other we need to get GTK# and MonoDevelop packaged.

(6) Package GTK# Installation

(7) Package MonoDevelop Installation

-or-

(6) Generate Glue & Library

(7) Implement NSSelector

(8) Package Phase I of the Mono to Cocoa implementation.


Somewhere in there, we also need to get the Mono Debugger working and
integrated, or no developer is going to buy into it.

These are all high level bullet items, and we need to break them down much
much more granular, but in my view, this is what we need to roadmap and more
importantly set timelines before we go public

Andy
