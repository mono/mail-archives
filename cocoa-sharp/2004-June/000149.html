On 28-Jun-04, at 2:39 PM, Miguel de Icaza wrote:

> Hello,
>
>> Some background about the issues of mapping:
>> 1)	objective-c allows instance and class methods with the same name: -
>> (id) class; + (id) class;
>> 2)	objective-c allows C# keywords as methods names: (private, public,
>> class, struct, ...)
>> 3)	objective-c does not have properties, therefore we need to detect
>> them with some pattern
>> 4)	objective-c uses 'method: arg with: arg2' pattern, which results in
>> 'method:with:' selector
>
> I personally need some more background to understand, maybe a couple of
> sample test cases would be useful.

Example of #1:
+ (id) class;
- (id) class;
Both of these exist in NSObject; 1 of them is a static method; 1 is an  
instance method.

Example of #2:
NSButton/NSBrowser (pretty much any gui object) uses a delegate (not a  
C# delegate; an objc delegate) to control behaviours; as a result they  
implement 2 methods:

- (id)delegate:
- (void)setDelegate:(id)aDelegate;

In cases where we detect "function"/set"Function" we combine these into  
a property like soo

public object Delegate {
	get { return NSBrowser_delegate(Raw); }
	set { NSBrowser_setDelegate(Raw, value); }
}

Example #3:
	See #2; we're detecting and generating properties based on a set of  
rules.

Example #4:
-  
(BOOL)shouldCloseWindowController:delegate:shouldCloseSelector: 
contextInfo:

in C# terms this is

public bool MethodName(bool arg1, object arg2, object arg3, object  
arg4){}

We're trying to determine the "rules" to create method name; currently  
we're flattening into
shouldCloseWindowController_delegate_shouldCloseSelector_contextInfo;  
but this is very very long.

We have now created a mapping xml to allow us to edit and tweak any of  
these rules; this discussion is based around naming conventions and  
what we want to do.

(Please not; my use of "object" isn't actually the case in most cases;  
I just didn't look up the actual arg value)
