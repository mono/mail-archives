I think I'm missing something here, but I've been carrying on with 
selector musings.

It looks like everything that derives from NSObject implements 
performSelector:SEL.

Could we not generate a class (NSSelector.cs) that would take a string 
and compile it to a SEL.  Pass that Selector into (for instance) a 
NSButton.setAction(), along with the class to act on casted to 
NSObject; like so:

NSButton.setAction((NSObject)button2, mySelector);

which would call glue which was:

void performSelector(NSObject* toActOn, SEL theSelector) {
   [toActOn performSelector:theSelector];
}

so in this example we could have:

mySelector = new NSSelector("setTitle: @"foo"");

pass that and button2 to button1, when clicked it should call glue and 
perform action; no?

Am I missing something obvious?

As for the NSSelector constructor; I'm sure we could clean it up using 
properties and such from the objects themselves.
