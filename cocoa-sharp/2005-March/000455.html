On Mar 21, 2005, at 12:59 PM, Patrick Beard wrote:

> On Mar 21, 2005, at 8:38 AM, Brian Ketelsen wrote:
>
>
>
>> I am a .Net developer by day, mac user by night, and I just ran  
>> across
>> Cocoa# last night while I was desperately searching for a way not to
>> have to learn ObjectiveC.   I looked everywhere for information on  
>> how
>> to get started with all of this, but can't find much other than
>> general project overview information.  Is there a quick and dirty
>> guide somewhere?
>>
>> My goal is to be able to write in a language that is comfortable to
>> me, but be able to deploy to the "world".  Will cocoa# fulfill that
>> goal?  Do the end users need to install mono?  Is there a runtime for
>> mono?
>>
>>
>
> Brian -
>
> Most of the work of learning Cocoa is not the Objective-C language,  
> but the Cocoa frameworks themselves. Since the majority of the  
> documentation on Cocoa assumes Objective-C knowledge (albeit not  
> much) it helps to get at least a reading familiarity with the  
> syntax. This reminds me of the early days of the Macintosh APIs,  
> when all the documentation was published in Pascal, which I always  
> translated in my head to C. Pascal is a language I find quite easy  
> to read, but I've never a line of it in my entire career.

At some point someone was working on a parser that would go through  
all the Cocoa API documentation and convert it to C# syntax.  This  
would still be a good project to get done.

>
> From what I understand so far, using Cocoa# isn't going to provide  
> you with a way to deploy to the "world." It is simply a way to use  
> the Cocoa frameworks from C# on top of the Mono runtime. The  
> resulting code will be very much tied to Mac OS X.


It really depends on how you setup your logic.  If you follow the MVC  
and separate your GUI from your logic then you should be able to have  
one code base that deploys on multiple systems.  You will still need  
to write GUI code for each platform though.  This might change with  
WMF...

Adhamh
