I've been working on Toolbar and ToolbarItem and I'm just about done  
and I'm testing Toolbars properties now, and I've noticed something  
odd...

The Toolbar has an initWithIdentifier constructor, this assigns it a  
unique name, it's argument is a cocoa.string.
I created the property that allows you to read the Identifier, much  
like the Identifier property on TableColumn..but this one acts  
different.

At first I tried having it return a string like I had to do with the  
TableColumn.Identifier property, well it worked using the following  
code...

public string Identifier {
			get {
				return (string)Native.NativeToManaged ((IntPtr)  
ObjCMessaging.objc_msgSend (NativeObject, "identifier", typeof  
(System.IntPtr)));
			}
		}


Then I remembered what I think kangaroo, and Andreas said about the  
importance of using Cocoa.String, so I swapped the code out to this...

public Cocoa.String Identifier {
			get {
				return (Cocoa.String)Native.NativeToManaged ((IntPtr)  
ObjCMessaging.objc_msgSend (NativeObject, "identifier", typeof  
(System.IntPtr)));
			}
		}

And guess what...it still worked.  That makes me curious as to why  
the TableColumn.Identifier property when set up the same way throws  
an error when you try to return it as a Cocoa.String.  It doesn't  
make sense, just thought I would ask the group and see if any one had  
a clue...  The one thing I did do differently in this case is that I  
defined the Toolbar in the code and assigned the identifier with the  
constructor, when I use a tableview and tablecolumns, I assign the  
identifier in the inspector window of Interface Builder, could that  
have something to do with it???
