On 12-Feb-06, at 1:10 AM, David Mitchell wrote:

> You’ve never seen me on this list before, but here’s a patch that I  
> banged out today.
>
>
>
> * Menu.cs
>
>             (Constructor) - initialize from native object
>
>             (Constructor) - initialize with title
>
>             (Constructor) - initialize with “” as title
>
>             (AddItem(string, string string)) – Added method
>
>             (AddItem(string, ActionHandler, string)) – Added method
>
>             (AddItem(menuItem)) – Added method
>
>             (Init) – Added method
>
>
>
> *MenuItem.cs
>
>             (Constructor) – initialize with title, action  
> (ActionEvent), and keyEquivalent
>
>             (SetAction) – Added method
>
>             (Click) – Fixed. (It was setting the action to an  
> NSString instead of a selector)
>
>             (SubMenu) – Added property
>
>             (Enabled) – Added property
>
>             (Other methods) – Modified to use  
> Cocoa.Object.SendMessage()
>
>
>
> *Application.cs
>
>             (SetAppleMenu) – Added method
>
>             (MainMenu) – Added property
>
>
>
> *Object.cs
>
>             (SendMessage) – Added two methods (with and without  
> additional arguments)
>
>             (SendMessageToSuper) – Added method
>
>             (Allocate) – Added method
>
>             (SendMessageToClass) – Added two methods (with and  
> without arguments)
>
>
>
> There are two main things accomplished by this patch.  The first is  
> that menus can now be defined completely independently of a nib  
> file (as a result of this, one could now create a very minimalistic  
> Cocoa# application without having any nibs).  The second is the  
> addition the SendMessage* functions to Cocoa.Object; while these do  
> not enhance the functionality of the code at all, they do allow for  
> much more concise code, in certain cases.
>
>
>
> Incidentally, is anyone actually trying to use MenuItem.Click?  It  
> seemed to be quite broken when I found it, and since it doesn’t  
> really conform to the way that C# events generally work, I think  
> it’s probably doing more harm than good right there.  If no one is  
> opposed, I’d like to either remove it entirely or replace it with  
> something that is more consistent with C# events.
>
>
>
> Thanks
>
> --Dave
