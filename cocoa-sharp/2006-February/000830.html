On 19-Jan-06, at 7:30 PM, Andreas Färber wrote:

> Hi guys,
>
> Recent I reported some Mach exception handling issues, especially  
> that the thread is never stopped in command line applications  
> (including NUnit tests). Having managed to get my application  
> running on my new implementation (without the previously observed  
> failures with gmcs), I have now found out why a Cocoa# AppKit  
> application exits nontheless, and it is quite disturbing...:
>
> By calling [[NSApplication sharedApplication] run] (Application.Run 
> ()) we are passing control out to unmanaged code which through the  
> bridge calls back into our managed code. However when the  
> application is quit, control is not tranferred back to our managed  
> Main function that called it!
> Instead native Cocoa is calling the C exit() function (or  
> equivalent) on its own, causing Mono to be "killed" without any  
> garbage collection!!!
> This is bad.

Why is this bad per se?  Garbage collection and finalizers are not  
guaranteed to be run by the C# spec.
