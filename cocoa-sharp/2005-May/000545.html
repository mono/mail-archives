>b) I could use nibtool, which generates a translation dictionary of the 
>form
>"origString"="translatedString";.
>nibtool -L my.nib > my.strings
>nibtool -w my_en.nib -d my.strings my.nib
>This works fine for the first translation. I then make a cp -R of my.nib to
>my.bak.nib and of my_en.nib to my_en.bak.nib.
>For the next translation, I could then call
>nibtool -I my.bak.nib -p my_en.bak.nib -W my_en.nib -d my.strings my.nib
>BUT this requires a new dictionary. And when I re-generate it using the
>above command, all my previous translations are lost! They are definately
>not taken from the backup files, I've tried that. Any clues on how to
>proceed? (my guess would be to somehow merge the newly generated dictionary
>file with the previous translation, but how?)

The nib files have to have the same name.  Instead of changing the name, 
change the relative path.
Don't try and do everything in one folder.  keep your dictionaries in 
separate folders as well.

>
>Also, the Application.LoadNib function that I had been pointed to does not
>seem to search my.app/Contents/Resources/English.lproj/my.nib?
>

It won't search English.lproj if there is already a nib in the Resources 
Folder.  It is a "top down" search so that some resources can be 
non-localized and some localized.

Good luck.   No matter what, simultaneous support of many languages in Cocoa 
is complicated and time intensive.  We only localize during our release 
process and then it is done by someone else (not me!) on our team.  If you 
really need to support many languages, you may need to increase your team.  
There's enough to worry about in Cocoa without worrying about supporting 15 
language versions of the App.
