Hi Paolo,

>The C# params keyword has nothing to do with varargs, it is just a C#
>compiler convention that has no effect on the underlying call
>convention. So it can't be used.
>  
>
Thanks for clarifying this!

>In the future we'll add support in mono to be able to use different
>signatures for the same p/invoke method, so in that case you'll be able
>to use the CLR vararg support (which is currently available for
>non-pinvoke methods) also for this task. But this won't bring
>significant benefits, so it's way down in the todo list.
>
The benefit would be to get rid of the necessity to use dynamic 
assemblies built on-the-fly for the overloads we are using; this is not 
nice to read and it probably has some impact on Code Access Security too 
- P/Invoke requires SecurityPermission for UnmanagedCode only and not 
also ReflectionPermission for ReflectionEmit. Not too important right 
now given the state of CAS on Mono (if I'm up-to-date on that) but it 
will need to be taken into account sooner or later.

I guess my best bet under these circumstances will be to try to 
factorize out this P/Invoke functionality into its own interop class...


Is there also a particular reason why Cocoa# is using a private static 
string ObjectiveCName field (causing 0169 warnings) as opposed to 
applying an Attribute to the classes?
