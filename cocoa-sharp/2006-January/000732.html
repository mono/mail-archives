We could just keep SheetHandler and make the first delegate argument  
a Cocoa.Object. I thought it might be more straightforward usage wise  
to have type specific handlers.

> 5: Native.ManagedToNativeArray seems to throw out primitive types;  
> is this correct?  Also foreach would probably be a little more  
> elegant here.

I'm currently just using it for System.String arrays. I figured we  
could add to it as more types were needed. Docs reflect this. I'll  
convert to foreach.

BTW: is there a simple test to see if an object is a primitive type  
that can be marshaled directly?

> 6: If you want to move all the DllImports into NativeMethods.Foo  
> I'm cool with that; but lets do it to all of them and split it out  
> into its own .cs

I needed a way to get sel_getUid to take a string instead of a IntPtr  
since Native defines and uses it that way elsewhere. Either way  
you'll probably want to have to have the same Native function in two  
different places unless there's some way of overloading a PInvoked  
method that I'm unaware of.
