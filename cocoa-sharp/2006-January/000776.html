On 1/14/06, Andreas Färber <andreas.faerber at web.de> wrote:
> > Objective-C allows automatic replacement of a method by means of class
> > posing. Maybe, using the engine's APIs...
> Yes, but posing as *each* Objective-C class would be a big task
> wouldn't it? Or would we only have to pose for NSObject?

Yes to the latter. Once a class poses as NSObject, it replaces
NSObject entirely, including dynamically making any object that
inherits from NSObject an instance of our new class instead.

> >> Another thing I have been reading about is the possibility of
> >> integrating Garbage Collection into Objective-C. This is mentioned in
> >> Apple's GCC documentation but it does not elaborate on how this would
> >> be
> >> working, but it mentions the Boehm GC on which I believe Mono's libgc
> >> is
> >> based too.
> >
> > Garbage collection will be added in Leopard. See man gcc, at the
> > -fobjc-gc switch.
> What I had seen is described in GCC 3.3 and 4.0.1 documentation (when
> searching for "garbage collection"), referring to an alternative
> libobjc_gc.a runtime library. I don't want to use such GC'ed native
> code myself but I thought if there was this possibility then we might
> be able to utilize it for Mono somehow...

It seems like nobody will use it directly -- you just structure your
application to use -finalize rather than -retain/-release and gcc
makes the necessary substitutions and links to the right routines at
runtime. How it will be implemented is only up to Apple, and only with
Leopard (or its betas) we'll know how it will work exactly (and how we
can "hook" with it).
We can simply override NSObject's -finalize by posing the same way we
can override -dealloc.
