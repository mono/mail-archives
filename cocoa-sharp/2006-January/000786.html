Hi Aaron,

> The problem he had does bring the 'where to alloc and init' issue into 
> light again. I think we need to define some guidlines for where the 
> allocation and initialization of objects should be done. Should they 
> never be done in default constructors? Should they be done only for 
> default constructors of classes that can't be inherited from? Should 
> we leave it up to the cocoa-sharp user to alloc and init (not ideal)? 
> I can definitely see the alloc and init thing causing all kinds of 
> confusion in the future.
The thing is this: In order to allow subclasses of NSObject to have 
non-standard init... methods we cannot put a call to init in the 
default Object constructor. Therefore user subclasses will have to call 
init themselves, me too I haven't found a way around this.
Alloc however should be sent implicitly by Object.ctor().

I had proposed this but have not worked on this yet. So if you have a 
quick fix feel free to patch it in. I have only just arrived at adding 
stub classes into Objective-C and mostly understanding the Mach 
exception handling messages.
