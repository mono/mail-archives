On 01/11/06 Andreas Färber wrote:
> The benefit would be to get rid of the necessity to use dynamic 
> assemblies built on-the-fly for the overloads we are using; this is not 
> nice to read and it probably has some impact on Code Access Security too 
> - P/Invoke requires SecurityPermission for UnmanagedCode only and not 
> also ReflectionPermission for ReflectionEmit. Not too important right 
> now given the state of CAS on Mono (if I'm up-to-date on that) but it 
> will need to be taken into account sooner or later.

Since UnmanagedCode permission is already needed and is the most
dangerous anyway, requiring Reflection permission is not a big deal.

What would be nice, instead, is a version of DynbamicMethod for
P/Invoke so it would be easier to create the methods and it would be
somewhat nicer, IMHO. This would be an extension available on Mono, but
this stuff works only on mono anyway.
