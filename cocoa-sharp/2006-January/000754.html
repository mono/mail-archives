Urs C Muff schrieb:

> You could register into a map on constructor and unregister on 
> destructor, however that has one problem: if an objective-c object has 
> a much shorter lifetime then it's .net counterpart, the same intptr 
> might be some completely different instance now [different type and 
> everything].  However if you create an objective-c object from within 
> .net using the non-intptr constructor, that object lifetime is known 
> much better, and it could register itself into a global map that could 
> be used in native2managed... 

I was wondering anyway why Cocoa.Object doesn't implement IDisposable; 
if Cocoa# were able to detect a native Objective-C object being 
destructed it could just dispose its managed counterpart and have the 
Dispose method (instead of a finalizer) remove it from the map way 
before it is GC'ed/finalized.

Another way to control object lifetime would be to inherit from 
System.MarshalByRefObject and work with ILease but that may be too much 
overhead, don't know, i have only used it for .NET Remoting. And the 
inner workings of Cocoa# are still a big puzzle to me...
