On Dec 31, 2005, at 7:10 AM, Andreas Färber wrote:
>
> If you look at my patch, you'll see that it too includes sheet  
> support for Alert, not too nice yet though.
> After all this time we're even getting conflicting patches, that's  
> what I was trying to avoid...

Yeah. I've been sitting on that since Dec 8. Didn't see the alert  
sheet in your notes or changelog.


>
>> - Added the ApplicationIconImage property to Application.
>
> Can we maybe simplify this name for .NET developers? Maybe  
> Application.Icon?

Makes sense.


>
> - I believe the AlertStyle remarks is the wrong place to document  
> default values used somewhere else. This should be in the value  
> element of Alert.AlertStyle. This would also apply to other  
> properties, such as Panel.FloatingPanel.

True.


> - Independent of what mechanism we use for the sheet callbacks, I  
> would think we should use Cocoa.Object instead of IntPtr for  
> contextInfo.

That was my initial feeling but as Kangaroo pointed out, contextInfo  
could be ANYTHING, so IntPtr was used.


> - Regarding AlertWithMessage I was actually getting crashes because  
> it didn't support null (only String.Empty) as second and third  
> string argument - can you add this feature, or do you want me to?

I'll add tests for null.


> - Regarding Application.ApplicationIconImage documentation: The  
> incorrect default icon is a bug in Cocoa#/Mono and therefore should  
> not be documented as is. With a different launcher the Info.plist  
> icon is working just fine.

The icon specified in the Info.plist shows up in the dock when you  
first launch the appliation, but calling the applicationIconImage:  
selector returns the default one. What launcher are you referring to?  
Not the one generated by macpack, right?

Since this is the current behavior of the library I think the  
documentation should warn people about this and give them a  
workaround until it's fixed.


> - I noticed that you too needed to import [DllImport  
> ("libobjc.dylib")] internal static extern IntPtr sel_getUid (string  
> name); I would propose to move this functionality into a method  
> public static IntPtr Native.ToSelector(string) so that we do not  
> have to import it wherever we need it.
>

Agreed. That idea popped into my head too as I was implementing the  
sheets.
