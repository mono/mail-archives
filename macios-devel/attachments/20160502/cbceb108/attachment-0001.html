<div dir="ltr"><div style="font-size:13px">Bonsoir!</div><div style="font-size:13px"><br></div><div style="font-size:13px">One of the tricky aspects of working, and now contributing, to Xamarin.[iOS | Mac] is dealing with breaking changes.</div><div style="font-size:13px"><br></div><div style="font-size:13px"><img src="https://i.imgflip.com/13houv.jpg" class="" tabindex="0"><br></div><div style="font-size:13px"><br></div><div style="font-size:13px">Whenever a public/visible API is modified the consumers of the assemblies are at risk of becoming unusable. That's a big problem when working with assemblies binaries (e.g. 3rd party components) because re-compiling them is not an option.</div><div style="font-size:13px"><br></div><div style="font-size:13px">This is why we strive to never [0] introduce breaking changes in our stable releases and, as much as possible, in any beta or alpha releases.</div><div style="font-size:13px"><br></div><div style="font-size:13px">Still that does not solve the issue of the broken API itself. To do this we need to take a few steps:</div><div style="font-size:13px"><br></div><div style="font-size:13px">1. <b>The workaround</b>: Add an [Obsolete] attribute to the "broken" API, explaining why and the best way/API to replace any code using it. Never remove or modify the "broken" but visible API;</div><div style="font-size:13px"><br></div><div style="font-size:13px">2. <b>The short term fix</b>: Add the new, fixed, API;</div><div style="font-size:13px"><br></div><div style="font-size:13px">3. <b>The long term fix</b>: At some point we might have to break compatibility (e.g. classic vs unified) or have a clean up opportunity (e.g. new platforms like watchOS and tvOS). At those stage we can "remove" the workaround/broken API.</div><div style="font-size:13px"><br></div><div style="font-size:13px">To allow this we use defines for profiles, e.g. </div><div style="font-size:13px"><ul><li style="margin-left:15px">XAMCORE_2_0 was used unified (versus classic);</li><li style="margin-left:15px">XAMCORE_3_0 is for tvOS / watchOS (but not for iOS or Mac); and</li><li style="margin-left:15px">XAMCORE_4_0 for a future (not shipped) break - <u>that might never happen</u>;<br></li></ul></div><div style="font-size:13px">If you have looked into our code then you likely have see those define quite frequently.</div><div style="font-size:13px"><br></div><div style="font-size:13px"><u>Now don't worry if any of this sounds confusing.</u> Everyone working on XI/XM always to double check each others to avoid those breaking changes from sneaking in :-) </div><div style="font-size:13px"><br></div><div style="font-size:13px">So if you see anything broken report it and/or do a pull request (PR) while we'll continue watching for them (and your help for this will be welcome). We will soon have a PR to show how it's "done" to solve PR #19 [1] without breaking changes.</div><div style="font-size:13px"><br></div><div style="font-size:13px">Thanks!</div><div style="font-size:13px">Sebastien</div><div style="font-size:13px"><br></div><div style="font-size:13px">[0] like any rule there's a (very) few cases where it cannot be applied. Whenever in doubt don't hesitate to ask<br></div><div style="font-size:13px"><br></div><div style="font-size:13px">[1] <a href="https://github.com/xamarin/xamarin-macios/pull/19" target="_blank">https://github.com/xamarin/xamarin-macios/pull/19</a></div></div>
