Hi all guys!
I'm writing a 3D engine using TAO Framework... searching around for
optimization, I found the NeHe lesson 45, about Vertex Arrays and VBOs... It's
quite interesting, so I decided to use the solution presented in the tutorial to
render scene with my engine... and here a problem comes...
In NeHe sample, Paul Frazee initalizes arrays in this way

// Generate And Bind The Vertex Buffer
glGenBuffersARB( 1, &m_nVBOVertices );// Get A Valid Name
glBindBufferARB( GL_ARRAY_BUFFER_ARB, m_nVBOVertices );	// Bind The Buffer
// Load The Data
glBufferDataARB( GL_ARRAY_BUFFER_ARB, m_nVertexCount*3*sizeof(float),
m_pVertices, GL_STATIC_DRAW_ARB );

Now... m_nVBOVertices is a uint, m_nVertexCount is the total vertices number,
and m_pVertices is a CVert*.
CVert is a struct defined as:
class CVert // Vertex Class
{
public:
	float x; // X Component
	float y; // Y Component
	float z; // Z Component
};
typedef CVert CVec;

Sosize, to load vertices data into the buffer, glBufferDataARB is called... the
size is 3*of(float)*TOT_VERTICES, and the pointer (that should be a GLVoid*...
or simply a void*)... so, I suppose that C++ structs are allocated in contiguous
memory blocks, so the 3 floats of CVert are like an array of 3 float.

Now the real question... if I use a struct like CVert (I'm using Sharp3D.Math,
and Vector3D to store vertices) in C#, I will experiment the same behave? I
looked at Gl.glBufferDataARB(...)... the function has 38 overloads, among them
there is one which takes a double[], but I fear that casting a struct to a
double[] is not a good idea in C#... so, should I use directly double arrays to
store vertices? Is there someoune with a snippet of code about this (in C# of
course :P )... I hope my question is not so silly... unfortunally I couldn't
find infos about glBufferDataARB (or Gl.glBufferDataARB) over the Net...
