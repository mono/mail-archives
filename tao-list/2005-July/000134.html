Last week - that's when I wanted to be finished, but Real Life has an
annoying tendency to get in the way.  The correct answer is probably the
classic one: "When it's done."  It should be weeks rather than months -
now that I've got things functioning on Linux where all my tools are.
 
It's a bigger job than I originally thought even though I'm not having
to write most of the code from scratch.  I'm checking every method
against the actual ODE code, comparing to the original Tao.Ode where
there is overlap, branching off now and then to read up on .NET interop
when I need to *and* reading the ODE userguide describing the code I'm
looking at.  That adds up to much bouncing around between windows and
not so much typing.  I also decided to start porting test_ode.cpp (part
of the ODE source) to C# so I have some assurance that the bindings are
working (so far they are).

Currently all the Mass functions
(http://ode.org/ode-latest-userguide.html#sec_9_2_0) are done and
partially verified with the test code mentioned above.  The plan is to
tackle the following, roughly in this order:  the world functions, the
rigid body functions, the joint functions, the geom functions, the
collision detection functions, the space functions and the trimesh
functions.  Now that I've got a rhythm going, I'm hoping for steady
progress, slow though it may be.

As mentioned in a previous post, the plan at the moment is to plough
through everything, get the basic bindings done, then post everything at
once.  If it's clear that it's just going to take time, I may try to
define some logical 'chunks' of functionality (probably the functional
categories mentioned above) and post those as I finish them.  Just so
people don't have to wait too long and can have something to play with.

>Also is the perf/precision trade off for single
>vs. double precision significant? 
>
Haven't tested this, but probably yes.  The ODE docs say what you might
expect:  "The ODE library can be built to use either single or double
precision floating point numbers. Single precision is faster and uses
less memory, but the simulation will have more numerical error that can
result in visible problems. You will get less accuracy and stability
with single precision."

>Has there been any benchmarks of this?  
>  
>
A quick google turns up nothing.  This information might be available
from the ODE community.

>Also, what has changed so far in terms of the OO part of

Nothing yet.  The initial focus is on getting the basic bindings done. 
There were a number of classes in the code Randy sent me that provide an

>and are structs now used instead of IntPtr's?

>Also will the
>real using alias be replaced with the float keyword?  It seems much
>less ugly that way IMHO and you can always search-replace for float if
>need-be and that would be easier then changing the using alias for
>each source file.  
>  
>
No.  In fact the alias is 'dReal', and there are more aliases now to
more closely match the ODE data types (see
http://ode.org/ode-latest-userguide.html#sec_4_2_0).  Ugly it may be
from a C# purist's perspective, but it's much easier to follow the ODE
documentation and read/port code this way.  It doesn't bother me so much
in the direct bindings since they are trying to adhere as closely as
possible to the published API.  The OO wrappers would be the place to
clean this up if it bothers people too much.  And once this is out of my
hands folks can do what they want.  The aliases definitely make things
easier right now.
