> 1. Should we use autoconf? 
Oh lord, yes.
> Would this help make Tao become part of
> distributions?

Yes.  I'm concerned with Gentoo at the moment - a lot of convenience 
functions in portage rely on the presence of autotools.    And Debian 
Tao packages are using  C.J. Collier's Tao autotools.  I don't see any 
harm in formalizing this, for the non-Windows side of things at least.
> 2. Should Tao be released in one large package or should each part be in
> its own package? For Linux, having separate packages may be preferable, but
> probably not on Windows and OSX.
>   
Being able to target specific parts at the build level would be useful  
- this would make it easy to package the various Tao bits individually 
for environments where this is useful, but still allow building 
everything for creating a monolithic package for convenience on 
Windows.  I think for Linux the best approach is to make things easy for 
packagers of various distributions (in the case of autotools, something 
like "--with-ode --with-sdl" as arguments to 'configure'), but still 
provide a large package for download as a fallback for those who want 
it.  The current approach used for Windows (one large installer package) 
is nice, though I'm not sure how updates for individual Tao pieces would 
be applied.
Don't know much about OSX.  Anyone out there have any info to offer?
> 3. What is the relationship of Tao with Mono? Would it be
> possible/preferable to be like GTK# where it is part of the Mono release
> but is also separately available?
>   
Something I've been wondering about, considering where Tao currently 
lives.  I would provisionally say "yes" to all parts of your question.  
Assuming the "Tao Philosophy" is a reasonable and accepted approach, it 
would be nice if Tao became a conduit for native bindings of all sorts, 
and perhaps a preferred mechanism for gestating and producing high 
quality native bindings that could be considered part of the 'official' 
Mono ecosystem, available as part of a typical Mono install (or at least 
an official, but optional add-on).  But since Tao also aims to provide 
cross-platform bindings for multiple CLR implementations (including the 
not-to-be-ignored MS.NET community), separately available binaries would 
be a must.

* The Tao Philosophy:  A good Tao binding is a low-level, high quality,
> low level binding to a particular native library which sticks to the
> conventions and architecture of the library as closely as possible even
> when these deviate from conventional .NET ideoms.  A Tao binding is a
> thin veneer over the native library that should feel comfortable to
> someone familiar with that library, and artifacts associated with that
> library (documentation, coding conventions, etc.) should be immediately
> useful.  To provide an interface more comfortable to a developer
> expecting .NET ideoms, a 'sister' framework can be created on top of the
> Tao binding which presents a more comfortable interface conforming to
> .NET ideoms to a user.  Examples of this are Tao.SDL/SDL.NET and
> Tao.Ode/OdeDotNet.  This model seems comfortable to people I've
> discussed it with.
