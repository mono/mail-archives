> actually, it would be better to put all those const ints in enums
> because then you can actually see the enum names in the debugger
> instead of just some number..
>
> also, because Gl has such a huge number of methods it seriously slows
> down intellisense in vs2005, which is annoying because it makes vs
> wait a couple of seconds everytime you type in Gl.
> Any ideas on how to solve that?
>
>  
>
I think the only way to solve the problem of intellisense being slow, at 
least from our vantage point, would be to place related functions in Gl 
in separate classes. However, I believe that doing this would cause many 
more problems, as there are functions which may belong to more than one 
group. It would be better to create a managed class-library wrapper that 
sits on top of unmanaged OpenGL, much like how Managed DirectX sits on 
top of unmanaged COM-based DirectX. This new wrapper would be a lot 
different from Tao.OpenGL, so it'd have to be a separate project.
