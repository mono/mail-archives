The only reason the long lived <a href="http://ASP.NET">ASP.NET</a> application is important is if you are sending a lot of notifications frequently, you need to maintain the same TCP connection to apple&#39;s push notification servers.  You wouldn&#39;t want to be creating a new connection for each message.  <div>
<br></div><div>In practice, you would probably be fine with creating a global instance of the library&#39;s NotificationService object in your Global.asax... Realistically that instance should be ok until the app pool is restarted, which I don&#39;t think would happen terribly often if there is a lot of activity.</div>
<div><br></div><div>I&#39;m not familiar with Manos de Mono... The only reason I would suggest you could rig up <a href="http://asp.net">asp.net</a> to be long lived and make this work, is if you wanted to do hosting really cheaply (eg: a shared hosting provider where you can&#39;t run custom executables).  There&#39;s of course nothing stopping you from making a nice .net service using this library directly (which is exactly what I do).  </div>
<div><br></div><div>It would be interesting though to see if you could make a similar application work over <a href="http://ASP.NET">ASP.NET</a>.  In my case, my server application has to monitor a bunch of IMAP connections, without the users necessarily contacting the web service subsequent times.  So for me the challenge would be making sure my <a href="http://asp.net">asp.net</a> app stays alive with nobody hitting pages on it.  </div>
