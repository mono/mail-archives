<p class="MsoNormal" style="margin-bottom:12.0pt">Hi,<br>
<br>
I tried your projects, and got somewhat different results (on an iPad2).<br>
<br>
Objective-C implementation: 0,296s<br>
MonoTouch (release mode): 0,677s<br>
MonoTouch (release mode, with LLVM enabled): 0,397<br>
<br>
So just MonoTouch release mode is 2.3x times slower, but once you enable LLVM, this goes down to 1.34x slower.<br>
<br>
I also noticed that BitmapContext.FillRect is called a lot (39635 times), so I commented it out and now I get these timings:<br>
Objective-C implementation: 0,224s<br>
MonoTouch (LLVM enabled): 0,293s<br>
<br>
So just calling FillRect 39635 times takes 0,072s in ObjC and 0,104s with MonoTouch - the discrepancy is due to the fact that the MonoTouch version obviously has to do more work (it has to do a managed-to-native transition (and back again)). So now MonoTouch
 is 1.31x slower.<br>
<br>
The point is that LLVM mode is exactly for this kind of cpu intensitive tasks, since our jit doesn't generate as optimized code as the LLVM one can.<br>
<br>
Now I started looking a bit deeper, and realized that the loop in IsInMandelbrotSet loops 100 times in the MonoTouch version and only 50 times in the ObjC version. Redoing the tests in MonoTouch, but this time only looping 50 times, I get this:<br>
<br>
MonoTouch (LLVM enabled): 0,290s.<o:p></o:p></p>
<div>
<p class="MsoNormal">Yes, that's correct. The MonoTouch (with LLVM) version is actually 1%
<strong>faster</strong> than the ObjectiveC version with identical code.<o:p></o:p></p>
</div>
<div>
<p class="MsoNormal"><o:p> </o:p></p>
</div>
<div>
<p class="MsoNormal">And as shown this difference can probably be increased (by not calling FillRect 40k times, but perhaps manually create the bitmap in managed memory and then draw the bitmap with one call - this change would likely benefit both implementations,
 but it would benefit MonoTouch most).<o:p></o:p></p>
</div>
