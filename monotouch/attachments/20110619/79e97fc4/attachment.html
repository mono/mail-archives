Nic,<br><br>for a variety of reasons (mainly the number of testers, and different types of testing - including &quot;content&quot; testing) we have three &#39;versions&#39; of our app... purple, orange and green. They are effectively &#39;white label&#39; along the lines you describe (different text, images, UIColors and accesses different back-end webservices).<br>
<br>we do it with compiler directives (and some post-build &#39;custom commands&#39; in MD)<br><br>images references (ie the path) and customizable text are all in &#39;constants&#39;, which change based on the #if PREVIEW or #if DEBUG directives.<br>
<br>we then create different build configurations that include/exclude the relevant compiler directives. the post-build command that exists in some builds to swap plists around looks like this <a href="http://twitpic.com/5di7mk">http://twitpic.com/5di7mk</a><br>
<br>for simplicity all test/debugging builds ships with ALL the images embedded, and the compiler directives just switch which ones are referenced. for appstore build there&#39;s a pre-build step to delete the unwanted image folders so they don&#39;t end up in the final .app <br>
<br>the one thing we don&#39;t care about, but you would have to manage manually, is the BundleIdentifier set in Options-iPhone Application... while you can easily assign specific prov-profile/certs based on build configuration, the BundleIdentifier is &#39;hardcoded&#39;... forgetting to change that would be a real pain!<br>
