<blockquote class="gmail_quote" style="margin:0 0 0 .8ex;border-left:1px #ccc solid;padding-left:1ex">SSL: it depends on the data. SSL is fairly cheap and easy to do, but<br>
if the data is not sensitive at all, maybe you dont need to. It's easy<br>
to change later tho (ie, before release, but after development)<br>
<br>
Authentication: A username/password is quite a good way (or rather,<br>
email + password) if they need to get at it from other devices. If<br>
it's just from the one device, you could use the UDID (or generate a<br>
GUID) and use that, so you know which device it is. If they later need<br>
to register to use the website (via the phone?), then associate a<br>
username/password with that account.<br>
<br>
You could just ask for an email to identify them, and set a random<br>
password until they set one (and they can't use the website until they<br>
set one)<br>
<br>
It does depends on how secure the data needs to be. Instapaper just<br>
went on a username (email) for ages, tho it now has passwords. But the<br>
data in there is pretty generic....<br>
