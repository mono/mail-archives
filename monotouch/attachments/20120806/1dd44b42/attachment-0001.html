</div>I suggest a small change of design to minimize your memory requirements:<br>
<br>
1. Avoid keeping data in memory.<br>
<br>
1.1. The ZIPped data is likely not useful to keep around, i.e. it<br>
won't be used until decompressed (into one or several files), but even<br>
if it was...<br></blockquote><div><br></div><div>I'd love to but I never even access the byte[] that has been downloaded in this case.</div><div><br></div><blockquote class="gmail_quote" style="margin:0 0 0 .8ex;border-left:1px #ccc solid;padding-left:1ex">
1.2. The 20MB download may take quite some time (e.g. cellular) and<br>
you do not want to hold on to large memory blocks for long times (e.g.<br>
be nice to other running applications);<br></blockquote><div><br></div><div>After watching the talk at WWDC 2012 on memory, that's the thing I'm trying to do now, avoid spikes of memory allocations.</div><div> </div>
<blockquote class="gmail_quote" style="margin:0 0 0 .8ex;border-left:1px #ccc solid;padding-left:1ex">
<br>
2. Keep using WebClient as it is simpler than HttpWebRequest - but<br>
instead of using DownloadData use DownloadFile [1] to a temporary<br>
location. That will download your ZIP file in chunks of 32KB, lowering<br>
your app memory requirements, and the file can be decompressed later<br>
(e.g. using streams to avoid loading it into memory) and then deleted.<br>
<br>
Speaking of streams an alternative would be use a OpenRead to get a<br>
Stream and process data, in small chunks, as it comes in. You get more<br>
control this way (if you need it) but you'll likely end up duplicating<br>
DownloadFile.<br>
