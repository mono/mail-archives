Actually, the problem with the tab bar controller occurs when you don&#39;t have your (populated) tab bar controller in your main nib.  For example, if you had a splash screen/login screen that you were trying to get up as quickly as possible and then loaded the tab bar controller separately.  In my app, that&#39;s what I&#39;m trying to make happen as quickly as possible.<br>
<br>However, your second point in interesting, if you&#39;re doing stuff in loadView (or viewDidLoad) then I think your code is still being executed during nib load time.  That&#39;s why I was suggesting if you want to really get the maximum benefit of not using a monolithic nib, you&#39;d look at overriding viewWillAppear, which should be getting called when you actually cause a specific tab view to be displayed, so you could defer constructing the UI for that tab till that point.   As I said before, the idea isn&#39;t that you&#39;re going to get your speed improvement just by doing the same thing in your code that  initWithNibName:bundle: does, it&#39;s going to come from breaking up the work so that it doesn&#39;t happen all at once when you load the main nib.<br>
