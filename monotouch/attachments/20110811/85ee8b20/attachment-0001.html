Hi Kent,<div><br></div><div>Sorry for the delayed response, I was hoping someone else more knowledgeable would answer. Turns out, I just had to do this myself. The solution is sub-optimal, so hopefully we can improve things a bit in a newer MonoDevelop because right now it&#39;s a bit like jumping through hoops.</div>
<div><br></div><div>The mtouch tool has an option `--xcode=[path]` that can be used in MonoDevelop&#39;s Project Options in the &quot;iPhone Build&quot; section (or you can just use mtouch directly in a terminal). What you need to do is create a new project directory (which will be used to dump some autogenerated assembly code to). Once you&#39;ve done that, pass that option to mtouch by adding --xcode=/path/to/project to the &quot;Additional arguments to pass to mtouch) textbox and then build your app in MonoDevelop (the Debug target may be the most useful, I&#39;m not sure). This will dump the AOT&#39;d assembly source code to /path/to/project but is, unfortunately, missing a few things to be a valid Xcode project. What you&#39;ll need to do next is to copy the Info.plist file from your MonoDevelop project into /path/to/project. This is where things get a little more complicated...</div>
<div><br></div><div>MonoDevelop seems to generate files like System.dll.6.s if it was configured to target armv6 (probably named System.dll.7.s if targeting armv7), but the project.pbxproj file that MonoDevelop autogenerates refers to System.dll.s so what you&#39;ll need to do is edit the project.pbxproj file (/path/to/project/&lt;projectName&gt;.xcodeproj/project.pbxproj) and fix the *.s references. Elsewhere in the file, you&#39;ll find a section defining OTHER_LDFLAGS which will wrongly include &quot;-u&quot; &quot;_catch_exception_raise&quot;. If you remove that, then opening the autogenerated project and building it should work.</div>
<div><br></div><div>The resulting app will have symbol names that are easy to map to their equivalent C# names, and, while not having &quot;friendly&quot; C# source code to read, will at least allow you to have a clearer picture of what is going on when you get that crash.</div>
<div><br></div><div>This method will allow you to debug using gdb on the device as well, which can be really helpful.</div><div><br></div><div>Hope that helps and sorry that this isn&#39;t a lot easier to do (if it were up to me, it would be a single button click!).</div>
