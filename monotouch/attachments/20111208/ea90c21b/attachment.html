<ul><li style="margin-left: 15px; ">ServiceStack is supported on Mono on either <a href="http://asp.net/" target="_blank" style="color: rgb(17, 85, 204); ">ASP.NET</a> or HttpListener hosts.</li><li style="margin-left: 15px; ">
I personally believe it promotes the &quot;right way&quot; to develop web services (i.e. based on remote services best-practices), for a quick summary between WCF and ServiceStack API design see: <a href="https://gist.github.com/1386381" target="_blank" style="color: rgb(17, 85, 204); ">https://gist.github.com/1386381</a></li>
<li style="margin-left: 15px; ">It&#39;s non-invasive and intuitive and there is very minimal API/configuration that gets in your way as your code-first POCO DTO&#39;s is what&#39;s used to define your API. </li><li style="margin-left: 15px; ">
It&#39;s more idiomatic to C# as it lets you return POCOs and exceptions are automatically converted and serialized with ideal HTTP error codes, etc. (The HTTP output is also completely customizable)</li><li style="margin-left: 15px; ">
Each service supports the can return any <a href="http://stackoverflow.com/questions/6245616/does-servicestack-support-binary-responses" target="_blank" style="color: rgb(17, 85, 204); ">http://stackoverflow.com/questions/6245616/does-servicestack-support-binary-responses</a></li>
<li style="margin-left: 15px; ">You can re-use your DTOs used to design your Web Services with on your client which provides succinct typed (Sync/Async) generic ServiceClients without any code-gen (e.g <a href="http://www.servicestack.net/RestFiles/#!files/tests/SyncRestClientTests.cs.txt" target="_blank" style="color: rgb(17, 85, 204); ">http://www.servicestack.net/RestFiles/#!files/tests/SyncRestClientTests.cs.txt</a>)</li>

<div style="direction:ltr;font-family:Tahoma;color:#000000;font-size:10pt">Go give Web.API a scan.. Its not your normal WCF experience.. I had REST services up and running in no time.. And if you return iqueriables you can actually query and filter your
 data via querystring args you pass in.. So rather than writing a &quot;query service&quot; or handling how you want to query data.. you can use the OpenData protocal and the new Web.API stuff will take the results from your service call (as an iqueriable) parse and

 I actually reuse those POCO cs files in my mobile app and <a href="http://JSON.NET" target="_blank">JSON.NET</a> deserializes right in to them.. Because they are shared source, change the POCO in one file and both my client and my server continue to talk the same language :-).. <br>
