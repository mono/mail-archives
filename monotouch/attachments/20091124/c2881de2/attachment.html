I believe that the overhead of NIB&#39;s doesn&#39;t come from the loading of the NIB file itself.  As you point out, it&#39;s compiled to a binary file.  If using NIBs actually had a lot of inherent overhead, it would be trivial to create a tool like the MonoDevelop code-behind to compile them to C# code that you could hand edit (which actually, would be pretty useful).  I think that the issue is the lazy instantiation of views loaded from NIBs isn&#39;t all that lazy and always ends up instantiating more than you probably need at startup.  You can see this if you put logging code in your viewDidLoad methods.  Now, one approach is to break your UI into multiple XIBs.  However, if you&#39;re using a UITabBarController as your main controller, which isn&#39;t all that uncommon, you&#39;re going to find that nothing works quite as expected, which is why most people recommend against it.  If you&#39;re coding things by hand, you&#39;re probably only going to get the benefits if you do a lot of UI construction in your viewWillAppear methods.  However, I&#39;d take any recommendations along these lines with a grain of salt and test any approach empirically.<br>
