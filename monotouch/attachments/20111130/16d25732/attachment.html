I can only answer to the first question, as WCF support in moonlight was very limited and I&#39;m not sure what is new in SL4 WCF (we absolutely don&#39;t support non-runtime assemblies such as Sliverlight SDK ones and WCF RIA Services), and I have never tried WCF on MT.<div>
<br></div><div>So, on the question 1...</div><div><br></div><div>The core part that involves dynamic code generation is ChannelFactory&lt;T&gt;.CreateChannel() and anything that calls that method.</div><div><a href="https://github.com/mono/mono/blob/master/mcs/class/System.ServiceModel/System.ServiceModel/ChannelFactory_1.cs#L138">https://github.com/mono/mono/blob/master/mcs/class/System.ServiceModel/System.ServiceModel/ChannelFactory_1.cs#L138</a></div>
<div><br></div><div>ClientBase&lt;T&gt;.CreateChannel() one of such methods that calls the problematic method above.</div><div><a href="https://github.com/mono/mono/blob/master/mcs/class/System.ServiceModel/System.ServiceModel/ClientBase.cs#L274">https://github.com/mono/mono/blob/master/mcs/class/System.ServiceModel/System.ServiceModel/ClientBase.cs#L274</a></div>
<div><br></div><div>Silverlight-based proxies (as well as those from *our* svcutil with -monotouch option) *overrides* this ClientBase&lt;T&gt;.CreateChannel to return the corresponding derived class of ClientBase&lt;T&gt;.ChannelBase&lt;T&gt; (which is also generated in each proxy). Hence it does not result in dynamic code generation.</div>
