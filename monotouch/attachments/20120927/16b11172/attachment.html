Hi Rolf. I have found the bug!<br>
<br>
And here is my adventure on what it is and how to fix it.<br>
<br>
I'm using AdMob library with Alex Desoto implementation, and the protocol<br>
crashes when theAdMob library does it's requests (because that i was getting<br>
random crashes also).<br>
<br>
Double checked the AdMob library is for iOS 6 and recompiled again the .net<br>
dll with no luck.<br>
<br>
After some googling, found that AdMob uses internally a ProtocolHandler to<br>
make it's requests, which gave me a little clue.<br>
<br>
To nail down the problem, first I removed the protocol and everything<br>
started to work and no random crash.<br>
<br>
So I modified my protocol to return always false to CanInitWithRequest, in<br>
theory the query must be propagated until a protocol handles it, so if it<br>
was a problem because my protocol was handling it's request it will work. To<br>
my surprise it crashed too.<br>
<br>
After that I removed the AdMob library, created a mini protocol with only<br>
CanInitWithRequest and registered my real protocol and this new mini<br>
protocol to see if calls where propagated between various protocols, and it<br>
did but a WebBrowser in my app stopped working, so a bit of light was seen.<br>
<br>
So, struggling my brain and a bit desperated went to the assembly inspector<br>
to see how was implemented the class and found a veri "nice" thing.<br>
<br>
CanInitWithRequest must be an static method but is declared as a virtual<br>
instance method. Very nice XD.<br>
<br>
When I wrote the protocol handler, monodevelop detected a<br>
CanInitWithRequest, so it said the "new" keyword must be used. But here is<br>
the trick, no static CanInitWithRequest method exists and don't know why but<br>
that is what was breaking everything.<br>
<br>
I've removed the new keyword and the browser worked again. And finally I<br>
have filtered my protocol to do not handle the admob request and all is<br>
working finally.<br>
<br>
So that's an example on how three letters can destroy a program XD.<br>
<br>
Hope this helps to anyone having the same problems.<br>
<br>
A question, should I fill this as a bug or this is an implementation<br>
decission??<br></blockquote><div><br></div><div>CanInitWithRequest is an instance method, not a static method, so this is just how it's implemented. Technically either 'new' or 'override' should work, as long as the method isn't static (but with 'new' you'll need the Export attribute, with 'override' you won't since the base method already has an Export attribute).</div>
