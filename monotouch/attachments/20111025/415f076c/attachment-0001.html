<blockquote class="gmail_quote" style="margin:0 0 0 .8ex;border-left:1px #ccc solid;padding-left:1ex;">After a too long and painful process, and with a lot of assistance from the<br>
community, I have finally managed to get UITableView to behave more like a<br>
grid that our users require to consume the information in our application.<br>
<br>
Because I did not find a single solution for all of our requirements and<br>
because of a lot of disinformation out there (i.e. UITableView can&#39;t scroll<br>
horizontally without being in a scroll view, you can&#39;t have multiple<br>
columns, etc.) I wanted to share some of the general techniques that we used<br>
to implement this behavior.<br>
<br>
First, our general end-user requirements were to have a multi-column grid<br>
that can scroll both horizontally and vertically with the column headers<br>
always visible. In addition, it was a requirement, due to the large amount<br>
of data that could be presented through these grids, to be as prudent as<br>
possible with resource consumption.<br>
<br>
When a UITableView is embedded in a scroll view, this last item becomes a<br>
major issue because this eliminates the native UITableView&#39;s process of<br>
recycling cells.<br>
<br>
As it turns out, the solutions to all of these issues are relatively<br>
straightforward, but not readily accessible.<br>
<br>
There are plenty of articles that describe how to create multi-column<br>
UITableViews.<br>
<br>
However, none of them addressed cell reuse if you add your own labels to<br>
create multiple columns. To find the labels associated with your columns in<br>
a reused cell in order to change the text of the cell, give each label in a<br>
given cell a unique tag (corresponding to the column number, for example),<br>
then call the ViewWithTag method on the cell to retrieve the correct label<br>
for the column you are working with (only if you are working with a reused<br>
cell).<br>
<br>
In order to create a header that does not scroll, you need to use a custom<br>
view in the UITableViewSource&#39;s GetViewForHeader method. To ensure it does<br>
not scroll, the view needs to have ClipToBounds set to false.<br>
<br>
If you try to use a gradient in the header, you will most likely be<br>
unsuccessful because it is always clipped when scrolled horizontally.<br>
Instead, use a standard background color and it should be displayed<br>
correctly.<br>
<br>
In order to support horizontal scrolling, both the header view and the<br>
individual row cells must have a frame width that is the sum of the widths<br>
of the individual columns.<br>
<br>
You must also set the UITableView&#39;s ContentSize after ReloadData(): the<br>
width should be the width of the view/cell calculated above (or the<br>
ContentSize.Width, whichever is greater) and the height should be the<br>
current ContentSize height. This will support both horizontal and vertical<br>
scrolling.<br>
<br>
Finally, when you scroll the new UITableView horizontally, you will most<br>
likely notice that the row separator lines are clipped. I was able to<br>
resolve this by creating a custom cell that inherits from UITableViewCell,<br>
overriding the Draw method and, before calling the base implementation of<br>
Draw, setting the frame size width to the sum of the column widths. This<br>
custom cell is then used in place of the standard UITableViewCell in the<br>
UITableViewSource GetCell override method.<br>
<br>
Hopefully these concepts will help someone with similar issues in the<br>
future.<br>
