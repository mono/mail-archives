As with anything, &quot;it depends&quot;.<div><br></div><div>File I/O should be as fast as buffered native I/O (I did a fair bit of testing of this for Mono a few years ago).</div><div><br></div><div>Most of what apps do is spent idling waiting for user events and idle time is idle time and using the MonoTouch APIs is (for the most part) just a thin wrapper around the iOS Touch APIs so there is very little overhead at all there.<div>
<br></div><div>Of course, if you do some intensive inner-loop processing, depending on what type of processing you are doing, you might be better off calling into a native library, but, then again, you might be able to get equivalent performance by simply refactoring that loop and/or using &#39;unsafe&#39; code. Since C#/Mono make it trivial to call out to native code later once you&#39;ve done some performance testing and found the hotspot to be in said loop, it&#39;s not really an issue (if you had to pick C# or C and couldn&#39;t use both, *that* would be a cause for concern).</div>
<div><br></div><div>Since image processing is largely CPU intensive (and normally done using SIMD - which, sadly, MonoTouch does not support (maybe a feature for a future version?)), this might be a task best suited for a native library or something. Then again, maybe Apple&#39;s CoreImage does what you need, in which case MonoTouch&#39;s binding should already have what you need as well (and there should be no significant overhead).</div>
