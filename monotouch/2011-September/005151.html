Hi

2.8 and 2.6 don't really have much differences in the code completion 
layer (future versions will have that, but I've made unit tests for 
every reported code completion error in the last years and we won't 
release the new layer until all these tests are green).

The current 2.8 requires a bit longer to generate the code completion 
database. You see that in the lower right corner. The zip file for 
monotouch documents did grow too large therefore we cache that now. But 
after that code completion should be faster than before - because the 
zip file operation doesn't occur anymore when hitting a key.

If the gathering of the class infos doesn't stop it's likely that there 
is a corrupt zip file floating around which takes ages for uncrunching - 
y the #ziplib is more or less a good weather approach that works best 
with uncorrupted zip files.

We learn which each release and we'll look in future versions for the 
zip file integridy.

Thanks for your patience
