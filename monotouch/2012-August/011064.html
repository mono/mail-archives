On Thu, Aug 2, 2012 at 2:46 PM, René Ruppert <rene.ruppert at gmail.com> wrote:
> I would like to comment on this one too.
> Sebastien, I remember that we had a similar discussion on SO. You said that I simply should not call Dispose() manually as one can never be sure that the object won't be required any longer and GC will take care of it once there are no more references. And indeed: remove the Dispose() in the project attached to the bug report prevents the crash.

Yes. The general issue is that the managed object is freed too soon
(since it's still needed by some native code).

Setting a field to null will, at some point in the future, allow the
GC to finalize it. It can take enough time (for the GC) to allow the
native side to complete it's job (so things might work most of the
time). OTOH doing a Dispose will immediately free the the instance -
so it's much more likely to cause this issue (if a native object still
holds a reference to it).

> BUT: In my project I removed all calls to Dispose(). Nevertheless I see the described errors sometimes. It looks like at some point, GC decides to collect the controllers even though they weren't disposed explicitly.

Yes, the GC can dispose of any managed objects when there's no more
(managed) references to it (but it cannot know if there's native
references).

> My conclusion is: I use Dispose() only as part of the using() pattern, I never call it manually.

It does not really matter, i.e. you can call manually Dispose instead
of `using` without any difference. It's just that `using` is generally
used on objects that are "short" lived (e.g. locals) and the
life-cycle is simpler (e.g. a FileStream). Things like `UIView` are
often kept in fields and their life-cycle can extend outside the
(managed) code you wrote.

We're looking at better solution to debug this and, based on this, we
should be able to propose some patterns to help (to avoid and/or fix
them).
