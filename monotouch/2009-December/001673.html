NSHTTPCookieStorage is not listed in the monotouch docs but is it used indirectly via NSMutableUrlRequest without explicit reference?
This question relates to a design dilemma I am having choosing between reusing some existing HttpWebRequest .net code or using equivalent Apple sdk classes. The specific problem is that my iPhone app will need to establish an authenticated session with a web service and then re-present a session cookie. Unique to the iPhone is that a user may typically pop in and out of the App within a server side web session timeout period. So to avoid server side session bleed I need to obtain the session cookie from iPhone storage on app startup.
NSHTTPCookieStorage handles this automatically but at the same time it seems to introduce a security risk because I suspect my app's session cookie could be read by a rogue app. If I reuse my HttpWebRequest code I will have to raid the dotnet CookieContainer object and store the current session cookie on app close down.
Basically I am in a design spin here and thought someone else would have been through this and arrived at a solution?
So in summary:Do folks agree NSHTTPCookieStorage is a security risk for session cookies?Has anyone here handled an https dialogue via NSMutableUrlRequest setting a session cookie manually?Will my release binary package get much larger if I use .Net HttpWebRequest and the .Net Cookie Container?Will the .Net HttpWebRequest class actually route http requests into NSMutableUrlRequest and NSHTTPCookieStorage regardless?
