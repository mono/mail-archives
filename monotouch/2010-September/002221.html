I think you have to look at it from an opportunity cost perspective not a
TCO.

I.e. an existing App store developer would already have spent all those
costs to be able to develop an app with objective-c, the choice that
prospective MonoTouch developers have to make is to evaluate whether or not
the additional cost are worth the benefits.

Personally if you're app is doing a lot of UI functionality I actually
prefer the well proven and documented path of xcode/interface builder. Using
MonoTouch/MonoDevelop is not a total productivity win as some time is lost
translating Obj-C apis to MT apis, searching documentation to find out how
some things are achieved in MT, working around any bugs, etc. Also I don't
know how far you can go without learning any Obj-c as I think at least being
able to read it is important if you're considering any iPhone dev.

However if you are a C# developer and have the ability to re-use your
existing libraries or have to do something that is notoriously difficult to
do in Obj-C (i.e. calling web services, data traversing or manipulation,
etc) then the ROI of your MonoTouch licence will be recuperated pretty
quickly provided your app is sufficiently popular.

Your mileage may very but in general the C# language and Base class library
are much more enjoyable to work with then the equivalent Obj-c/Cocoa
apis. The great thing is that you can download and develop with MonoTouch
for free so you can gauge the value of the product for yourself. Also now
that Apple has lifted all 3rd party dev tools restrictions I'm much more
confident in the continued use and improvement of the technology.

- Demis
