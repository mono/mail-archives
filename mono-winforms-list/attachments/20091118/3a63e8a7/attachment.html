Hey,<br><br>There are some limitations on our implementation of Windows.Forms, just as any other re-implementation can have. We provide it however, to let developers to bring their applications from .net to linux with Mono.<br>
<br>Now: if you are going to create a new app, then you can go with Gtk#, and that&#39;s a different story, with its own advantages/disadvantages.<br><br>Carlos.<br><br><div class="gmail_quote">2009/11/18 StApostol <span dir="ltr">&lt;<a href="mailto:stapostol@gmail.com">stapostol@gmail.com</a>&gt;</span><br>
<blockquote class="gmail_quote" style="border-left: 1px solid rgb(204, 204, 204); margin: 0pt 0pt 0pt 0.8ex; padding-left: 1ex;"><div>Frankly, the best way to create a cross-platform GUI is to use GTK# from the beginning. With WinForms, you&#39;ll encounter several issues that will make things rather difficult down the road: flaky databinding, slow rendering (especially for grid views), inconsistent rendering between versions, inconsistent ordering of events (quite, eh, fun to debug), lack of theming.<br>

</div><div><br></div><div>By far the best solution is to abstract the GUI away from the main application and use different technologies for different platforms (e.g. WPF on Windows, GTK# on Linux and Mac OS X). However, this takes a lot of resources and planning to implement correctly.</div>

<div><br></div><div>The second best option is to use GTK# all the way. You&#39;ll trade some of the &quot;native&quot; look and feel for easier development (see e.g. MonoDevelop on Windows).</div><div><br></div><div>The third option is to use WinForms everywhere, keeping in mind the caveats I mentioned. WinForms is not meant for cross-platform development - it has no native look and feel (unless you create custom widgets for everything, which is certainly an option) and it is painful even on Windows (try using native Vista fonts, for example or creating a DPI-independent GUI - that&#39;s right, you can&#39;t really do that). Finally, it&#39;s a pretty much dead technology that hasn&#39;t been updated in the last 5 years. (Yes, there are still good reasons for using it, as long as you keep the drawbacks in mind).</div>

<div><br></div><div>Finally, it might be worth checking out Silverlight/Moonlight. These have a pretty nice development environment and can be used both online and offline (with a little work).</div><div><br></div><div>My 2cc :)</div>
