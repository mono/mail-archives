> Hi All,

> Sorry for having been so quiet. I was hoping I'd get all the changes=20

>
> The first patch is in CVS already, it's located under=20
> mcs/class/System.Windows.Forms/wine-patch
>
> I've also attached the patch and a readme.
>
> Keep in mind that it's not complete yet. The patches will allow you to =

> use Wine as a shared library. Unlike the previous patch, where wine=20
> itself was hacked up to be compiled either the traditional way or as a =

> shared library I tried to have a patch that could make it back to the=20
> main wine repository. So the only actual changes to wine itself was=20
> that I prevent wine from calling the WinMain of a passed-in winelib=20
> executable, if the name of the executable is 'mono-winelib'.
>
> Everything else is handled in the mono-winelib executable. It acts as=20
> a winelib binary as well as the library that gets loaded to initialize =

> wine as a shared library. Simply dlopen() and import the=20
> MonoWineInit() function and run it. That function in turn will=20
> initialize wine and then return. At that point any caller should be=20
> able to import and use any symbols/functions from wine.
>
> Keep in mind that I have done only minimal testing with the approach=20
> so far, since I still need to develop a second patch, to make the=20
> x11drv coexist with cairo. It seems that there's some offscreen=20
> caching happening in cairo and when wine draws on the screen cairo=20
> isn't aware and 'overwrites' the wine changes. I'm still in the=20
> process of debugging the issues, give me a few days more on that.
>
> Check the readme that goes along with the patch for some symbolic=20
> links that have to be created manually currently before the wine patch =

> can be used.
>
> I have also attached a patch that's not in CVS yet (mcs-0.30.patch)=20
> which fixes up SWF to properly load the new wine library. Pathnames=20
> are currently fixed to /usr/local/lib/wine. Once the hard problems are =

> worked out I'll take care of these things.
>
> Hope this wasn't too much detail. Please send me any feedback you may=20
> have, but keep in mind that SWF (while it may start displaying windows =

> again) will still not work properly due to the cairo/x11drv issue (and =

> potentially some of the new work done in System.Drawing)
>
> I'm also working on a roadmap for SWF itself, so that we can=20
> coordinate work on the library, but I figure the basic framework=20
> (Wine/Cairo/SWF) needs to be in working order first.
>
>
> Cheers,
>  Peter

After applying the patch to wine, run ./configure in the wine directory and 
'make &amp;&amp; make depend &amp;&amp; make install'.
For now the patch assumes that the prefix_dir is /usr/local.

all the dlls you plan on using in System.Windows.Forms:
ln -s /usr/local/lib/wine/kernel32.dll.so /usr/local/lib/libkernel32.dll.so
ln -s /usr/local/lib/wine/user32.dll.so /usr/local/lib/libuser32.dll.so

ln -s /usr/local/lib/wine/advapi32.dll.so /usr/local/lib/libgdi32.dll.so
