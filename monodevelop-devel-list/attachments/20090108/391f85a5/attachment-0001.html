Having used Resharper on a previous job, I like that they have it all on the first-level context menu, but vary the list according to where you are in the source structure. Taking your example, if you are in a field you can rename it but not the class it is contained, but you can do some actions on the class like moving it to another namespace or source-file, I think the key word is to have commands that are&nbsp; &quot;unequivocally&quot; associated with the current context OR (exclusive) its containers. If I want to rename a class, I should position the cursor on its name or, maybe, in first-level whitespace in it: a blank line inside the class definition block, but not inside a member&#39;s definition block.<br>

I recently implemented &quot;Go to definition&quot; and &quot;Find references&quot;<br>
commands as top-level items in the text editor context menu. This<br>
allows them to be bound to keyboard shortcuts, F12 and Shift-F12.<br>
However, I haven&#39;t committed them yet as I have a few concerns about<br>
how it relates to the existing refactoring/navigation context menus.<br>
<br>
Currently, MD dynamically generates submenus for the symbol under the<br>
cursor, and its enclosing/parent items, i.e. the enclosing member, the<br>
enclosing class, and (if it&#39;s a variable) the type. Each of these<br>
menus allows refactoring and navigation operations on the associated<br>
item.<br>
<br>
There are a few problems with this. Since the submenus are dynamically<br>
generated, user extensions cannot add new refactoring operations to<br>
the menu, nor can extensions override the handling of these commands.<br>
Also, the commands cannot be keybound -- and even if they could, it<br>
would not be obvious which commands the shortcuts would apply to,<br>
since the menus contain identical commands.<br>
<br>
So, my question is: are all of the submenus really necessary? How<br>
useful is it to be able to implement members on a class from the<br>
context menu on a field within it? What I&#39;d like to do is replace all<br>
these menus with a simple refactor/navigate menu that would apply only<br>
to the item under the cursor, as this would be keybindable and<br>
extensible. However, would it be noticeably less usable or functional?<br>
