<p class="MsoNormal"><o:p> </o:p></p>
<p class="MsoNormal">The safety of stackalloc and span<T> will mostly just fall out from existing concept in the language.  <o:p></o:p></p>
<p class="MsoNormal"><o:p> </o:p></p>
<p class="MsoNormal">Although span<T> is frequently referred to as stack only it’s actually a ref-like type.  Essentially a type which has the same set of restrictions as ref parameters or locals (one of which is stack only).  Generally though the distinction
 between stack only and ref-like is unimportant.  This is one case though where it’s important and advantageous. 
<o:p></o:p></p>
<p class="MsoNormal"><o:p> </o:p></p>
<p class="MsoNormal">In C# 7.0 we introduced the concept of ref return and ref locals / parameters.  As a part of this work the language (and verifier) had to add rules around what values are safe to return and the valid lifetime of ref locals.  This takes
 into account values which are known to be on the heap, and values which are known to be stack bound. 
<o:p></o:p></p>
<p class="MsoNormal"><o:p> </o:p></p>
<p class="MsoNormal">Given that span<T> is a ref-like type it will inherit all of the same rules and restrictions.  When combined with stack alloc it allows for us to safely model stack allocating memory as a span<T> value.  There will need to be some tweaks
 to allow stackalloc in safe code but that’s pretty small and overall the change is just playing into rules we have already setup.  <o:p></o:p></p>
<p class="MsoNormal"><o:p> </o:p></p>
<p class="MsoNormal">As to why span<T> is a ref-like …  At its core span<T> is conceptually a struct with two fields:<o:p></o:p></p>
<p class="MsoNormal"><o:p> </o:p></p>
<p class="MsoListParagraph" style="text-indent:-.25in;mso-list:l0 level1 lfo2"><![if !supportLists]><span style="mso-list:Ignore">-<span style="font:7.0pt "Times New Roman"">         
</span></span><![endif]>ref T data: a pointer to the first element of the span<o:p></o:p></p>
<p class="MsoListParagraph" style="text-indent:-.25in;mso-list:l0 level1 lfo2"><![if !supportLists]><span style="mso-list:Ignore">-<span style="font:7.0pt "Times New Roman"">         
</span></span><![endif]>int length: the count of accessible members. <o:p></o:p></p>
<p class="MsoListParagraph"><o:p> </o:p></p>
<p class="MsoNormal">The actual implementation of span<T> tends to use other mechanisms though: IntPtr, intrinsics, etc …  This is due to a lot of details, in particular the CLR not supporting ref fields directly.  But from the perspective of the language and
 safety we need to model this type as being a struct with a ref field for correctness irrespective of how a particular runtime implements it.
<o:p></o:p></p>

<p class="MsoNormal"><o:p> </o:p></p>
<p class="MsoNormal">Assuming that safe Span<T> stackalloc exists, the verifier (and compiler) must track Spans that aren’t allowed to be returned. That also enables adding a constructor to Span that takes a ref T (resulting in a span to a single item). Combined
 with a safe cast this can be used to safely make fixed fields accessible as a span.<o:p></o:p></p>
<p class="MsoNormal"><o:p> </o:p></p>
<p class="MsoNormal">(It would also be nice If the needless C# restriction that fixed fields are unsafe is removed.)<o:p></o:p></p>
