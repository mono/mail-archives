<p class="p1"><span class="s1">I have written a description of my prototype implementation of adaptive ASCII/UTF-16 strings in Mono:</span></p>
<p class="p2"><span class="s1"></span><br>
</p>
<p class="p3"><span class="s2"><a href="http://www.mono-project.com/docs/advanced/runtime/docs/ascii-strings/" id="LPlnk834666">http://www.mono-project.com/docs/advanced/runtime/docs/ascii-strings/</a></span></p>
<p class="p2"><span class="s1"></span><br>
</p>
<p class="p2">Introduction:</p>
<p class="p2"><br>
</p>
<p class="p1"><span class="s1">> For historical reasons, System.String uses the UCS-2 character encoding, that is, UTF-16 without surrogate pairs.</span></p>
<p class="p1"><span class="s1"><br>
</span></p>
<p class="p1"><span class="s1">> However, most strings in typical .NET applications consist solely of ASCII characters, leading to wasted space: half of the bytes in a string are likely to be null bytes!</span></p>
<p class="p1"><span class="s1"><br>
</span></p>
<p class="p1"><span class="s1">> Since strings are immutable, we can scan the character data when the string is constructed, then dynamically select an encoding, thereby saving 50% of string memory in most cases.</span></p>
<p class="p2"><span class="s1"></span><br>
</p>
<p class="p1"><span class="s1">I would like to solicit feedback on this proposal from runtime developers and users alike. In particular:</span></p>
<p class="p2"><span class="s1"></span><br>
</p>
<p class="p1"><span class="s1">- Specific objections regarding performance characteristics, compatibility issues, &c.</span></p>
<p class="p1"><span class="s1">- Questions about unclear or underspecified parts of the proposal</span></p>
<p class="p1"><span class="s1">- Real-world use cases that would benefit from this optimization</span></p>
<p class="p1"><span class="s1">- Suggestions for suitable real-world benchmarks</span></p>
