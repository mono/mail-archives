The first, ObjectDisposedException (in comment #14), is normal. It generally
occurs when a POST is done and the user close the stream. Right now
InternalWebRequestStreamWrapper catch the exception (it's not propagated to the
application code) and deal with it differently. 

An exception-less code path will be added later along with other SL3 specific
features (like a real client HTTP stack) that would currently break our
SL2-based tests.
