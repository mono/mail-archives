When using class DataContractSerializer to serialize a CLR object in Moonlight,
the default order of xml nodes in then result document is different from
Silverlight and .NET framework, causing problems while
serializing/deserializing objects between enviroments (in my own scenario,
Moonlight and a WCF service). 

The problems seems to be related with subclassing. Think about two classes: A
and B, B extends A, both have public properties decorated with [DataMember]. If
I try to serialize an instance of B with DataContractSerializer, the following
behavior arises:

Silverlight/.NET framework - All data members of A are serialized first in
alphabetical order, followed by data members of B;

Moonlight - The entire pool of data members from A and B are serialized in
alphabetical order, which means that data members from A and B can be mixed in
the result document.

I have followed a workaround to get my app to work, which was setting the
serialization order explicitally in all data members of both classes:

[DataMember(Order = n)]

Obviously this is not a good solution when you have hundreds of classes to
serialize and a complex class extension graph, so I think it should be fixed.

I think this is pretty clear situation. I'm able to provide further details if
you need.
