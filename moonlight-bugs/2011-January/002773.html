Otto, that stackoverflow discussion is confusing SDK and platform assemblies.

> System.Xml.Linq, System.Windows, System.ComponentModel.Composition, System.Runtime.Serialization and System.Windows.Browser

SDK: System.Xml.Linq 
SDK: System.ComponentModel.Composition
Platform: System.Windows
Platform: System.Runtime.Serialization
Platform: System.Windows.Browser

Now, like the discussion said, SDK assemblies must be part of the XAP (or an
ExternalPart) otherwise your application won't work.

However I'm not 100% sure what will happen if you add platform assemblies in
your XAP. SL will either :

(a) use the one that shipped with the plugin - in this case your XAP is bigger
than it has to be (more time to download, more memory to decompress...)

(b) try to use the one you provided - then you'll likely have weird problems
when SL5 is released because the internals (including [InternalsVisibleTo]
between platform assemblies) will be different.

Anyway in both cases there's no gain in having platform code inside your XAP
(only minor and/or future problem ;-)
