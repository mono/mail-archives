--- Comment #1 from Rui Craveiro <rui at ruicraveiro.com> 2011-01-24 15:22:07 UTC ---
In the test-case I attached, by mistake I left outcommented a possible CLR to
represent the dependency property that is the problem. After submitting I
noticed the CLR property was incomplete, as it had default getter and setter.

A better example would be:

        //// if you uncomment the following line, it will also work in ML
        //public string MyDemoProp { 
        //    get 
        //    {
        //        return GetValue(MyDemoPropProperty) as string;
        //    }
        //    set
        //    {
        //        SetValue(MyDemoPropProperty, value);
        //    }
        //}

Nevertheless, for the purpose of the demo, the behaviour remains the same. With
the CLR property (either the incomplete one I submitted first or the version I
am presenting in this comment), there is no exception. Without the CLR property
you will have an exception with the managed parser and none in Silverlight or
with the unmanaged parser.
