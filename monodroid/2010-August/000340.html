The interface in java.io was Serializable, NOT ISerializable.

<opinion>
It was never a Java standard to "I"-prefix their interfaces. I think it
would actually behoove Mono to keep the old names in place, just to avoid
these conflicts. It'll also help avoid weird name-resolution conflicts in
peoples' heads as they read Java-based Android docs/articles (which I would
expect will happen from time to time) when working with MonoDroid.
</opinion>

> Unfortunately, name collisions are going to have to be a fact of life:
> 
>   - Java.Lang.Object conflicts with System.Object
>   - Java.IO.ISerializable conflicts with System.ISerializable
>   - Java.Util.IList conflicts with System.Collections.IList
>   - Java.Util.IList<T> conflicts with
>     System.Collections.Generic.IList<T>
>   - etc., etc., etc.
> 
> It's particularly bad for the collections (ICollection also conflicts,
etc.).
> 
> There were three ways we could go:
> 
>   - Keep the original Java names, Pascal-Cased.  This is not without
>     conflict (Java.Util.List conflicts with
>     System.Collections.Generic.List), and isn't consistent with .NET
>     naming conventions.
>   - Prefix Java interfaces with 'I'.  This is what we did for
>     consistency with .NET naming conventions and familiarity.
>   - Use some other prefix.  We couldn't quickly agree on what
>     "some other prefix" would be; 'J' is already a prefix used in
>     Java Swing, 'IJ' is just getting silly, etc., and we still
>     have class names that collide (Java.Util.ArrayList,
>     Java.Lang.Object, etc.).
> 
> Unless/until someone thinks of a better way to avoid the type name
> collisions, we'll just have to get  used to fully qualifying the Java
types.  In
> short, it's probably a Bad Idea to have 'using Java.Lang;' in your source
code...
> 
>  - Jon
