> I have to agree with Jonathan. If I'm going to use MonoDroid to
> develop an App for Android, the main reason I'm going to is to keep my
> knowledge and familiarity with the C# language and .Net conventions.

> tools that Google provides. The main benefit for using MonoDroid is I

>     On Mon, 2010-08-23 at 13:48 +0100, Matthew Baxter-Reynolds wrote:
>     > I wonder with this whether you turn it on its head.  The most likely
>     > scenario is that you have a developer who wants one code base that
>     > will compile in MonoTouch for iOS and MonoDroid for Android - and to
>     > an extent natively using MS' own tools for WP7.
>
>     As I believe we've mentioned elsewhere, this is currently impossible
>     (and will likely remain the case).  You'll be able to share *some*
>     code
>     between iOS/Android/WP7 (hopefully lots, though that depends upon the
>     design of your app), but not all.  In particular, the GUI code will be
>     using different types depending on whether you're using MonoTouch
>     (MonoTouch.UIKit), MonoDroid (Android.Views), or WP7 (System.Windows).
>     Unless/until Novell writes Moonlight support for MonoTouch and
>     MonoDroid
>     (which may never happen), there will be no cross-platform UI toolkit.
>
>     Furthermore, I'm not convinced that such a thing is useful; see:
>
>     http://lists.ximian.com/pipermail/mono-list/2009-November/043780.html
>     http://lists.ximian.com/pipermail/mono-devel-list/2010-February/034207.html
>
>     An elaboration on the above is that Android provides a hardware Menu
>     button, which neither iOS nor WP7 provide.  I haven't looked, but I
>     doubt that there'd be a sane way to hook the Silverlight API up to the
>     Menu button, and the probable result would be a WP7-looking app on
>     Android...which (as mentioned previously, repeatedly) would look
>     foreign
>     and user hostile to most actual users.
>
>     > so long as you can support something like this:
>     >
>     > using JU = java.util;
>     > using System.Collections;
>     >
>     > ArrayList foo = new ArrayList();
>     > JU.ArrayList bar = new JU.ArrayList();
>
>     That's a feature of the compiler, not the class library, and is
>     consequently very well supported. ;-)
