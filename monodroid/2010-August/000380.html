Ted,
   To your argument about refactoring an interface to/from an abstract
class, as long as the abstract class is purely abstract ( no implementations
) then in a sense it's an interface and can keep the "I" prefix. it only
become tricky when you introduce a layer of implementation through a common
base class. Refactoring an interface to a concrete class has the same
problems in both C# and Java regardless of the naming. In C# you typically
would see a base class introduced with a name that leaves off the "I"
prefix, in Java, the suffix "Base" is added to the interface name.

But this isn't just about naming, it's about convention. And this convention
isn't something that Mono started, it was started by Microsoft. My personal
opinion is that MonoDroid is 1) about bringing .Net developers to the
Android platform, 2) bringing Android developers to .Net/Mono. Keeping the
naming conventions that have been set forth by Microsoft will make the
transition for most .Net/Mono developers easier than if you give them the
raw Android API's, But your right, in that the Android ->  Mono set will
have more problems.
