On Mon, Aug 23, 2010 at 12:28 PM, Jonathan Pryor <jpryor at novell.com> wrote:

> On Sun, 2010-08-22 at 23:33 -0700, Ted Neward wrote:
> > <opinion>
> > It was never a Java standard to "I"-prefix their interfaces.
>
> No, it's a .NET standard, and MonoDroid is trying to make things
> consistent for .NET developers.  Hence I-prefixed interface names, the
> generation of events instead of interface-based callbacks, etc.
>
> (The use of events is a real killer, as C#/etc. lack anonymous inner
> classes, which is the only sane way to make Java's interface-based event
> handling system sane.)
>
> Plus, perhaps I'm alone, but I had a visceral "that's weird" feeling
> whenever I did:
>
>        class MyActivity : Activity, View.OnTouchListener {
>                // ...
>        }
>
> It just looks very "weird" to me to have multiple "base classes" with no
> I-prefixed types.  It looks...wrong.
>
> >  I think it
> > would actually behoove Mono to keep the old names in place, just to avoid
> > these conflicts.
>
> As my original name stated, keeping the original names would NOT avoid
> conflicts.  There are lots of them.
>
>  - Java.Lang.Object vs. System.Object
>  - Java.Lang.Exception vs. System.Exception (and lots of
>    similar/identical Exception subclasses)
>  - Java.Util.List<T> vs. System.Collections.Generic.List<T>
>  - Java.Util.ArrayList vs. System.Collections.ArrayList
>  - etc., etc.
>
> There is NO WAY to avoid name conflicts unless we prefix every Java type
> name with some "unique" prefix...which looks ugly.
>
> > It'll also help avoid weird name-resolution conflicts in
> > peoples' heads as they read Java-based Android docs/articles (which I
> would
> > expect will happen from time to time) when working with MonoDroid.
>
> One of my doc projects is to embed links into the Android documentation
> from docs.monodroid.net, though that's only MonoDroid->Android, not the
> other way around.
>
> That said, we already know that there will be more fundamental
> differences in the binding than just I-prefixed interface names (like
> the aforementioned generation of events for *Listener interfaces), and
> many of these will be to make MonoDroid *usable* to C# devs (e.g. events
> to make up for the lack of anonymous inner class support).  We might be
> willing to undo the I-prefixed interfaces, but other things will be
> unavoidable changes unless you like the "Writing Java in C# but worse!"
> development mentality.
>
>  - Jon
