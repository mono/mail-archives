On Mon, 2010-08-23 at 09:09 +0100, Wayne Douglas wrote:
> I've asked about this on the android dev list - i cant believe they
> made this xml format with no way of validating it!

They can't, not really.  It's like XAML [0]. See the "Hello GLCube"
tutorial:

        http://monodroid.net/Tutorials/Hello_GLCube

The Resources/layout/main.xml file contains an XML element which is
interpreted as the Java class name to use:


    <mono.samples.gLCube.PaintingView
      android:id="@+id/paintingview"
      android:layout_width="fill_parent"
      android:layout_height="fill_parent"/>

Now, some attributes will be pre-defined, but not all of them (the class
can specify it's own attributes), and afaik there's no way to query the
class to see what attributes and sub-elements are valid.

Furthermore, since ~any View (and likely others) can be used as an
"element", it's non-trivial to provide code completion for "all"
elements that can be used in a given spot (though a subset can certainly
be used).

 - Jon

[0] OK, I'm sure XAML has some degree of IDE validation, but only
because the IDE can use reflection to see what properties are exposed,
what their types are, what values are valid, etc., etc.  While Java has
java.lang.reflect, attributes aren't handled via properties (as XAML
does), but instead by invoking a constructor accepting an AttributeSet
parameter, where the AttributeSet contains the attributes as attributes
defined in the XML.  See e.g.

http://developer.android.com/reference/android/view/View.html#View(android.content.Context, android.util.AttributeSet)
http://developer.android.com/reference/android/util/AttributeSet.html

Furthermore, I see no methods on e.g. android.view.View that would allow
an IDE to ask the type "which attributes do you consider valid?", so
this is another place where .NET making properties a distinct language
construct creates a superior developer experience. ;-)
