Jepp, agree. UI Creation and Databinding is the main Issue when a developer
starts using monodroid. But what's interesting too is synchronisation and
storage of Data. Let's say you're Client is talking to a Webservice and asks
for a Customer-Database. Where should I store the retrieved Data for times
without being Online and how can I synchronize the changes? Is there a way
to realize a change Notification for keeping the data-traffic small?

> I think the big issue is the mindset going into developing with Monodroid
> that you expect a System.Windows.Forms control. You instead are thrown at
> the layouts and resource files for everything. While understandably, it is
> android. So maybe a little bit of, here is what you would code for on the
> desktop, here are the steps to translate that into a monodroid application.
> Then maybe a few different scenarios and ways to do things in monodroid that
> you have to do different.
>
> Another nice piece would be battery issues. How should you code for a long
> running application, what should you do, what should you not do. If
> possible, maybe base it on an existing application that all android users
> find useful. Something that puts the icon in the notification bar, etc.
>
> The last one might be performance, how you should code, or techniques
> to squeeze out performance on a mobile device. Not like a chapter just
> explaining the stringbuilder is the best to use, but things best way to
> redraw (gaming specific), memory considerations, slow system calls.
>
> Just my two cents
