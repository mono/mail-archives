OK, maybe you can (see below), but I haven't tested this, so I'd certainly
love someone to work out the details.  But first...

Why won't it work?

It doesn't work because MonoDroid only binds and exposes android.jar, and
com.google.android.maps is obviously not part of android.jar.  (If it were,
you wouldn't need <uses-library/>.)

We fully intend to support binding and exposing arbitrary .jar's, _after_
MonoDroid 1.0.

In the mean time, there is (hopefully) a way to make this work, despite lack
of native MonoDroid support, which boils down to using JNI:

	
http://docs.monodroid.net/monodoc.ashx?link=T:Android.Runtime.JNIEnv/*

Sadly, no docs there, but they pretty much mirror the actual JNI functions
except that the JNIEnv* pointer doesn't need to be passed:

	
http://download.oracle.com/javase/1.5.0/docs/guide/jni/spec/functions.html

For example, instead of:

	jclass k = (*env)->FindClass(env, "java/lang/String");

you'd do:

	IntPtr k = JNIEnv.FindClass ("java/lang/String");

Coupled with `monodroid -js`/the AndroidJavaSource Build Action, and you
should be able to either use JNIEnv to ~directly create and invoke the
desired Java types, or (if necessary) write Java code which subclasses the
types in the .jar file and then use JNIEnv to create and invoke your custom
Java types.

That's the theory; the practice is that I haven't tested this, so I don't
know if it will _actually_ work.  Specifically, I haven't looked up how the
custom .jar is bundled into the .apk (assuming it is); I've only see
<uses-library/>, and I haven't yet needed to look up the rest of the
details.

If someone with some free time would like to explore this and post their
findings, I'd be eternally grateful. :-)
