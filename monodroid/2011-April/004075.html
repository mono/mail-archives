Some spelunking in the Android sources, and I have a plausible explanation: leaky abstractions. :-)

Whereas mono+sgen will indiscriminately halt every registered thread in the process -- even if executing "native" code -- in order to perform a GC, the Dalvik GC will not. As a result, at some point Dalvik wants to perform a full GC, and thus wants to freeze every thread registered with Dalvik -- including the Mono ThreadPool thread (because it called RunOnUiThread() -- and Dalvik requires that the threads stop themselves.

You have a long-running operation, which "never" returns to Dalvik, and thus can't be frozen so that Dalvik can perform a GC. :-)

In short, there is nothing Mono can do here: you can't have long running threads that Dalvik knows about that never yield to Dalvik.

A workaround is to split the long-running operation into smaller ones:

        protected override void OnCreate(Bundle bundle) 
        { 
            base.OnCreate(bundle); 

            textview = new TextView(this); 
            textview.Text = "Working.."; 
            SetContentView(textview); 
            ThreadPool.QueueUserWorkItem(o => MemUsage(0, 1000000));
        } 

        void MemUsage(int start, int end) 
        { 
            List<string> l = new List<string>(); 
           int max = Math.Min (start + 1000, end);

            // Simulating Heavy Memory Usage 
            for (int i = start; i < max; i++) 
            { 
                string h = "Hello"; 
                string w = "World"; 

                int a = i + 2; 

                l.Add(h + w + a); 

                Log.Info("BUGTEST", i.ToString()); 

                if (i % 100 == 0)
                    RunOnUiThread(() => { 
                        string log = textview.Text; 
	
                        if (testCase == 1) 
                        { 
                            if (log.Length > 1000) 
                            { 
                                log = log.Substring(0, 999); 
                            } 
                        } 

                        textview.Text = h + w + a + System.Environment.NewLine + log; 
                    }); 
            } 
            if (start + 1000 < end)
                ThreadPool.QueueUserWorkItem(o => MemUsage (start+1000, end));
        }

(You would _think_ that calling into Java -- which is done via RunOnUiThread()! -- would be sufficient for Dalvik to hijack the thread...but apparently not. I declare a Dalvik bug.)

WIthout this change, it would stop processing by ~15000 iterations. With this change, it goes to ~20000+ before the process exits "cleanly":
