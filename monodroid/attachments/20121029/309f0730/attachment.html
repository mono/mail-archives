Hi all,<div><br></div><div>I'm having this dilemma on Activity/Service "binding" that I don't quite know what is the best way to achieve.</div><div>Here's a common scenario...</div><div><br></div><div>
We have an activity named <i>LoginActivity</i>. This activity has two fields, username and password.</div><div>The authentication operation consists in making a web request, and processing the result. Visually, we want to display a nice progress bar while we wait.</div>
<div><br></div><div>That's it. Pretty common.</div><div>Obviously, we want to consider things like the activity getting paused or the screen orientation changing.</div><div><br></div><div>After reading the docs and going through a few forums, I came up with a solution.</div>
<div><br></div><div>First, the dialog issue. If the screen orientation changes, the view is destroyed and recreated, so the dialog just disappears (and worse, it's lost in space - leaks).</div><div>What I did was to keep a reference to the progress dialog shown. I retain a flag stating that the progress dialog is visible by overriding the <i>OnRetainNonConfigurationInstance</i>. I also make sure I dismiss the dialog when the activity gets trashed (destroyed). When the new view gets created, I recover the flag and show a new progress dialog. This works well.</div>
<div>There were other methods, like <i>configChanges</i>, but I didn't want to handle the view resources myself, so this seemed like a good and clean approach.</div><div><br></div><div>Now for the operation, I created a service to handle it. More specifically an <i>IntentService</i>. The request is done in background, and when done, I send a broadcast with the result. Because the activity might pause (or trashed), I send a sticky intent broadcast, to make sure it reaches its destination. When received, I clear up the sticky intent.</div>
<div><br></div><div>This works quite well...</div><div>Though I was happy with the result, I questioned if this was the best approach. Some defend that using sticky intents is evil, even though I do a very assertive use. Some say that service binding is a better approach, though I don't know if it can be used with an IntentService and it seems like an awful lot of trouble for such a simple task (maybe I'm just not looking correctly).</div>
<div><br></div><div>So, any thoughts on this approach and on best practices for accomplishing this are greatly appreciated.</div><div>Cheers.</div><div><br clear="all"><div><br></div>-- <br>Gonçalo Oliveira<br>
