<DIV>Wouldn't it be an option to wrap the Text property value type&nbsp;with a special subclass that overrides ToString to make this work "better"?</DIV>

<DIV><BR><BR>&gt;&gt;&gt; Jonathan Pryor &lt;jpryor@novell.com&gt; 01.02.2011 16:16 &gt;&gt;&gt;<BR>The reason TextView.Text is an IEnumerable&lt;char&gt; is because android.widget.TextView.getText() [0] is a java.lang.CharSequence (an interface type), and android.widget.EditText.getText() [1] overrides TextView.getText() while changing the return type from CharSequence to android.text.Editable [2].<BR><BR>(C# doesn't support covariant return types on methods or properties, so this isn't directly mappable in C#, thus we can't directly expose EditText.Text as an Android.Text.IEditable...)<BR><BR>Thus demonstrates the major reason that we're not exposing TextView.Text as a string: it's NOT a string, it IS changed in subclasses to return different types, and if we were to expose it as a string you would LOSE data, as many types actually use SpannedString [3] to encode formatting and related information into the string, and none of this additional information could be readily stored in a string.&nbsp; The approach we're currently exposing would support casting the IEnumerable&lt;char&gt; to e.g. Android.Text.ISpannedString to manipulate the underlying data with full fidelity.<BR><BR>- Jon<BR></DIV></BODY></HTML>
