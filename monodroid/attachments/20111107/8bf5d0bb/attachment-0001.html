<p class=MsoNormal><font size=2 color=navy face=Arial><span style='font-size:
10.0pt;font-family:Arial;color:navy'>It cost me and my team weeks because we
didn't notice the issues until we went live.  The overhead and cost of web
services is documented all over Google (no one particular place) as I recall. 
There seemed to be more written about it on the Java side.  So, after spending
some time to isolate the problem, we got our network team involved and then
isolated the problem.<o:p></o:p></span></font></p>

<p class=MsoNormal><font size=2 color=navy face=Arial><span style='font-size:
10.0pt;font-family:Arial;color:navy'><o:p>&nbsp;</o:p></span></font></p>

<p class=MsoNormal><font size=2 color=navy face=Arial><span style='font-size:
10.0pt;font-family:Arial;color:navy'>During the research, one of the engineer's
mentioned that the phone speed was close to an old 1200 baud modem, and that's
when it struck me.  Why not communicate like we did during the 70's?  Heck we
got great speed out of even those 300 baud modems.  So, basically we post/get a
xmodem/ymodem package back and forth. <o:p></o:p></span></font></p>

<p class=MsoNormal><font size=2 color=navy face=Arial><span style='font-size:
10.0pt;font-family:Arial;color:navy'>We don't chunk it up into packets, but
basically take the data and encrypt it and then add a CRC to it and post it. 
On the server side we uncrypt it and check the CRC, now TCP/IP keeps the data
clean, so the CRC is just because the engineer's wanted to be able to post
without TCP/IP.  Theoretically,  we could do a dialup an post to a modem
without TCP/IP therefore the CRC would come into affect.<o:p></o:p></span></font></p>

<p class=MsoNormal><font size=2 color=navy face=Arial><span style='font-size:
10.0pt;font-family:Arial;color:navy'><o:p>&nbsp;</o:p></span></font></p>

<p class=MsoNormal><font size=2 color=navy face=Arial><span style='font-size:
10.0pt;font-family:Arial;color:navy'>The other thing was serializing objects to
XML really creates a PIG XML set if you really look at it.  For example, your
XML has elements that are empty or set to the default and look something like
this:<o:p></o:p></span></font></p>

<p class=MsoNormal><font size=2 color=navy face=Arial><span style='font-size:

middle name is empty but you're sending the overhead to tell the ever it is

<p class=MsoNormal><font size=2 color=navy face=Arial><span style='font-size:
10.0pt;font-family:Arial;color:navy'>The second example uses only 17 characters
instead of 85 which is 80% smaller.  Now if you take out the SOAP, you can see
the difference you can start making.  <o:p></o:p></span></font></p>

<p class=MsoNormal><font size=2 color=navy face=Arial><span style='font-size:

<p class=MsoNormal><font size=2 color=navy face=Arial><span style='font-size:
10.0pt;font-family:Arial;color:navy'>Notice that 1 is used instead of the
proper name, also notice that element 2 is not sent therefore the server will
assign the default to it.  Also, we use the pipe character to end the element
(downside you cannot send a pipe).  Overall do this for a large object and a
few thousand phones and all these bytes add up.  It does mean you have to build
some custom classes on each end to handle the serializing and know that element
1 maps to the First Name we keep the mapping in the class and then just version

<p class=MsoNormal><font size=3 face="Times New Roman"><span style='font-size:
12.0pt'><br>
&gt; Web service is just a URL and we dynamically build them all the time. <br>
&gt; Matter of fact we use a random number generator. &nbsp;For example, we
have <br>
&gt; 20 websites and we us ws_x.hostURL.com. &nbsp;ws_0 ws_1 ws_2 etc etc
&nbsp;we also <br>
&gt; dynamically build the hostURL too. <br>
&gt; <br>
&gt; Note however, we dropped web services in our android projects because of <br>
&gt; the overhead. &nbsp;We replaced it with a simply post/get. &nbsp;The
network <br>
&gt; overhead of directly consuming web services caused network to memory <br>
&gt; issues. &nbsp;When we added all the SOAP it was BLOATed and caused issues
on <br>
&gt; phones etc which have less bandwidth and memory. <br>
&gt; <br>
&gt; After messing around with web services we decided the overhead was too <br>
&gt; much so we built a webpage that accepts Post/Gets. &nbsp;Now, we build a <br>
&gt; special XML record set that only has elements that are not empty (or not <br>
&gt; set to default - no need to pass them). &nbsp;We then encode it, add SOH +
<br>
&gt; BLOCK + data + &nbsp;CRC + ETX (yes the old xmodem data pack) and post it
to <br>
&gt; the website. &nbsp;The website makes sure there is a SOH and ETX,
calculates <br>
&gt; the CRC on data and compares it. Then we process the xml if it passes. <br>
&gt; This reduced our overhead by over 80% compared to using web services. <br>
&gt; We pass binary data the same way. &nbsp;Bottom-line we never noticed
issues <br>
&gt; on computers, but when we went to hand held devices (with slower <br>
&gt; bandwidth) the web services was transmitting to much data and causing <br>
&gt; memory issues. <br>
