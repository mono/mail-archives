<div>Binding Services to Activity - Best Practice?</div><div><br></div><div>I&#39;ve been wrestling with what the best way to bind my activity to a service is.  In the java world, there seems to be a couple common ways:</div>
<div><ol><li>Create an anonymous type of BroadcastReceiver inside the Activity, so that you have access to the variables in the Activity&#39;s scope and can properly deal with them.  Here&#39;s an example of what i mean: <br>
<a href="http://stackoverflow.com/questions/4555215/android-broadcastreceiver-within-activity">http://stackoverflow.com/questions/4555215/android-broadcastreceiver-within-activity</a><br>They basically dynamically register and unregister the broadcast receiver on the resume and pause overrides of the activity.  <br>
<br>In C#, the best way to mimic this that I can think of is to subclass BroadcastReceiver, stick an event in the OnReceive override, then keep an instance of your subclassed receiver in your activity, and dynamically register and unregister it in the OnResume and OnPause.  This way doesn&#39;t sound terrible.<br>
<br></li><li>Using Binder, and ServiceConnection to get the instance of the Service.  This doesn&#39;t really help with the communication from the Service back to the Activity though.  It&#39;s more useful to call a method on the service or inspect something on the service, if the Activity initiates it.<br>
<br>This is easy enough to do in C#, and I&#39;ve got an enhancement request filed with a suggestion for some reusable generics types to make this easier (ServiceBinder&lt;TService&gt; and ServiceConnection&lt;TService, TBinder&gt; - See: <a href="https://bugzilla.novell.com/show_bug.cgi?id=665345">https://bugzilla.novell.com/show_bug.cgi?id=665345</a> ).<br>
This seems a bit clunky as well, and if you look at my enhancement, it still requires an event, to know when we actually have a good reference to the service instance.  I don&#39;t like this method very much, although I recognize it may be necessary to communicate with 3rd party services beyond our control (although I doubt that&#39;s a common use case).<br>
</li></ol><div>I&#39;ve sort of come up with my own way that seems to work, but I&#39;d like some opinions on any possible caveats I&#39;m simply not seeing.  So here goes:</div><div><br></div><div>Declare a <i><font class="Apple-style-span" color="#666666">public static event Action SomethingHappened;</font></i> in the Service.  The Activity could then simply Register and unregister from the event in the OnResume() and OnPause() overrides.  This would be quite elegant, I think, but of course it uses a static event, which is probably not the best practice.  I&#39;m also not sure what implications this could have for multiple instances of things.  If you had several instances of the Activity, all instances would receive the event.  This could be useful or annoying, depending on what you need for your application.  </div>
</div><div><br></div><div>A slight variation of this would be to create the event at the instance level, and use the ServiceConnection and Binder to get an instance of the Service, and then subscribe to the instance&#39;s event.  This is a lot of verbosity for not a lot of savings.  I think in most cases the static event would be worth the cost, just for code readability.  </div>
<div><br></div><div>Does anyone have an opinion on the subject?  I&#39;d be curious to know what other people are doing!</div><div><br></div>---<br>Jon (aka Redth)<br>
