Byron,<br><br>Here are few thoughts.<br><br>Performance.&nbsp; I find performance a hard thing to measure.&nbsp; For example, I originally started programming C on Unix 20+ years ago.&nbsp; My career path took me VB on Windows, and then to .NET.&nbsp; Not once was I asked about performance by a customer when the sucject of frameworks came up.&nbsp; typically, in a line of business app, performance is a db issue.&nbsp; Now, with a mobile app, performance has a bunch of various potential&nbsp; bottlenecks.&nbsp; 1 ghz ARM processors with 512 megs of ram can be limiting.&nbsp; 3g networks are not that fast.&nbsp; Having said all of that, there are still the .NET issues.&nbsp; It takes time for an app to spin up.&nbsp; How much?&nbsp; Unfortunately, the linked is not done so we just don't know.&nbsp; We won't know about memory utilization, working sets, or any of that cool stuff at this point in time.<br><br>There is a size overhead with the currently planned embedded runtime.&nbsp; With that size, you get the .NET framework.&nbsp; You don't have to go learn the framework built into Android.&nbsp; You get to use the .NET framework you already know.&nbsp; You don't have to go learn Eclipse, you get to continue to use Visual Studio, which contains a large ecosystem that makes your life easier. Its not that eclipse or java as implemented in Android has anything wrong, heck its a very good toolset.&nbsp; Its just that can take time to come upto speed&nbsp; on java/dalvik and Eclipse.&nbsp; Eventually, we'll get a separate runtime published by Novell so the size will go down for the individual apps. &nbsp; (Sorry if that wasn't suppossed to be public guys)<br><br>On a programming, you have to be efficient with what you do. Remember, its not a desktop system, don't create a ton of objects.&nbsp; Be efficient with what you have.&nbsp; You have really limited resources, cpu, memory, screen.<br><br>as for database choices, depending on remote databases will mean that your app is continually hitting the 3g/wifi antennas.&nbsp; That will hit the battery.&nbsp; I can drain a battery on my iphone faster than you can imagine because I hit the antennas too much.&nbsp; I think I have this fixed enough be "caching" data in sqlite and only hitting the web services when I need to<br><br>On the performance front, I would always expect Mono to be slower than .NET on the same operation in the same situation and hardware.&nbsp; I don't use the desktop/server versions of Mono, so I can't compare Mono and .NET on the same hardware.&nbsp; Novell has limited resources in comparison to Microsoft, so I don't expect them to be able to catch them with regards to performance.&nbsp; I know that Novell is working on changes to the base Mono framework to narrow the performance gap, specifically in the area the gc.&nbsp; I don't know how that will translate to MonoTouch and MonoDroid.<br><br>Hopefully, a few of these random thoughts (some can call it drivel) are helpful.&nbsp; As for the resources, check out our MonoTouch book that shipped back in July. Bryan Costanich has once coming out shortly.&nbsp; Mike Bluestein has one coming out as well.&nbsp; MonoDroid is much further behind with regard to learning resources.&nbsp; I have an article coming out on it, not sure when it will be published. Its a beginner article, so probably not for this group.<br><br>Wally<br><br><hr id="stopSpelling">Date: Sun, 12 Dec 2010 09:43:07 +1100<br>From: <a href="mailto:mrhassell@gmail.com">mrhassell@gmail.com</a><br>To: <a href="mailto:monodroid@lists.ximian.com">monodroid@lists.ximian.com</a><br>Subject: Re: [Monodroid] Doing a presentation for a local .NET User Group<br><br>Oh Byron,<br><br>One more thing is if you are a professional .Net developer then Monodroid is the obvious choice as you would already be familiar with the stack and working with importing library's.<br><br>In effect I prefer working with Monodroid to the Eclipse environment on anything overtly complex. The smal
