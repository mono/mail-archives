&gt; <br>&gt; That&#39;s not specific enough. :-)<br>&gt; <br>&gt; The class libraries that Mono for Android provides is...interesting. Succinctly, it&#39;s a .NET 4.0-influenced assembly set with an overlapping set of Silverlight types and members, lacking the UI bits. (Which likely doesn&#39;t help at all.)<br>
&gt; <br>&gt; Instead of asking about framework versions, you should instead ask about types (and possibly assemblies). :-)<br>&gt; <br>&gt; For example, the Action&lt;...&gt; and Func&lt;...&gt; family of delegate types are supported, up through Action`16 and Func`17, which is what .NET 4.0 and Silverlight support.<br>
&gt; <br>&gt; However, the IEnumerable&lt;T&gt; interfaces are not covariant, and thus is like Silverlight. The C# compiler we provide on OSX is C#4-ish (it supports generic variance declarations, but no `dynamic`, and generic variance isn&#39;t entirely helpful when most of the class library is lacking support...).<br>
&gt; <br>&gt; Assemblies are .NET 4.0-like, not Silverlight-like, so the System.Net namespace is in System.dll (not System.Net.dll). System.Xml.dll contains XmlDocument and System.Xml.Linq.dll contains XDocument, like .NET 4.0 (while Silverlight drops XmlDocument and moves XDocument into System.Xml.dll).<br>
&gt; <br>&gt; In short, there isn&#39;t really a short way to answer your question, except &quot;maybe.&quot; You can also view the class library documentation, which lists all of the namespaces and types provided:<br>&gt; <br>

&gt; <br>&gt; A related question is whether Portable Library Projects are supported. I don&#39;t know; I haven&#39;t tested them. They are not a priority for 1.0, but we will support them in a later release.<br>&gt; <br>&gt; Finally, are there any assemblies that aren&#39;t supported? Regarding framework assemblies, it again depends upon what your reference is: with respect to .NET 4.0, we&#39;re missing Windows Worfklow and Windows Presentation Foundation (among others, I &#39;m sure), and the WCF we provide is Silverlight-like, not .NET 4.0-like. If your reference is Silverlight, we&#39;re missing System.WIndows.dll and anything related to it.<br>
