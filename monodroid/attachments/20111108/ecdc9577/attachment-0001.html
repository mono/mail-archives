      <div>I'm using Xamarin, specifically because I'd like to share as
        much code as possible between the platforms. I've noticed I can
        only reference library projects that target "Mono for Android
        x.x". I'd hoped to be able to simply create a Mono class library
        and only use the classes supported by MonoDroid or MonoTouch and
        reference said project from either platform.</div>
    </blockquote>
    Our team does it with single solution and projects for each platform
    and shares on source<br>
    code level with links to source files (cs, xaml etc) not IL (by
    referencing assemblies).<br>
    <br>
    Architectural only 2 layers Business Logic BL and Presentation Layer
    PL with<br>
    assemblies per platform... Controlers and ViewModels in MVC and MVVM
    can be <br>
    moved to those assemblies...<br>
    <ul>
      <li>Business logic (BL) per platform linked with Project Linker. <br>
      </li>
      <ul>
        <li>Starting with WP7x project as "smallest common denominator -
          subset" <br>
          (because of various restrictions - Android Mono4Android and
          iOS <br>
          MonoTouch are still in front).</li>
        <li>all projects have WP7 project as "linked source project"
          (Project Linker terminology)<br>
          meaning code in them/their folders is just a link to cs file
          in WP7x prj<br>
        </li>
      </ul>
      <li>Presentation Layer PL dlls (User Control library project
        types)<br>
      </li>
      <ul>
        <li>our team uses them for reusing <br>
        </li>
        <ul>
          <li>some UI code if it can be shared across projects<br>
          </li>
          <li>xaml controls (write up is on the way).</li>
        </ul>
        <li>MonoDroid - does not apply cos layouts from other <br>
          assemblies cannot be used - yet</li>
      </ul>
      <li>Presentation Layer EXEs - main entry points - per platform<br>
      </li>
      <li>Unit testing Projects</li>
      <ul>
        <li>if Windows Forms BL assembly reuses bunch of WP7x code and
          adds<br>
          some more, then NUnit test for WF or WPF project will cover a
          lot from<br>
          WP7x and You are not stuck with WP7 based UnitTesting in
