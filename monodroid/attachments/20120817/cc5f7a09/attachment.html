      <pre wrap="">Hi

I guess I'm missing something. I'm developing a mobile version of a client
that is already available on PCs in multiple forms. Those different clients
are meant to share as much code as possible in order to cut down on
repetitive work. So, there's a bunch of libraries that are to be shared
amongst clients (and they build against standard .NET 4.0)

Here's where it gets tricky:
I'm creating derived classes from some of the base classes in my libraries,
and override methods where I need to behave differently for Android. While I
can step through the overridden code just fine, I cannot step into any
method that is defined in a base class. And to make matters worse, even
though the base class has proper error handling and should never throw an
exception in the method I'm calling, I started getting Unhandled Exceptions
(specifically a System.ServiceModel.EndpointNotFoundException). If I run the
exact same method with the same parameters in a PC based client, not only
can I step through it, but there are no exceptions (and when I do something
wrong on the PC client, then the exception is properly caught, and logged).</pre>
    </blockquote>
    It seem like You are having <br>
    <ol>
      <li>common assembly/dll built against desktop/client version netfx
        4 and</li>
      <li>delta (derived subclasses) assembly for Android and/or other
        platforms</li>
    </ol>
    <p>If this is the case -  I would strongly suggest not to do so!<br>
      We had a case (SharpSNMP based app) which worked flawless for
      several months and suddenly started<br>
      to explode in our face -  after some update. Colleague of mine
      just dropped desktop version of<br>
      SharpSNMP into MA project and it worked - for a while.<br>
    </p>
    <p>Workflow:<br>
    </p>
    <ul>
      <li>share code through linking and extensive use of partial
        classes<br>
      </li>
      <ul>
        <li>VS add existing class/code file as link (add existing item
          +/ Add as link (not just **Add**) <br>
          this is very tedious, but can be avoided with tools such as VS
          productivity tools, power<br>
          commands and VS project linker for whole projects</li>
        <li>for code sharing - common subset (lowest/smallest common
          denominator) is mandatory.<br>
          This would be platforms that supports minimal set of
          features/capabilities - in our experience<br>
          WP and Silverlight (example: only async methods***, no
          UdpClient, XmlSerialization etc, etc)</li>
        <ul>
          <li>put all common code that can be extracted (compiled) into
            this project</li>
          <li>link it in other projects (like onion add features) again
            our experience levels from inside</li>
          <ul>
            <li>MT and MA -  those are the same for BCL, but separation
              in more projects allows<br>
              separation of build processes for each platform - so what
              guys from Xamarin do<br>
              with msbuild/xbuild does not concern us (until needed)
              they can do optimizations<br>
              and we have faith in them</li>
            <li>RIA <br>
              could be before MA/MT - matter of decission</li>
            <li>desktop/client</li>
            <li>server (asp.net)</li>
          </ul>
        </ul>
      </ul>
    </ul>
    <p>Sample (our workflow for business logic layer, with service layer
      usually more platform specific code, UI is very specific):<br>
    </p>
    <ul>
      <li>in VS create lib projects for each platform</li>
      <ul>
        <li>BL.WP</li>
        <li>BL.MA</li>
        <li>BL.MT</li>
        <li>BL.dClnt (desktop client -  if not hosting services) etc</li>
        <li>BL.Srvr.ASPnet (if needed)</li>
      </ul>
      <li>change properties for each project</li>
      <ul>
        <li>Default Namespace (recommended - I had problems with
          compilation if this was not the case)</li>
        <li>Assembly name (prefix before dll or exe) <br>
          optional - I doubt someone will deploy App.dClnt.exe on other
          platforms, but when searched <br>
          or browsed BL.MT.dll would be clear it is dll with business
          layer for MonoTouch...</li>
      </ul>
      <li>with VS Project Linker add project links</li>
      <ul>
        <li>MA, MT, dClnt etc -  get their source project BL.WP<br>
          Project linker nomenclature:</li>
        <ul>
          <li>source project - where .cs physically resides</li>
          <li>target[ing] project - where links to .cs files appear<br>
          </li>
        </ul>
      </ul>
      <li>start implementing/porting layer BL in **common denominator**
        = WP<br>
      </li>
      <ul>
        <li>add class <br>
          partial class Person in Person.cs (Name, Last, first , gender,
          date of birth)<br>
          see comment**</li>
        <li>this adding of class will make project linker to add links
          to this class to all targeting projects,<br>
          so Person.cs appears as link in all other projects<br>
        </li>
      </ul>
      <li>add platform specific code samples:<br>
        this means add new code file or partial class file in each
        project when needed<br>
        we do it manually <br>
      </li>
      <ul>
        <li>copy class file (partial - remember?) <br>
        </li>
        <li>remove everything inside, but namespace and classname</li>
        <li>rename it with some meaningful suffix  Person.WP.cs ,
          Person.MA.cs<br>
        </li>
      </ul>
      <ul>
        <ul>
          <li>WP<br>
            Person.Silverlight.cs (Silverlight.cs suffix will tell
            Project linker not to add links in targets)</li>
        </ul>
        <ul>
          <li>MT/MA<br>
            Person.MT.cs (MT.cs suffix is our convention)</li>
          <li>desktop (this is my IceCreamSandwich)<br>
            Person.WF.cs or Person.WPF.cs or whatever, **but** inside:<br>
            [Serializable]<br>
            partial class Person{}<br>
            Note: attributes are sorta "additive" across partial classes
            and this attribute will not<br>
            colide with Silverlight (Wp and/or RIA) serialization cos it
            is in desktop project.<br>
            For Silverlight add Person.Silverlight.cs with:<br>
            [DataContract]<br>
            partial class Person{}<br>
          </li>
        </ul>
        <li>so in each target project You end up with <br>
        </li>
        <ul>
          <li>link to Person.cs and <br>
          </li>
          <li>one or more Person.<sufix>.cs with deltas for
            platform</li>
        </ul>
      </ul>
    </ul>
    <p>Got it?<br>
    </p>
    <p>** I cannot understand why VS still has project item (Class.cs)
      not as partial - it does not hurt, but helps a lot<br>
      Some suggestions to MonoDevelop guys (M. Hutchinson and others) to
      make MD a lot better w/o much<br>
      effort to add some features from (VS power commands, productivity
      tools and project linker) such as:<br>
      <br>
    </p>
    <ul>
      <li>copy +/ paste as link for files and folders</li>
      <li>copy as reference,  add as reference for projects and refs
        in(reference lists) although MD edit references is<br>
        a lot better then native VS reference adding</li>
      <li>linking projects like Project Linker</li>
      <ul>
        <li>but with configurable exclusion patterns<br>
          Project Linker does not link (propagate links) for files with
          extensions <br>
        </li>
        <ul>
          <li>*.Silverlight.cs or *.WPF.cs (and some others I believe)<br>
          </li>
        </ul>
      </ul>
      <li>some ops on classes (this is possible with item and project
        templates and we do it, but to add some sugar<br>
        to MD)<br>
      </li>
      <ul>
        <li>make partial</li>
        <li>make partial and split 2 files with partial keyword!</li>
        <li>join dependency (DependsOn in .csproj) to make code readable<br>
          I'm doing that with axml file and it code behind<br>
        </li>
      </ul>
    </ul>
    <p>Cons:<br>
    </p>
    <ul>
      <li>complexity<br>
        with adding platforms number of projects linearly grows<br>
        n-layers on m-platforms will cause  n*m projects<br>
        we can discuss this, but adding single platform is adding
        complexity in itself and linear <br>
        growth is not bad (this is what I have heard)<br>
      </li>
      <li>tools<br>
        need to install and to get them know</li>
    </ul>
    <p>Pros:<br>
    </p>
    <ul>
      <li>cleaner code<br>
        no #ifdefs</li>
      <li>projects are compiled separately (oh yeah and packaged)<br>
        meaning optimizations etc is up to build process and not our
        concern<br>
      </li>
    </ul>
    <p>*** MT and MA allow sync methods (WebClient). If this is allowed
      and possible it does not mean that this is<br>
      OK. Try not to use them, but for proof of concepts. Mobile devices
      (I would say not only mobile apps, but any<br>
      app that behaves nicely) should be responsive - implying async is
      good and forget about sync methods ASAP.<br>
    </p>
    <p>Tools:<br>
      <br>
    </p>
    <p>I'm preparing small writeup how to port/migrate/expand existing
      apps or libs to target more platforms<br>
      based on experience with Google GData and SharpSNMP (there are few
      more others ready to be published<br>
      on github).<br>
    </p>
