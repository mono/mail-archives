I've gone ahead and rewritten my app to use HttpWebRequests directly. This is the simple code I'm using

                HttpWebRequest request = (HttpWebRequest)WebRequest.Create(uri + "/IsAlive");
                request.Proxy = new WebProxy();
                request.AllowAutoRedirect = true;
                request.Timeout = 10000;
                HttpWebResponse response = (HttpWebResponse)request.GetResponse();

Using this approach I've been able to rule out Restsharp as the reason for the problem, but the problem itself does persist.

I then went ahead and rewrite the connectivity check to use a simple socket connection (TcpClient) - so now I'm in the situation where my connectivity check reports that the server is online, but the login operation that follows (using a HttpWebRequest) does time out.

So, I'm wondering - looking at threads like this: http://stackoverflow.com/questions/4654480/android-httpclient-perfomance - is that the issue I'm hitting?

I'm running comet style long polling in my app, so at all times (unless connectivity goes down - but I just realize I'm not calling abort on that connection, I'm simply aborting the task that executes the poll) I already have one of two requests, and as I'm checking connectivity to two servers, if there's indeed a limit of two requests, that would indeed explain a lot.

So, could it be that I'm hitting the "two concurrent requests to the same server" limit, and if so, is there a way to get around it? Looking at the RestSharp source, they do create requests just the way I am:
