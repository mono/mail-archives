<br>

<br>
In my code - I've had this problem earlier (both of you participated in the<br>
thread last year), and if this were it, the problem would appear differently<br>
(not being able to make any new connections - I can make new connections<br>
just fine when I get those "timeouts" or rather connection freezes). In fact<br>
I even keep a log of the number of parallel connections just to be on the<br>
safe side - so whenever I have 2 or more parallel connections, it is noted<br>
in my log - and I have yet to see 4 parallel connections (I'm not looking at<br>
my code right now but I may have even set the limit  higher).<br>
<br>
Note that it is not timeouts in making the connection or keeping the<br>
connection alive - the connection is up and running as far as the TCP layer<br>
is concerned (confirmed by wireshark traces), the server is processing - but<br>
long poll means the client may have to wait for a long time until something<br>
is returned to a request). Eventually, my client will abort the connection<br>
since it thinks it has not received any data, but the server has tried to<br>
send some.. it is as if the connection has become "frozen" in place. In a<br>
network trace I see nothing that would show a problem with the connection.<br>
<br>
I haven't had time for a simple repro project - but I've found that 4<br>
minutes is about as long as I can go... if the server responds later, the<br>
answer may never make it to the client (my client has a dynamic timeout<br>
duration for the connection controlled by the server - and it is always<br>
slightly longer than the server side timeout, so the server will after a<br>
while simply send the current state even if nothing has changed so that we<br>
don't have any errors in the communication chain). I've experimented for<br>
hours, and with the server delaying responses by 5 minutes, the response<br>
then only gets to the client unreliably (one time it works, the next it<br>
doesn't, etc.), larger values mean the same, and at 4 minutes, things are<br>
reliable. Also, just because one connection is frozen doesn't mean I can<br>
initiate another... that works fine and not only that... if I do so, then<br>
the response from the long poll has a good chance of being delivered after<br>
all - it is as if making another connection revitalizes the existing<br>
connection (but this is also not 100% reliable - the additional connection<br>
may work and should trigger a result being sent to the long poll connection,<br>
but sometime this doesn't work).<br>
