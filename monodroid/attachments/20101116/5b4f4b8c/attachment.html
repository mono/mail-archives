</div>No.<br>
<br>
There are two sets of assemblies, the &quot;core&quot; assemblies and the<br>
platform-specific assemblies.<br>
<br>
The &quot;core&quot; assemblies are mscorlib.dll, System.dll, etc. -- the things<br>
you&#39;ll find on most any .NET implementation.  Code using the core<br>
assemblies WILL be portable between MonoDroid and MonoTouch (and<br>
probably WP7 and Silverlight and desktop .NET and...).<br>
<br>
The platform-specific assemblies wrap and (more or less directly) expose<br>
the underlying platform; this consists of monotouch.dll for MonoTouch,<br>
and Mono.Android.dll for MonoDroid.<br>
<br>
Code using platform-specific assemblies WILL NOT be portable.<br>
Consequently, all UI code written on MonoTouch WILL NOT be directly<br>
usable on MonoDroid (and vice versa) [0].<br>
<br>
The solution, of course, is to carefully design your app so that<br>
platform-specific code is kept separate (MVC, anyone?), thus allowing<br>
greater reuse of your core algorithms and data structures.<br>
