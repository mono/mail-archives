This sounds like a problem I was having a few months ago. It would be nice<br>
to have a little more detail on how you are setting up your rendering<br>

this information will be useful to you.<br>
<br>
I am assuming that you have set up an AndroidGameView with a run loop that<br>
is started in the handler for an OnLoad event. You probably also have some<br>
kind of OO framework to handle initializing OpenGL textures (among other<br>
renderable objects) such that they can be drawn later in response to<br>
OnRenderFrame events without the need to reconstruct them each time. If this<br>
is at all similar to your approach, there are two issues you need to be<br>
aware of:<br>
<br>
1. AndroidGameView maintains a Graphics context that is destroyed any time<br>
the view is paused. The graphics context maintains references to various<br>
resources such as textures, framebuffers, buffers, renderbuffers, shaders,<br>
and program contexts. *When the graphics context is destroyed the GPU will<br>
also destroy any resources associated with it.* This is normal behavior and<br>
is documented on the OpenTK site. So to handle this you will need to make<br>
sure that you have a way to signal a context change (do this upon your<br>
activity resuming) to your renderer so that it can remove any cached<br>
references to resources the GPU has deleted and then recreate them.<br>
<br>
2.  When the graphics context is destroyed in response to a screen rotation<br>
or when the device sleeps it is never recreated. I am assuming that there is<br>
a problem with the recreation of the SurfaceView because you never get<br>
another OnLoad event. This is apparently a known bug although I can&#39;t dig up<br>
a link at the moment. Unfortunately, I have not found a way around this<br>
problem that works consistently.<br>
