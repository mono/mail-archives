    Hi<br>
    <br>
    Few years ago I tried to reuse code (Domain, Business logic) from
    Silverlight on "binary" <br>
    level in WPF and that was doable, but just not feasible.<br>
    That's the reason I prefer and we argued a lot, but came to
    conclusion that sharing code<br>
    is better.  <br>
    <ol>
      <li>Sharing code with Project Linker</li>
      <ol>
        <li>hack if necessary in csproject file for extra extensions
          M4A.cs, MTch.cs<br>
          normally Project Linker is used for WPF/Silverlight targeting
          projects</li>
        <li>process</li>
        <ol>
          <li>determine smalles subset usually WP7<br>
            WP7 has smaller subset of features <br>
            example all methods are Async while MTch and M4A allow sync
            (WebClient)<br>
          </li>
          <li>link other projects (MTch and M4A) to WP7</li>
          <li>each new cs file added to WP7 proj shows in other projects<br>
            ability to add multiple src files.<br>
            Files with *.Silverlight.cs extension will not be linked
            into other projs.</li>
          <li>classes should be partial so if there is platform diff add
            deltas like:</li>
          <ol>
            <li>Class.Silverlight.cs to WP7 project<br>
            </li>
            <li>Class.MTch.cs to MTch project</li>
            <li>Class.M4A.cs to M4A project</li>
          </ol>
          <li>If needed some stuff can be manually added as link with
            copy-as-link/paste<br>
            VSCommands, Power Commands or Productivity Tools (not sure)<br>
          </li>
          <li>If project linker does not pickup some links, just go to
            source project (WP7)<br>
            do exclude/include and voila everything comes as link in
            linked projects.</li>
          <li>all above works in MonoDev so when needed to switch there
            shouldn't be any <br>
            problems<br>
          </li>
        </ol>
      </ol>
      <li>heavy use of partial classes<br>
        common stuff in Class.cs this is propagated by Project Linker
        through other linked<br>
        projects (except extensions - hack from 1.1)</li>
      <li>team consent and discipline - naming<br>
        all controls same names across projects. <br>
        We had to make table with what is called what and where.<br>
        Sample:<br>
        we call all labelName if that is the purpose of Control/Widget
        (short text)<br>
        WinForms - Label<br>
        WPF - TextBlock<br>
        SL - TextBlock<br>
        SLWP - TextBlock<br>
        MTch - UITextView<br>
        M4A - TextView</li>
      <li>For more VS centric devs small shell script to add/remove
        MonoTouch ProjectType GUID <br>
        for easier loading into VS (dos batch using sed.exe and bash on
        MacOSX)<br>
      </li>
    </ol>
    <p>Other things would be that <br>
    </p>
    <ul>
      <li>code can be linked into desktop projects and as we know unit
        testing for desktop<br>
        projects is still ahead of WP7 test projects or MTch and M4A.<br>
        And the question is whether we need UnitTests for each platform
        <br>
        (we do only if there are diffs)</li>
      <ul>
        <li>unit test for  mobile still involve emulators/simulators, so
          testing with nUnit is faster<br>
        </li>
      </ul>
      <li>source is really compiled for each platform, so some problems
        might show up earlier than<br>
        when just tossing assembly as reference into project.</li>
    </ul>
    <p>HTH<br>
    </p>
