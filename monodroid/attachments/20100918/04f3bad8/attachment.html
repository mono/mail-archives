Alas, there's only one sensible answer: use the same library, don't try<br>
to &quot;port&quot; between them.<br>
<br>
You can use Mono.Data.Sqlite on Windows (you just need to install the<br>
native SQLITE3.DLL with your app; it's entirely P/Invoke driven).<br>
<br>
Alternatively, you could take the System.Data.SQLite sources and<br>
(presumably) recompile them to target the MonoDroid profile, which would<br>
give you the existing System.Data.SQLite API. When doing this, you'll<br>
need to build the &quot;managedonly&quot; version, as mixed mode assemblies
won't<br>
work, and make sure you DllImport(&quot;sqlite3&quot;) (as MonoDroid will have
a<br>
config file to map &quot;sqlite3&quot; to the system libsqlite3.so file).<br>
<br>
Alas, the latter approach will make your app larger (an extra assembly<br>
to ship with your app), but Mono.Data.Sqlite.dll is only 166KB in size,<br>
so that's potentially negligible.<br>
<br>
As an alternate-alternate approach, you could just add 'using' aliases<br>
to your code:<br>
<br>
&nbsp; &nbsp; &nbsp; &nbsp;#if MONODROID<br>
&nbsp; &nbsp; &nbsp; &nbsp;using SQLiteConnection =
Mono.Data.Sqlite.SqliteConnection;<br>
&nbsp; &nbsp; &nbsp; &nbsp;// repeat for other types<br>
&nbsp; &nbsp; &nbsp; &nbsp;#endif<br>
<br>
Alas, you'll need to do this in each file you use SQLite within.<br>
