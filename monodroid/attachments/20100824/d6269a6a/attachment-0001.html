<div>Ted,</div><div>   To your argument about refactoring an interface to/from an abstract class, as long as the abstract class is purely abstract ( no implementations ) then in a sense it&#39;s an interface and can keep the &quot;I&quot; prefix. it only become tricky when you introduce a layer of implementation through a common base class. Refactoring an interface to a concrete class has the same problems in both C# and Java regardless of the naming. In C# you typically would see a base class introduced with a name that leaves off the &quot;I&quot; prefix, in Java, the suffix &quot;Base&quot; is added to the interface name.</div>
<div><br></div><div>But this isn&#39;t just about naming, it&#39;s about convention. And this convention isn&#39;t something that Mono started, it was started by Microsoft. My personal opinion is that MonoDroid is 1) about bringing .Net developers to the Android platform, 2) bringing Android developers to .Net/Mono. Keeping the naming conventions that have been set forth by Microsoft will make the transition for most .Net/Mono developers easier than if you give them the raw Android API&#39;s, But your right, in that the Android -&gt;  Mono set will have more problems.</div>

</div>Yep, it&#39;s just you. Why should the developer care about the difference between an interface and a base class? If you know C# or VB, you know that only the left-most class can be anything other than an interface, and if there&#39;s ever *really* a question, right-click-Jump-to-Definition tells you for certain (or the compiler/LanguageService lets you know either by flagging an error or red squiggly). In a world where we&#39;re writing code inside a text editor, that kind of naming pattern makes sense, but in a statically-typed language.... not so much.<br>
