        While I think we all want a better design time layout solution,
        this is an issue that we can work around short/medium term. &nbsp;For
        me, the debugger and the performance are issues #1 and #1, by a
        long shot. &nbsp;I did a talk back at the beginning of May on mobile
        at a high level. &nbsp;Two guys that work at a company I am familiar
        with said that they had looked at mono for android and decided
        to not not use monotouch and mono for android because of the
        debugger in monodroid. &nbsp;Would it have been a huge sale? &nbsp;I don't
        know, but it would have been a mono design win and it would have
        been a win for me/us as well as they would have been interested
        in our training. &nbsp;If these problems had been fixed, would the
        sale have been made? I'm not sure, but I know that this company
        was using .NET for their development so I think it had a good
        possibility.</div>
    </blockquote>
    <br>
    Our team has managed to overcome debugging issues through
    multi-targeting. <br>
    <br>
    Usually we star our project with several targets (WP7, silverlight,
    WF, WPF). <br>
    Business-Logic is in separate assembly shared as project link and c#
    files as source links. <br>
    It woudl be simplified MVC, MVVM model with Controller merged either
    in Business or<br>
    Presentation (View Layer).<br>
    <br>
    Great help is ProjectLinker tool (VS addin, see VS gallery, on
    codeplex) developed by<br>
    MS project prism team for sharing code between WFP and
    Silverlight....<br>
    <br>
    It is possible to exclude links with *.WPF.cs and *silevlight.cs, so
    they are not linked accross projects.<br>
    Tool has some flaws (we have tried to introduce *.WF.cs, *.M4A.cs,
    WP7.cs extensions for other<br>
    project types, got the code but had not enough resources to pull it
    through, so filled an issue on<br>
    codeplex).<br>
    <br>
    We have more than 70-80% code written in BusinessLogic layer and use
    DataBinding for <br>
    WF, WPF, Silverlight etc.<br>
    So most of the debugging is in BL and can be done with desktop
    targets WF or WPF...<br>
    And this is not so slow...<br>
    <br>
    When this is done then there is some debugging done with M4A, but it
    is not a lot...<br>
    <br>
    <br>
    Hope this experience helps someone...<br>
