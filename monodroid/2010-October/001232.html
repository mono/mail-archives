XAML is a better model in that it uses the class libraries. Simply having a
static XSD would make it hard to support extensions. It can also look inside
the values of attributes. This would be helpful in supporting @string etc.

While this is too much to expect in an initial release it's the kind of
capability which provide major advantages to using VS.

  As far as I read android sources, there isn't any xsd files (neither in
the runtime nor in the sdk sources), and android.widget.LayoutInflator[*1]
only uses reflection i.e.
Context.getClassLoader().loadClass().

Since users can use their own classes that for each derive from View, simple
xml schemas wouldn't apply. But for the foundation classes, it would be
possible to generate incomplete xsds from the API definition, or as an
easier choice, reflection from Mono.Android.dll.

I haven't practically used VS for years, so I don't know what and how VS
helps you with xsds though.
