On Jun 25, 2013, at 11:04 AM, Paul Johnson <paul at all-the-johnsons.co.uk> wrote:
> This is fine in Java
> 
> static SharedPreferences getSamplePreferences(Context context) {
>        return context.getSharedPreferences(
>          DeviceAdminReceiver.class.getName(), 0);
>    }

java.lang.Class.getName() returns the fully-qualified name of the class, e.g. `String.class.getName()` will return "java.lang.String".

Question: do you _need_ to use the Java name? if not, you can use Type.FullName:

	return context.GetSharedPreferences(typeof(DeviceAdminReceiver).FullName, 0);

If you absolutely need to use the Java name, e.g. for compatibility, you could hardcode it (but what's the fun in that?!), or (again) use Class.FromType():

	return context.GetSharedPreferences(
			Java.Lang.Class.FromType(typeof(DeviceAdminReceiver)).Name,
			0);

> Also, out of interest, do I need to add the manifest stuff that's listed on the website into my manifest file?

I don't understand which manifest stuff you're referring to. If the Java sample you're porting contains AndroidManifest.xml fragments, you should try porting those to use the [Activity] custom attribute (and related) first.
