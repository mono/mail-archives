And any VS project too - if You reference assemblies as dlls  (I like to 
call it on binary level or IL level)
with add reference - Browse and  not as project references.

I had similar problems with one WinForms app and 2 dlls made by junior devs.
Dlls were tested in single NUnit assembly and were referenced as dlls. 
Everything was ok.

Then one dev came from other, more complex project that had WebApp + 
bunch od other stuff.
So in order do speed up his development and testing he had in his
  - Tools / Options / Projects and solutions / Build and Run / Only 
build startup projects and deps
checked!!
(I do that sometimes to shut-up other exes - usually monotouch).

But in UI app only dll B was referenced, as long as there were bin  and 
obj folders with dlls A and B
in App C everything was fine. Don't ask me how they came up with bin and 
obj (was it before
or after turning option from above).
As soon as one would

  * make distclean (manually) remove bin and obj
  * pull the source from source control


We have tried to reconstruct this and this is scenario:

  * senior dev pulled out the source
  * project deps where referenced as dlls (binary level)

         <Reference  Include="ClassLibraryA, Version=1.0.0.0, Culture=neutral, processorArchitecture=MSIL">
           <HintPath>..\ClassLibraryA\bin\Debug\ClassLibraryA.dll</HintPath>
         </Reference>

    vs project reference

         <ProjectReference  Include="..\ClassLibraryB\ClassLibraryB.csproj">
           <Project>{324E9740-1C2B-4831-BD61-FC2B0C8B7B77}</Project>
           <Name>ClassLibraryB</Name>
         </ProjectReference>

  * senior dev rebuilt solution (Build solution F6 and/or Rebuild all)
    remember his setting -  it built only dll B
  * got the error
    one could see it coming if References node is expanded (warnings for
    missing dlls):
    Error    2
    The type or namespace name 'ClassLibraryB' could not be found (are
    you missing a using directive or an assembly reference?)
    C:\Users\moljac\Documents\Visual Studio
    2010\Projects\HolisticWare.DependencyTransitivityTests\WindowsFormsApplicationC\Form1.cs

    21    4    WindowsFormsApplicationC
    Error    3
    The type or namespace name 'ClassLibraryB' could not be found (are
    you missing a using directive or an assembly reference?)
    C:\Users\moljac\Documents\Visual Studio
    2010\Projects\HolisticWare.DependencyTransitivityTests\WindowsFormsApplicationC\Form1.cs

    21    34    WindowsFormsApplicationC
    Explanation: DLL B was not built cos A was not "Startup dependency"
    and wasn not built.
  * so he built A and B and everything was ok.
  * but the project grew (some temp files) and sometimes to be sure that
    app is starting from
    scratch they would delete (bin and obj) directories.
  * and with sets of builds, cleans, discleans (removal off bin and obj)
    You might end up
    that A dll is not in bin folder of App C and when pressing F5 You
    could get
    (as I did when I tried to reconstruct You problem with winforms this
    morning)

    A first chance exception of type 'System.IO.FileNotFoundException'
    occurred in WindowsFormsApplicationC.exe
    'WindowsFormsApplicationC.vshost.exe' (Managed (v4.0.30319)): Loaded
    'C:\Windows\Microsoft.Net\assembly\GAC_MSIL\System.Configuration\v4.0_4.0.0.0__b03f5f7f11d50a3a\System.Configuration.dll'
  * let say manual removal of A in B\bin and C\bin, rebuild would pass,
    but not regenerate it in C so
    exception may happen

Note The guys from MS tried to minimize those errors with stuff like 
<ProjectName>.FileListAbsolute.txt
This helps, but not always...

So this is not Mono4Android problem. Check folders, see why dlls are not 
copied, or use
project references that minimize such problems even more.
