On Feb 16, 2011, at 12:13 AM, Adam Langley wrote:
> I am 'deserializing' data from a data file that can be loaded into structs  
> using straight Marshal.PtrToStructure() - this only works because the  
> Android emulator is little endian (so is my data).
> My question is - will the Mono runtime ALWAYS be little endian?

The Mono runtime _itself_ will _not_ always be little endian; Mono runs on big-endian platforms such as PowerPC and S390.

That doesn't address the immediate concern of Mono on Android, though, as ARM is bi-endian (supports either big-endian or little-endian).  Despite this, all currently shipping Android devices use ARM little-endian, and thus Mono is currently little-endian on Android.

Could this change, theoretically?  Certainly.  Could it change, practically?  I personally doubt it, as it would require recompiling all native shared libraries (as any arrays/data/etc. would otherwise be wrong), creating a world of hurt for app developers.  Thus, if you don't care about big-endian platforms such as PowerPC, you (probably) don't need to worry about it (too much/this week).
