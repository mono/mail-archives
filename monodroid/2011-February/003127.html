On Thu, 2011-02-17 at 19:00 +0100, Morten Nilsen wrote:
> EditText.Text isn't actually a string..
> You can cast it using many methods, including Convert.ToString
>  
> I forecast that this question will arise again and again unless the
> Text property actually gets changed into a string at some point.
> While I appreciate the reasoning, it is a really unexpected
> behaviour..

We're working on it for P13.  The current thinking is to generate
overloads that take either string or Java.Lang.ICharSequence instead of
the current IEnumerable<char> one-to-one conversion.  

The less-beautiful part is the CharSequence return values, like this
case, which is really sugar for TextView.getText ().  We plan to expose
the following:

public string Text { get; set; }
public Java.Lang.ICharSequence TextFormatted { get; set; }

The Foo property/method will just be a thin wrapper around the
FooFormatted property/method which does the ToString()/Convert some
people seem to be struggling to discover with the current
IEnumerable<char> bindings.
