On Feb 21, 2011, at 11:02 PM, Adam Tybor wrote:
> Awesome, that is kind of like proguard for java based android apps.

Only partially. According to the ProGuard page [0], ProGuard:

	"shrinks, optimizes, and obfuscates your code by removing unused code..."

..and that's the only thing that the Linker has in common with ProGuard -- the Linker shrinks assemblies by removing code that it believes won't be required at runtime (via static analysis [1]), resulting in a smaller apk [2]. The MonoDroid linker does nothing else that ProGuard does: it won't rename anything, it won't obfuscate anything. It _only_ removes unused IL.

 - Jon

[0] http://developer.android.com/guide/developing/tools/proguard.html

[1] Static Analysis is not without it's limitations, as anything requiring runtime evaluation isn't tracked. For example, Type.GetType("Some" +_ "Type") may not preserve the type `SomeType`, which is why we have the Android.Runtime.PreserveAttribute custom attribute and the flexibility to link only SDK assemblies instead of everything.

[2] For comparison, the monodroid-samples/HelloWorld app, without the shared runtime (which is what will eventually be deployable to the Android Market, we won't support shared runtime deployment at this time), the unlinked .apk size is 15 MB, while the linked size is 4.2 MB, or 28% of the unlinked size. We greatly recommend use of the linker to shrink the size of bundled assemblies.
