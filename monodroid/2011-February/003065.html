To make life easier, you could always just add an extension method…

public static AndroidExtensions
{
    public static string GetText(this EditText editText)
    {
        return editText.Text.ToString()
    }
}

Then use it:
string val = EditText.GetText();

I know, I know, we've been through this before.  And I know about the problems of keeping things consistent.  And about what the platform provides.  And that all the MonoDroid guys will hate this email.

I find the EditText's .Text property not coming out as a string for user input really annoying.  I just typed
string val = EditText.Text;
and I got the visual studio "this is bad back to the corner and put the dunce cap on" red line.  Because I've been down this path before, I knew to wrap it with a Convert.ToString().  I just don't think that needing to add a Convert.ToString() is intuitive.  I think .NET developers will get frustrated on needing to add this.  Is there some way to create .TextAsString property or similar that would be the contents of the .Text property represented as a string?  Nathan suggested an extension method.

Yeah, yeah, I know this sounds like one of those little tiny things.  I have asked several of my friends on IM about a .Text property and those that are in the .NET space all think that .Text should be a string.  Nathan, who does Java Android as well as .NET likes it the way it is, and I understand why.
