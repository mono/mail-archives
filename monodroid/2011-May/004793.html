If you're doing any Sqlite, you probably know (from bug reports) that
anything referencing the System.Data.DataTable breaks because the library
doesn't include the sqlite3_column_origin_name() function (and perhaps
more).
 
So, am I correct in assuming we are limited to cmd.ExecuteNonQuery() for
INSERT UPDATE DELETE and cmd.ExecuteReader() for SELECT?
 
I also found out that a SqliteConnection does not support
connection.GetSchema( string tblname ), so even creating a DataTable
requires a SqliteDataReader to iterate over the columns to build a
DataTable, and then iterate the Reader to fill the table rows.
 
I have a faint memory of SQL programming like this at least 15 (or more
years ago) when our C-compilers had a pre-preprocessor pass to convert our
embedded-SQL statements into some bizarre set of calls to a database
provider's library.
 
Oh well, Sqlite does seem to work even without much ADO; but, I wanted to
check to see if anyone had some other ideas, or libraries of which I'm
unaware.
