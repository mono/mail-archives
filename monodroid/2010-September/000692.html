On Fri, Sep 10, 2010 at 10:11, Jonathan Pryor <jpryor at novell.com> wrote:
>
> On Fri, 2010-09-10 at 00:56 -0300, Daniel Cazzulino wrote:
> > Things like Trace, TraceListener and TraceSource are quite useful and
> > used in many projects. Porting may be a problem without those.
>
> The MonoDroid profile is based on the Silverlight profile, and
> Silverlight doesn't contain these types either.  (It would thus be fair
> to assume that Windows Phone 7 won't contain them either [0].)
>
> The System.Diagnostics.Debug type is present, but offhand I have no idea
> where those messages actually go.  We'll need to think about this. [1]
>
> Plus, as Miguel mentioned we're trying to keep our class library small
> (-ish); as it is, the assemblies that ship with Moonlight total 4.4MB in
> size, while the current MonoDroid profile assemblies are 8.5MB
> (excluding Mono.Android.dll).  Ideally, we'd like to shrink that [2],
> and adding additional dependencies won't help with that objective. :-)
>
> > Is there an alternative API to do this in Android?
>
> There are (oddly?) two:
>
>        http://docs.monodroid.net/index.aspx?link=T:Android.Util.Log
>        http://docs.monodroid.net/index.aspx?link=T:Android.Runtime.Logger
>
> Part of me wonders if we should remove the latter [3].  It was added as
> part of MonoDroid development (while binding all the Java types), and
> likely isn't necessary any longer.
>
> Use of either will create messages which are visible in the 'adb logcat'
> Android SDK tool.
>
>  - Jon
>

>

>
> http://msdn.microsoft.com/en-us/library/system.diagnostics(VS.95).aspx
>

>
