Byron,

Here are few thoughts.

Performance.  I find performance a hard thing to measure.  For example, I originally started programming C on Unix 20+ years ago.  My career path took me VB on Windows, and then to .NET.  Not once was I asked about performance by a customer when the sucject of frameworks came up.  typically, in a line of business app, performance is a db issue.  Now, with a mobile app, performance has a bunch of various potential  bottlenecks.  1 ghz ARM processors with 512 megs of ram can be limiting.  3g networks are not that fast.  Having said all of that, there are still the .NET issues.  It takes time for an app to spin up.  How much?  Unfortunately, the linked is not done so we just don't know.  We won't know about memory utilization, working sets, or any of that cool stuff at this point in time.

There is a size overhead with the currently planned embedded runtime.  With that size, you get the .NET framework.  You don't have to go learn the framework built into Android.  You get to use the .NET framework you already know.  You don't have to go learn Eclipse, you get to continue to use Visual Studio, which contains a large ecosystem that makes your life easier. Its not that eclipse or java as implemented in Android has anything wrong, heck its a very good toolset.  Its just that can take time to come upto speed  on java/dalvik and Eclipse.  Eventually, we'll get a separate runtime published by Novell so the size will go down for the individual apps.   (Sorry if that wasn't suppossed to be public guys)

On a programming, you have to be efficient with what you do. Remember, its not a desktop system, don't create a ton of objects.  Be efficient with what you have.  You have really limited resources, cpu, memory, screen.

as for database choices, depending on remote databases will mean that your app is continually hitting the 3g/wifi antennas.  That will hit the battery.  I can drain a battery on my iphone faster than you can imagine because I hit the antennas too much.  I think I have this fixed enough be "caching" data in sqlite and only hitting the web services when I need to

On the performance front, I would always expect Mono to be slower than .NET on the same operation in the same situation and hardware.  I don't use the desktop/server versions of Mono, so I can't compare Mono and .NET on the same hardware.  Novell has limited resources in comparison to Microsoft, so I don't expect them to be able to catch them with regards to performance.  I know that Novell is working on changes to the base Mono framework to narrow the performance gap, specifically in the area the gc.  I don't know how that will translate to MonoTouch and MonoDroid.

Hopefully, a few of these random thoughts (some can call it drivel) are helpful.  As for the resources, check out our MonoTouch book that shipped back in July. Bryan Costanich has once coming out shortly.  Mike Bluestein has one coming out as well.  MonoDroid is much further behind with regard to learning resources.  I have an article coming out on it, not sure when it will be published. Its a beginner article, so probably not for this group.

Wally
