Applications can be defined in any addin. They are loaded by
MonoDevelop.Core, but loaded from the addin system. The only reason
the "build" tool is in MonoDevelop.Core is that it makes sense to put
it in there instead of proliferating addins.

The MonoDevelop.CSharp dependency is not a problem either.
MonoDevelop.Ide has a code formatter service with an extension point
that and the C# addin plugs into. So you can use the code formatter
service and it'll work for any file type that has a formatter
registered by an addin.

You can see this in action in the file template system:
https://github.com/mono/monodevelop/blob/master/main/src/core/MonoDevelop.Ide/MonoDevelop.Ide.Templates/SingleFileDescriptionTemplate.cs#L267

So the tool can be in any addin (or even a new addin you could publish
on the addin build service) - and it would use DesktopService to look
up the file's mime type, use that to look up a formatter from the
CodeFormatterService, look up the named policy from the policy
service, and pass the policy and the file to the formatter.
