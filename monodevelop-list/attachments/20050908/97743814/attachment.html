Add reference dialog draft.<BR>
<BR>
Background<BR>
<BR>
The actual dialog, It's mainly a port from the sharpdevelop dialog, which was designed for a windows environment, and doesn't fit on a linux desktop. For it to succeed, the add reference dialog should be as simple as posible and have integration with all the sources available on the system. This is, with mono's GAC, available projects, pkg-config, etc....<BR>
<BR>
Conclusion<BR>
<BR>
We already have integration with the GAC and with available projects, but the integration with pkg-config is a bit dificult. Pkg-config doesn't provide information about which language is a especific package for. So, we should not list all available packages, because this would be a mess, and is not the way to go.<BR>
<BR>
Also, the dialog, should be extendable so if new sources appear the dialog integrates tightly with them.<BR>
<BR>
The approach of tabs for available sources, is not as extendable and simple as I imagine should be.<BR>
<BR>
Solution.<BR>
<BR>
I suggest a tree based approach. With this, anybody could add nodes to the tree. The nodes evindently should be ordered by some topic. The ones I think should be there by default are the ones that follows:<BR>
<BR>
+ Global assembly cache<BR>
<BR>
+ Available projects<BR>
<BR>
+ External assemblies history<BR>
<BR>
A plugin then, could add a category. So for example, the nunit plugin could register his own category, so it includes all the assemblies that is need for using nunit. I'm also thinking about a gtk# plugin that registers at load time, through parsing the pkg-config output, a Gtk# category.<BR>
<BR>
If a category is selected, and the add button is clicked, all the assemblies from that category should be added.<BR>
<BR>
We also, could have a way to let users add categories that they use extensively. This could be added by hand from available assemblies, or from the output of a command. Think about a Database + System web category, for example.<BR>
<BR>
We could have also a filter entry, where a user inputs some pattern, and the tree only shows the available assemblies that match the pattern.<BR>
<BR>
