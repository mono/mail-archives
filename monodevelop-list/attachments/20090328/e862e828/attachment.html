Hi, After speaking to mhutch on IRC the other day, I decided to try and see if I can take several of the best features of the Mylyn Plugin for Eclipse, which has been lauded to improve productivity among teams and deal with information overload. I&#39;d love to see this feature in MonoDevelop, and since it&#39;s a deliverable independent extension, I think it&#39;ll make a nice GSoC project. Please reply with any feedback, questions, or suggestions, I&#39;ll be on #monodevelop with the handle &quot;antileet&quot; if you have any feedback.<br>

<br>Thank you.<br><br>Proposal draft content:<br>Title: Mylyn-like functionality for developer collaboration.<br><br>Deliverable:<br><br>(A) A bug report management system with a single frontend and multiple, pluggable backends to write to local database, bugzilla, trac, jira, etc with the following functionality:<br>

  (i)  Create, edit delete and query for bug reports with an interface built in to the IDE.<br>  (ii) Edit bugs, change status, assignment, cc list, and many other parameters exposed by the backend. <br>  (iii) Comment on bug reports from within the IDE, with ability to link to lines of code and previous bugs, attach and download files, apply attached patches and test them. The IDE will   also unobtrusively notify you of unread comments to watched bug reports, preventing them from overflowing your inbox.<br>

  (iv) Ability to save queries for bugs and even categorize bug reports locally. Suggested examples include bugs assigned to co-workers, and long term goals for milestone releases, etc.<br><br>(B) A system encouraging task driven development. A bug report is treated as a &quot;task&quot; that has to be worked on. Tasks can be local - for personal use and planning, or on a remote repository. In this case, a bug report is equivalent to a task with only slightly different background. I hope to expose the following functionality of the task driven development:<br>

  (i) On marking a task as &quot;Active&quot;, and thus choosing to work on it, you are not interested in any other details. The GUI will hide all other tasks, and will not show notification of new bugs, comments, etc, and will let you do your work.<br>

  (ii) A small notice in the corner of the window reminds you on what you are working on, and provides an easy and quick way to add comments to the bug/task, thus alerting your co-workers about your progress(almost like a twitter feed for your bug).<br>

  (iii) [TIME PERMITTING]Cloning mylyn&#39;s famous &quot;Context&quot; Feature. Give a particular task a context comprising of files, functions, classes, etc. Now the solution view, and other views show only these files and even the jump-to-type and jump-to-file show only these. This prevents overload on larger projects, and enables tasks to be finished sooner. This feature would obviously be optional and has to be willingly turned on.<br>

<br><br>Benefits:<br>1. Functionality similar to what&#39;s found in Mature, Proprietary IDEs.<br>2. Possibly very useful for organizations and large software development teams, and might help improve adoption of MonoDevelop in corporate environments.<br>

3. Useful even locally for individual users. Local tasks will probably use ideas from David Allen&#39;s &quot;Getting things done&quot; technique.<br>2. Will be implementation independent and universally useful.<br>3. Makes communication between developers more seamless and intiutive, with a bonus ability to link to code from bug reports and bug reports from code.<br>

4. Prevents you from having to check your email/internet which can lead to distractions.<br><br>Inspiration:<br>Almost all ideas have been taken by the mature Mylyn project , which is used as an Eclipse plugin. A lot of extra features and ideas are in concept now, and not drafted officially.<br>
