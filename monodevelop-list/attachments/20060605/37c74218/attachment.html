I'd be happy if it were possible to manually add gtk.Fixed to the stetic design.&nbsp; I tried modifying gui.stetic manually but compilation only showed that the Fixed class was unresolved.<BR>
<BR>
While I'm familiar with java layout manager classes and I can conceptually map gtk.Table to BorderLayout one interesting decision about Stetic is there is no default layout.&nbsp; Also it seems that there is no corollary to the flow layout that would allow easily placing multiple widgets onto a frame.&nbsp; With Stetic, placing a widget onto a top level window only allows adding that *one* child.<BR>
<BR>
Javadocs show subclasses but mondocs seemingly make it hard to find classes that implement high-level concepts.&nbsp; For example, I look at gtk.Bin and see Button.&nbsp; That's mentioned in Remarks but not explicitly as a subclass, but anyway...&nbsp; If I add a Bin::Button to the gui designer, I'm prevented from adding any other kind of control, including other containers.<BR>
<BR>
I'm sure that latecomers like myself are missing a lot of the design and tradeoff conversation.&nbsp; Another issue for me though is I have yet to find hard crunchy documentation about Stetic itself.&nbsp; Given Monodevelop 0.11, I had to create a few sample projects, really, just to verify that GTK was the only place Stetic allowed visual GUI design.&nbsp; I probably just missed the place where this was stated, I'm just saying that was my experience.&nbsp; Linux.&nbsp; Older Monodevelop. Upgrade.&nbsp; Dive In!~<BR>
<BR>
So I probably missed some important things.&nbsp; A tutorial and a core Stetic resource would be real helpful at this point.<BR>
<BR>
I agree generally that a fixed layout is probably a bad idea for 95% of any application, including those you only expect to use for yourself.&nbsp; The tradeoff is &quot;is it easy using Stetic to modify the UI&quot;.&nbsp; At this point I have no idea but am excitedly anticipating finding out&nbsp; ;)<BR>
<BR>
Thanks!<BR>
-rsh<BR>
