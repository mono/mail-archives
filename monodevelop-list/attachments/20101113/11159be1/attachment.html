With some more help (many thanks) I progressed a bit further and seem
to be able to install a working .deb file, but I also seem to have
found a few of 'oddities':<br>
<br>
1. I need to add this line<br>
<blockquote>m4_include ([expansions.m4])<br>
</blockquote>
into the file configure.ac just before the 3 SHAMROCK_EXPANDS_... lines<br>
in order to avoid errors when running ./configure<br>
<br>
2. I also need to edit the resultant configure file (on around line
539) and change the line<br>
<blockquote>ac_default_prefix=/usr/local<br>
to<br>
ac_default_prefix=/usr<br>
</blockquote>
<br>
These 2 changes then produce a (package_name) user executable as below<br>
<blockquote>#!/bin/sh<br>
  <br>
exec mono "/usr/lib/myrss/myRSS.exe" "$@"<br>
</blockquote>
which is installed into the folder /usr/bin as myrss - as required on
Debian systems, as I understand it...<br>
<br>
Without these changes, the path in the script was not being resolved
and was left as:<br>
<blockquote>#!/bin/sh<br>
  <br>
exec mono <a moz-do-not-send="true"
 href="mailto:@expanded_libdir@/@PACKAGE@/myRSS.exe" target="_blank">"@expanded_libdir@/@PACKAGE@/myRSS.exe"</a>
"$@"<br>
</blockquote>
(or was only partially resolved with the @PACKAGE@ label being replaced
with the package name)<br>
<br>
<br>
3. in Monodevelop, I've found that the application's .desktop file's
properties are not saved when MD is closed - the Target directory needs
to be set to .desktop applications directory but reverts to Program
files when MD is re-opened...(even though MD does say 'item saved')<br>
<br>
<br>
Patches 1 and 2 above seem to be kludges and probably break
'architectural independence' rules in Debian packaging, so any ideas on
how to better fix this would be appreciated.<br>
<br>
Point 3 above could be a bug? If anyone with more experience than me
feels it is, I will log a bug report?<br>

application on Linux, the problem is you need to write an installer for
mono &amp; co. as well, which is what really kills your productivity.

Quandary emailed&nbsp; on 25/10/10 11:23:
      <br>
      <blockquote type="cite">&nbsp;I still wanted to add 3 things:
        <br>
        <br>
1. As a commerical solution, you can use InstallAnywhere (.bin files
you see for example in Google Earth, Sybase, or Oracle Java). I'd not
do that however, as it doesn't check for dependencies.
        <br>
        <br>
2. The installed version of mono on all the different Linux
distributions and distribution versions is not quite the same. So if
you for example depend on mono 2.6 (as opposed to 2.4, current Ubuntu
Lucid Lynx version), you'll basically have to package mono 2.6 as well.
This happended to me because there's a bug in 2.4 concerning XML and
XmlTextReader/writer encoding.
        <br>
        <br>
3. If you have a RPM file, you can use the Debian-Tool 'alien' to
convert it into a .deb file automagically (careful).

Unfortunately MonoDevelop doesn't have any built-in support for
          <br>
packaging. The best it can do is create a zip of binaries, or a
          <br>
tarball with build/install scripts. I know, we've wanted packaging
          <br>
support for a while
          <br>
(<a class="moz-txt-link-freetext" href="http://monodevelop.com/Developers/Tasks/Packaging">http://monodevelop.com/Developers/Tasks/Packaging</a>) but it's difficult
          <br>
to implement fully and there hasn't been huge demand.
          <br>
          <br>
Linux packaging depends on the distro, unfortunately - the major
          <br>
package formats are rpm (openSUSE, SLE*, Redhat, Centos, Fedora, etc)
          <br>
and deb (Debian, Ubuntu, etc.), though a few distros use other systems
          <br>
(Gentoo etc).
          <br>
          <br>
I've only used RPM myself. The way it works is that you write a
          <br>
specfile: a description, a list of dependencies, the commands to build
          <br>
and install (or simply unzip and cp) the app, and a list of files to
          <br>
package up. You then feed it to a tool that builds and installs the
          <br>
package into a fake root and collects the installed files and packages
          <br>
them up. I think deb is fairly similar. If you're lucky, you might be
          <br>
able to make packages that are usable on several distros or versions,
          <br>
depending how similar they are.
          <br>
          <br>
If your app is open-source, I suggest you investigate the openSUSE
          <br>
Build Service which, given source and spec files, can build packages
          <br>
for multiple distros automatically. The packages already there will
          <br>
provide good examples to base yours on.
