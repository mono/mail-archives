<BLOCKQUOTE>
    In SpecTcl, one creates the desired widget layout in the windows, one adds extra code that should be run each time such a window is created and this is saved in a file named something like <I>foo.ui</I> (SpecTcl's own file format). Hereafter, SpecTcl can generate a Tcl/Tk file (<I>foo.ui.tcl</I>) which contains a function (<I>foo_ui</I>). When called with an existing frame or toplevel as the first argument, this function creates the widgets as children of its first argument. All that is left for you to do is to source the Tcl/Tk file (<I>foo.ui.tcl</I>), create a toplevel or frame and call the function (<I>foo_ui</I>) with the frame/toplevel as the first argument.<BR>
</BLOCKQUOTE>
It seems that Stetic is fairly tightly coupled to GTK and that's OK by me.&nbsp; If it's eventually going to handle other GUI frameworks then some constraints move into an abstraction layer around the implementing toolkit.&nbsp; I'd rather see Stetic do one thing well than approximate a lot of things.<BR>
<BR>
OTOH, if Stetic remains a GTK designer, then -to me- the only real issue is &quot;can it handle all GTK widgets?&quot;.&nbsp; If gtk.Fixed is not deprecated then an obvious case for its support in Stetic can be made, despite the obvious pitfalls.&nbsp; I'd rather be given the rope and I'll take responsibility if I'm hanged.&nbsp;&nbsp; ;)<BR>
<BR>
I get the impression that you see a rather more elegant use of SpecTcl concepts and that is what I'm actually asking about.&nbsp; Can you elaborate please?<BR>
