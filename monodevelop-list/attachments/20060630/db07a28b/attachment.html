I have been thinking about a better way to generate code from the stetic XML. <br><br>The actual schema is something like this:<br><br>Visual Designer<br>&nbsp;||<br>&lt;==&gt;<br>&nbsp;||<br>XML<br clear="all">&nbsp;||<br>&nbsp;=&gt;<br>&nbsp;||<br>
Code Generation<br>&nbsp;||<br>&nbsp;=&gt;<br>&nbsp;||<br>Source Code<br><br>But there are some issues with the generated code:<br><br>1.- The widgets are not intances of our class (Gtk.Window) and we need to cast (from and to) the widgets, not checking types with the compiller
<br>2.- The signal handlers are connected on execution time (not compiller check)<br>3.- No code completion for the widgets (same reason)<br><br>So, the opposite option is to generate all the widgets as instances, but then the problem is to parse the source file (model of 
<a href="http://VS.NET">VS.NET</a> 2002/2003)<br><br>But there's other way, taking an approach similar to the <a href="http://VS.NET">VS.NET</a> 2005. It's to create a partial class. So, for example, if we create a visual interface (stetic) for this class:
<br><br>public class MyWindow : Gtk.Window<br><br>we can change it to a partial class (currently supported my both mcs and gmcs):<br><br>public partial class MyWindow : GtkWindow<br><br>and create other file (like MyWindow.Generated.cs
) with the same partial class, implementing all the widgets of the visual design. <br><br>With this, all the issues are solved and there's no need to parse the source file because the visual designer is always based on the xml file (just like the actual behavior).
<br><br>But, oops.. ok the problem is that this is only supported for C# (because of the 'partial' keyword)... uhm... so it should be better to have the code in a hidden #region? (I like partial)...<br><br>Ideas?<br><br>Juan Cristóbal Olivares
