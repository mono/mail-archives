If I were running the Mono project I would be very, very careful about
how things were grouped together from a marketing perspective.&nbsp; Novell
is now a Linux company, but Mono's stated goal is to be as ubiquitous
on Windows, Mac OS, and Unix as is Java.&nbsp; Tying in packages that
purposely don't run on Windows isn't going to help Novell get Mono into
places that are Windows strongholds.<br>
<br>
I, for one, have never understood the need for MonoDevelop to fork from
Sharp Develop.&nbsp; The time and effort put into porting to GTK# could have
been better spent getting SWF working in Mono so that the Sharp Develop
code would run on Mono unaltered, and infinitely improve Mono's market
position as a tool to allow truly ubiqitous client-server development
platform that would outperform Java in every way.<br>
<br>
MonoDevelop has a very important psycological position in the world of
.Net development.&nbsp; It's name infers that where Mono goes, MD goes as
well.&nbsp; If someone new to Mono goes balls-to-the-walls out to get going
with Mono, tries to download the branded IDE, and then discovers it
doesn't work on Windows, he's going to be pretty pissed and quite
likely to blow off Mono as vaporware or hypeware. (did I just coin a
new term?)<br>
