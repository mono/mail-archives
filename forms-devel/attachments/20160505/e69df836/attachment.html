<div dir="ltr"><div>Topic: OnPlatform and Device.OS</div><div><br></div><div>TargetIdiom and TargetDevice on their own are fine and easily extended, however when combined with the OnPlatform API's we have no method of adding new platforms to Xamarin.Forms without incurring an ABI break in the core lib. This is due to some poor planning back in the day (hi there) and is needing to be fixed as we now have community interest in making a Mac backend.</div><div><br></div><div>The reason for the ABI limitation is due to the fact that Device.OnPlatform takes optional parameters AND has a default parameter which means we cannot simply add the new platforms to that API call without breaking old code.</div><div><br></div><div>Device.OnPlatform (iOSAction, androidAction, winPhoneAction, defaultAction);</div><div><br></div><div>would call to the wrong parameter if a hypothetical Mac action were added in there.</div><div><br></div><div>In order to avoid an ABI break I am proposing the following:</div><div><br></div><div>1) Obsolete Device.OnPlatform</div><div>2) Extend TargetPlatform to include Mac</div><div><br></div><div>The OnPlatform class (different from the Device.OnPlatform) does not suffer this limitation and can be kept. Users wishing to use OnPlatform semantics in the future should simply do:</div><div><br></div><div>switch (Device.OS) {</div><div>  case TargetPlatform.Mac:</div><div><br></div><div>    break;</div><div>  case TargetPlatform.iOS:</div><div><br></div><div>    break;</div><div>  default:</div><div><br></div><div>    break;</div><div>}</div><div><br></div><div>At this time I do not believe there is a strong need to actually replace the Device.OnPlatform call with something else as the language already provides this feature (albeit slightly more verbose).</div><div><br></div><div>Jason</div><div><br></div><b>What is this email?</b><div>This email is intended to start a conversation with the forms developers and the community about the above topic. This is t
