On 09/04/2009 10:24 AM, Glen Turner wrote:
> I'm trying to debug an application that uses DB4O for persistence. 
> After the first call to DB4O, monovs-server reports "Could not find file
> "/tmp/monovs-21bbd9d6-a2dd-4af4-a628-8871b9d4aa96/Db4objects.Db4o.dll.mdb". 
> The next time the code hits a breakpoint it stalls and Visual Studio
> does not appear to get informed that the breakpoint has been hit.

Hi,

You can just ignore the missing .mdb message, that's not a problem.

We need to investigate why the debugger hangs when it hits a breakpoint
and fix that.  Does this happen every time you debug something which is
using DB4O, or could you provide us with a small test case and steps to
reproduce this ?

> The code runs fine using "Run Remotely in Mono" and also debugs
> perfectly until the first call to DB4O.
> 
> I have tried to exclude the call from debugging by using the
> DebuggerNonUserCode attribute to no avail.

This attribute shouldn't affect this particular bug at all from what I
can tell so far - it's only checked during single-stepping to decide
whether or not to step into a method.

There must be something broken in the breakpoint code or when sending
the notification that the breakpoint has been hit to Visual Studio.

> I am using a native OpenSuse 11.1 box for testing, with all mono
> components updated today.

Ok, so I assume you're using the latest RPM packages which came with
OpenSuSE 11.1 ?

> Is this a bug or expected behaviour in the absence of debugging symbols
> for the whole application?

Not having debugging symbols doesn't matter here from what I can tell so
far, it looks like a bug in the breakpoint code which needs to be fixed.
